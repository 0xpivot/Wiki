## Understanding SQL Injection

SQL injection is a type of security vulnerability that allows an attacker to manipulate SQL queries executed by an application. This manipulation can lead to unauthorized access to sensitive data, data corruption, or even complete system compromise. In the context of API security, SQL injection vulnerabilities can arise when user input is improperly sanitized and used within SQL queries.

### What is SQL Injection?

SQL injection occurs when an attacker injects malicious SQL code into an input field that is later executed as part of a SQL query. This can happen in various scenarios, such as login forms, search boxes, or any other input fields where user input is directly incorporated into SQL queries.

#### Example of SQL Injection

Consider a simple login form where the username and password are checked against a database:

```sql
SELECT * FROM users WHERE username = 'input_username' AND password = 'input_password';
```

If an attacker inputs `input_username` as `' OR '1'='1` and `input_password` as `' OR '1'='1`, the resulting SQL query becomes:

```sql
SELECT * FROM users WHERE username = '' OR '1'='1' AND password = '' OR '1'='1';
```

This query will return all rows from the `users` table because the condition `'1'='1'` is always true.

### Types of SQL Injection

SQL injection can be categorized into different types based on the nature of the vulnerability and the method used to exploit it:

1. **Error-Based SQL Injection**: The attacker uses the error messages generated by the database to extract information.
2. **Union-Based SQL Injection**: The attacker combines the results of two or more SELECT statements using the UNION operator.
3. **Blind SQL Injection**: The attacker infers the structure of the database and extracts data without receiving direct feedback from the database.

### Blind SQL Injection

Blind SQL injection is a technique where the attacker cannot directly see the output of the injected SQL query. Instead, the attacker relies on conditional responses from the server to infer the structure of the database and extract data.

#### How Blind SQL Injection Works

In blind SQL injection, the attacker sends a series of SQL queries to the server and observes the server's response to determine whether the query was successful. This can be done by manipulating the time taken for the server to respond or by observing changes in the server's behavior.

### Error Handling in SQL Queries

One common way to perform blind SQL injection is by exploiting error handling mechanisms. By injecting SQL code that causes errors, the attacker can infer the structure of the database.

#### Example: Using ORDER BY Clause

The `ORDER BY` clause is often used in SQL queries to sort the results. An attacker can use this clause to test the number of columns in a table.

```sql
SELECT * FROM users ORDER BY 1;
```

If the query executes successfully, it means there is at least one column. The attacker can increment the number until the query fails, indicating the total number of columns.

```sql
SELECT * FROM users ORDER BY 5; -- Fails if there are fewer than 5 columns
```

### Injecting Conditional Statements

Another technique is to inject conditional statements that evaluate to true or false. This can be done using comparison operators like `=` or `!=`.

#### Example: Using Comparison Operators

```sql
SELECT * FROM users WHERE 1=1;
```

This query will always return true and can be used to confirm that the injection is working.

```sql
SELECT * FROM users WHERE 1=2;
```

This query will always return false and can be used to confirm that the injection is not working.

### Extracting Data Using Substring

Once the attacker has determined the structure of the database, they can use functions like `SUBSTRING` to extract data one character at a time.

#### Example: Using SUBSTRING

```sql
SELECT SUBSTRING(column_name, position, length) FROM table_name;
```

By iterating through the positions and lengths, the attacker can reconstruct the data.

### Real-World Examples

#### CVE-2021-21972: Apache Struts Vulnerability

Apache Struts is a popular framework for building Java-based web applications. In 2021, a vulnerability (CVE-2021-21972) was discovered that allowed attackers to perform SQL injection attacks. This vulnerability was exploited to gain unauthorized access to sensitive data.

#### CVE-2020-14882: WordPress REST API Vulnerability

WordPress is a widely used content management system. In 2020, a vulnerability (CVE-2020-14882) was found in the WordPress REST API that allowed attackers to perform SQL injection attacks. This vulnerability was exploited to gain unauthorized access to the database.

### How to Prevent / Defend Against SQL Injection

#### Detection

To detect SQL injection vulnerabilities, organizations should implement automated tools and manual reviews. Automated tools like static application security testing (SAST) and dynamic application security testing (DAST) can help identify potential vulnerabilities.

#### Prevention

1. **Input Validation**: Validate all user input to ensure it conforms to expected formats and ranges.
2. **Parameterized Queries**: Use parameterized queries or prepared statements to separate SQL logic from user input.
3. **Stored Procedures**: Use stored procedures to encapsulate SQL logic and reduce the risk of injection.
4. **Least Privilege Principle**: Ensure that database accounts have the minimum privileges necessary to perform their tasks.

#### Secure Coding Practices

Here is an example of a vulnerable SQL query and its secure counterpart:

**Vulnerable Code**

```python
import sqlite3

def get_user(username):
    conn = sqlite3.connect('database.db')
    cursor = conn.cursor()
    query = f"SELECT * FROM users WHERE username = '{username}'"
    cursor.execute(query)
    result = cursor.fetchone()
    conn.close()
    return result
```

**Secure Code**

```python
import sqlite3

def get_user(username):
    conn = sqlite3.connect('database.db')
    cursor = conn.cursor()
    query = "SELECT * FROM users WHERE username = ?"
    cursor.execute(query, (username,))
    result = cursor.fetchone()
    conn.close()
    return result
```

#### Vulnerable Scenario

**HTTP Request**

```http
POST /api/users/login HTTP/1.1
Host: example.com
Content-Type: application/json

{
  "username": "' OR '1'='1",
  "password": "' OR '1'='1"
}
```

**HTTP Response**

```http
HTTP/1.1 200 OK
Content-Type: application/json

{
  "message": "Login successful",
  "data": {
    "id": 1,
    "username": "admin",
    "email": "admin@example.com"
  }
}
```

POST /api/users/login HTTP/1.1
Host: example.com
Content-Type: application/json

{
