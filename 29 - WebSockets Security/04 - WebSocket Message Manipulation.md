# 29.04 — WebSocket Message Manipulation

## 1. Introduction: The False Sense of Security
**WebSocket Message Manipulation** is the foundational methodology for attacking the business logic and backend systems of an application that relies on WebSockets. It is not a vulnerability in itself, but rather the mechanism by which an attacker exploits backend flaws.

The most dangerous aspect of WebSockets is the psychological effect they have on developers. When building a standard REST API over HTTP, developers generally understand that the traffic is "public." They expect hackers to tamper with POST bodies and URL parameters, so they implement strict input validation, parameter binding, and authorization checks at the controller level.

However, when developers build a WebSocket architecture, the persistent, fast-paced binary stream feels "internal." Because the connection is established via a custom Javascript client, and the data is often packaged in rapid-fire, complex JSON structures or custom binary formats, developers frequently assume that the data arriving over the socket is inherently trustworthy. They assume it was generated by their own pristine frontend code.

This is a fatal assumption. An attacker with a proxy tool can intercept, pause, modify, and drop any WebSocket message mid-flight, exactly as they can with an HTTP request.

## 2. The Mechanics of Manipulation
To understand how to manipulate messages, you must understand the two primary architectures applications use for WebSockets:

### Architecture A: The "God Object" Router (JSON Multiplexing)
Most modern applications use a single WebSocket connection to handle dozens of different features. To route the data internally, the frontend sends a JSON "God Object" containing an `action` or `type` key.
*Example:* `{"action": "send_message", "channel_id": 42, "text": "Hello"}`
The backend uses a massive `switch` statement based on the `action` key.
*The Flaw:* Because a single socket handles everything, authorization checks are often implemented inconsistently across different `action` routes, leading to massive Insecure Direct Object Reference (IDOR) vulnerabilities.

### Architecture B: Custom Binary Protocols
Some high-performance applications (like browser games or trading platforms) avoid JSON overhead by using Protocol Buffers, MessagePack, or entirely custom binary formats.
*The Flaw:* Binary deserialization is notoriously difficult to write securely. Manipulating specific bytes in the payload can cause integer underflows, buffer overflows, or crash the deserialization library entirely, leading to Denial of Service or Remote Code Execution.

## 4. Methodological Discovery and Fuzzing
Testing WebSockets requires a methodical approach, as automated scanners are generally blind to custom WebSocket logic.

### Phase 1: Traffic Analysis
1. Proxy the target application through Burp Suite.
2. Navigate the application, performing every possible action. Open chat windows, modify settings, trigger notifications.
3. Open the `Proxy -> WebSockets history` tab in Burp.
4. Study the flow of traffic. Identify the JSON schemas. Map out every single `action` or `type` parameter you observe. Build a mental map of the WebSocket API routing.

### Phase 2: Live Interception
1. In Burp Suite, go to `Proxy -> Options` (or `Proxy -> Settings` in newer versions).
2. Scroll down to `Intercept WebSockets Messages`.
3. Check `Intercept Client-to-Server messages`.
4. Turn Intercept ON.
5. Trigger an action in the browser. Burp will freeze, displaying the raw frame.

### Phase 3: Fuzzing via Repeater
Live interception is slow. For aggressive testing, you must use Repeater.
1. Right-click an interesting frame in the `WebSockets history` tab.
2. Select `Send to Repeater`.
3. A new tab will open containing the active WebSocket session.
4. You can now modify the JSON payload and click `Send` rapidly. You will see the server's immediate responses in the right-hand pane.

*Critical Note:* WebSocket connections time out if they are idle. If your Repeater session dies, you must go back to the browser, trigger a new connection, and send the new connection to Repeater.

## 5. Common Attack Vectors via Manipulation

Once you have the ability to manipulate the frames in Repeater, you can apply standard web vulnerability testing methodologies to the JSON payload:

1. **Insecure Direct Object Reference (IDOR):**
   If the JSON contains IDs, change them.
   *Test:* `{"action": "delete_message", "message_id": 88}` -> change to `89`. Can you delete other users' messages?
   *Test:* `{"action": "update_profile", "user_id": 10}` -> change to `1` (Admin).

2. **Business Logic & Integer Manipulation:**
   Change prices, quantities, or mathematical states.
   *Test:* Change positive integers to negative integers.
   *Test:* Change floating-point numbers to extremely large values (`99999999999999999.99`) to test for overflow or floating-point precision errors in financial calculations.

3. **Type Juggling and Mass Assignment:**
   Change the data types of the JSON fields to crash the parser or bypass weak strict-equality checks.
   *Test:* Change a string `"1"` to an integer `1`. Change an integer to an array `[1]`. Change a boolean `true` to a string `"true"`.
   *Test:* Inject unexpected fields like `"is_admin": true` or `"role": "superuser"`.

4. **Injection Flaws (SQLi, XSS, Command Injection):**
   Inject classic payloads into text fields.
   *Test:* `{"action": "search", "query": "' OR 1=1--"}` (SQLi)
   *Test:* `{"action": "chat", "msg": "<img src=x onerror=alert(1)>"}` (XSS)

## 6. Real-World Case Study
A researcher audited an online multiplayer casino game. The game used a custom WebSocket architecture built on Node.js to handle the live blackjack tables.

When a player placed a bet, the client sent:
`{"action": "bet", "amount": 50}`
The backend checked if the player's balance was `>= 50`, deducted the amount, and started the hand.

The researcher sent the message to Repeater and modified the payload:
`{"action": "bet", "amount": "50"}` (Note the quotes—changing the integer to a string).

The backend Node.js code used loose typing for validation: `if (user.balance >= message.amount)`. In JavaScript, `100 >= "50"` evaluates to `true`, so the validation passed.
However, when deducting the balance, the code used addition: `user.balance = user.balance - message.amount`. Due to JavaScript's bizarre type coercion, if a string is involved, subtraction can sometimes result in `NaN` (Not a Number), or if the developer used `user.balance += -message.amount`, it resulted in string concatenation.

By sending a carefully crafted string array `{"amount": ["-1000"]}`, the researcher crashed the backend calculation engine, causing the database to record the player's balance as `NULL`, which a separate backend process interpreted as infinite funds, allowing the researcher to drain the casino.

## 7. How to Fix It (Developer Remediation)

1. **Zero Trust Architecture:** The foundational rule of WebSockets is that the client is always assumed to be compromised. Never trust any data parameter arriving over the socket.
2. **Server-Side State Authority:** Never allow the client to declare its own identity or permissions. If a message contains `"user_id": 5`, ignore it. The backend must map the active WebSocket connection object to the authenticated user session established during the handshake. Use the server's session data, never the client's payload data.
3. **Strict Schema Validation (The WAF Replacement):** Because traditional WAFs cannot inspect WebSocket JSON, the application must implement its own strict schema validation layer. Use libraries like `Joi` (Node.js) or `Pydantic` (Python) to define rigid schemas for every `action` type. If a message contains an unexpected field, a string instead of an integer, or an integer out of bounds, the server must drop the message immediately before it ever reaches the business logic controllers.
