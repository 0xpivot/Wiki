# 97.13 Mythic Event Feed and Team Collaboration

## 1. Introduction to Multi-Player C2 Operations

Modern Red Team engagements are rarely solo endeavors. They involve distributed teams operating across different time zones, managing multiple access vectors, and coordinating complex objective-based operations. 

Mythic was built from the ground up to support "Multi-Player" operations. Unlike older frameworks that rely on simplistic shared text logs, Mythic incorporates a robust Event Feed, detailed operational logging, Role-Based Access Control (RBAC), and external ChatOps integrations (Slack, Discord, custom Webhooks) to maintain team synchronization and operational security (OpSec).

## 2. The Mythic Event Feed Mechanics

The Event Feed is the central nervous system of a Mythic operation. It provides a real-time, chronological timeline of critical events across the entire infrastructure.

### 2.1 Key Event Types
- **Agent Check-ins:** Notifications when a new payload successfully authenticates and establishes a session.
- **Tasking Execution:** Records of high-impact commands being queued by operators.
- **Payload Generation:** Logs of when and who generated a new malicious payload.
- **File Uploads/Downloads:** Tracking data exfiltration and tool staging.
- **Errors and Alerts:** Notifications of agent crashes, parsing errors, or dropped C2 connections.

## 3. Integrating Webhooks and ChatOps

To reduce the need for operators to constantly stare at the Mythic UI, Mythic supports extensive webhook configurations. When a critical event occurs, Mythic packages the event data and fires a POST request to a defined endpoint.

### 3.1 Custom Webhook JSON Payload
When a new agent checks in, Mythic sends a highly structured JSON payload to the webhook endpoint. Operators can use intermediary services (like AWS API Gateway, n8n, or custom Python scripts) to format this for specific chat applications.

```json
{
    "action": "new_callback",
    "operation": "Op_Chimera",
    "operator": "mythic_admin",
    "data": {
        "id": 142,
        "display_id": 142,
        "uuid": "4f8a9b21-...",
        "user": "NT AUTHORITY\\SYSTEM",
        "host": "PROD-SQL-01",
        "pid": 4182,
        "os": "Windows Server 2019",
        "architecture": "x64",
        "integrity_level": 3,
        "payload_type": "Apollo"
    }
}
```

### 3.2 ChatOps Implementation Example
A common setup involves routing Mythic webhook traffic to a Discord channel dedicated to the engagement.

1. Create a Discord Webhook in the target channel.
2. In Mythic Settings, configure the custom webhook URL.
3. Utilize Mythic's built-in webhook formatting or point to a custom translation server.
4. Result: Immediate situational awareness for the entire team when initial access is achieved, allowing designated lateral movement operators to immediately jump into action.

## 4. Role-Based Access Control (RBAC) and Operations

Mythic segments engagements into "Operations". Users are assigned specific roles within an operation, ensuring strict compartmentation and preventing accidental interference.

### 4.1 Operational Roles
- **Operation Admin:** Full control. Can modify C2 profiles, add operators, and delete the operation.
- **Lead Operator:** Can issue tasks, generate payloads, and approve specific high-risk commands.
- **Operator:** Standard execution privileges. Can interact with assigned agents.
- **Spectator:** Read-only access. Extremely useful for White Cell (observers) or Blue Team (defenders) during purple team exercises to monitor the exact commands being executed in real-time.

### 4.2 MITRE ATT&CK Mapping
Every command executed by an operator can be strictly mapped to MITRE ATT&CK techniques. The Event Feed aggregates this data, allowing the team lead to generate comprehensive execution reports at the end of the engagement, significantly reducing the administrative burden of final reporting.

## 5. Operational Security (OpSec) Logging

Mythic acts as an immutable ledger of Red Team activity. Every keystroke, uploaded file, and downloaded artifact is cryptographically hashed and logged.
- **Deconfliction:** If the Blue Team detects a malicious artifact (e.g., `evil.exe`), they can ask the Red Team. The Red Team can search the Mythic Event Feed by SHA256 hash. If it exists in the feed, it was part of the simulation; if not, it represents a real-world parallel intrusion.
- **Artifact Tracking:** Mythic tracks exactly which payload UUID was executed on which host, providing an exact lineage of lateral movement.

## 6. Real-World Attack Scenario

**The Environment:** A massive 2-week Red Team engagement against a multinational corporation. The team consists of 5 operators: 2 focused on Initial Access, 2 on Active Directory exploitation, and 1 Team Lead overseeing the operation.

**The Execution:**
1. The Initial Access operators successfully land a phishing payload. A new Apollo agent calls back to Mythic.
2. Mythic instantly fires a webhook to the team's Slack channel: `[ALERT] New Callback: PROD-EXCHANGE-01 | User: IIS APPPOOL\DefaultAppPool | Integrity: Medium`.
3. The AD exploitation operators, seeing the alert in Slack, immediately log into the Mythic UI.
4. Because of RBAC, the Initial Access operators pass the agent token to the AD operators, who begin enumerating the domain.
5. The Team Lead monitors the Event Feed, ensuring no out-of-scope hosts are touched. The lead notices an operator attempting to upload a noisy port scanner.
6. Utilizing Mythic's tasking review process, the Team Lead blocks the noisy task execution directly from the UI, preserving the team's stealth.
7. Post-engagement, the Team Lead exports the entire Event Feed, cleanly mapping the intrusion chain to MITRE ATT&CK for the final client debrief.
