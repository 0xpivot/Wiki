# 95.02 Deploying Sliver Team Server and Multiplayer Mode

## Overview

The deployment phase of a Command and Control (C2) infrastructure is arguably the most critical for Operational Security (OPSEC). A poorly deployed Team Server can easily be fingerprinted by blue teams, scanned by automated threat intelligence platforms (like Shodan or Censys), and quickly blocked.

Sliver, unlike many older frameworks, separates the operator client from the team server backend. This architecture natively supports "Multiplayer Mode," allowing multiple red team operators to connect to a single backend securely. This document details the advanced deployment of the Sliver Team Server, securing the infrastructure, and managing multiplayer operations.

---

## Advanced Deployment Considerations

Deploying a raw Sliver executable on a VPS and running it interactively as root in a `tmux` session is sufficient for basic testing but highly insecure for professional emulation engagements. An advanced deployment requires system hardening, running the server as a daemon, and employing redirectors.

### 1. System Integration and Daemonization

To ensure persistence and proper lifecycle management of the server, it should be integrated into the host operating system's service manager (`systemd` on modern Linux distributions).

**Conceptual Systemd Service File (`/etc/systemd/system/sliver.service`):**
```ini
[Unit]
Description=Sliver C2 Team Server
After=network.target
Wants=network.target

[Service]
Type=simple
User=sliver
Group=sliver
ExecStart=/opt/sliver/sliver-server daemon
Restart=on-failure
RestartSec=5
LimitNOFILE=65535
# Security hardening
NoNewPrivileges=yes
PrivateTmp=yes
ProtectSystem=full

[Install]
WantedBy=multi-user.target
```
Running the server in `daemon` mode (`sliver-server daemon`) detaches it from standard I/O, preventing the interactive console from spawning. The server then listens only for operator connections on the multiplexed gRPC port (default 31337) and active listener ports.

### 2. Securing the Management Interface

By default, the `sliver-server` exposes port `31337/TCP` for operator client connections. This port uses mutually authenticated TLS (mTLS). Even though it is encrypted and authenticated, exposing port `31337` to the public internet is a major OPSEC violation, as standard port scanning will reveal the service.

**Defensive Mitigations & OPSEC Best Practices:**
1. **Change the Default Port**: Modify the listening port to something less suspicious (e.g., 443, 8443, or high-random ports) via the server configuration file (`~/.sliver/configs/server.json`).
2. **Restrict Access via Firewall (UFW/iptables)**: Only allow incoming connections to the management port from known Operator IP addresses (e.g., VPN gateways or specific jump boxes).

```bash
# Example UFW rules for restricting management access
ufw deny 31337/tcp
ufw allow from 198.51.100.45 to any port 31337 proto tcp
```

## Multiplayer Mode Configuration

Multiplayer mode allows a distributed red team to collaborate in real-time, sharing sessions, loot, and access to generated implants. Authentication is handled strictly through X.509 client certificates generated by the team server.

### Generating Operator Profiles

To grant an operator access, the administrator must run the `new-operator` command from the server console.

```bash
# Generating a new operator configuration on the server
sliver > new-operator --name Alice --lhost team-server.evil.com --lport 31337

[*] Generated new client config: /home/sliver/Alice_team-server.evil.com.cfg
```

This `.cfg` file contains:
1. The server's IP address and port.
2. The Team Server's Certificate Authority (CA) public key.
3. A newly generated private key and certificate specific to "Alice".

The operator "Alice" then imports this configuration into her local Sliver client:

```bash
# Importing the profile on the local operator machine
$ sliver-client import /path/to/Alice_team-server.evil.com.cfg
$ sliver-client
```

### Role-Based Access and Auditing

Sliver logs all commands executed by operators. In a mature threat hunting scenario, discovering the server backend allows defenders to reconstruct the entire attack chain by parsing the `.sliver/logs/` and database entries.

## Redirectors and Obfuscation

To protect the true IP address of the Team Server from being discovered by incident responders, Red Teams utilize **Redirectors**.

A redirector is a simple, disposable server that forwards specific traffic to the Team Server. If the redirector is flagged by a Blue Team, it can be destroyed and replaced without losing the core Team Server infrastructure.

### Dumb Redirectors (socat / iptables)
The simplest form of redirector operates at Layer 4 (Transport). It blindly forwards traffic from a specific port to the backend server.

**Example `socat` TCP Redirector:**
```bash
# Run on the Redirector node
# Forwards incoming traffic on 443 to the Team Server's mTLS listener on 8888
socat TCP4-LISTEN:443,fork TCP4:teamserver-ip:8888
```

### Smart Redirectors (Nginx / Apache / Caddy)
Layer 7 (Application) redirectors are used for HTTP/S listeners. They inspect the incoming HTTP requests. If the request matches a specific profile (e.g., an expected `User-Agent` or URI path defined in the Sliver profile), it forwards the traffic to the Team Server. If the request appears to be from a Blue Team scanner or unauthenticated user, it redirects the traffic to a benign site (e.g., `https://google.com`).

## Threat Hunting & Detection Engineering

Detecting a Sliver Team Server directly is challenging if properly configured, but poor OPSEC provides hunting opportunities:

1. **Shodan/Censys Fingerprinting**: Unconfigured HTTP listeners on Sliver often return distinct HTTP 404 responses or specific TLS certificates. Hunters can search for servers presenting certificates issued to default Sliver CA names (if the operators failed to use Let's Encrypt or custom certs).
2. **Network Traffic Analysis (NTA)**: Identifying operator connections involves looking for sustained, long-running mTLS connections originating from VPN endpoints to high ports on cloud-hosted VPS providers. 
3. **JA3 / JA3S Hashes**: Sliver uses specific Golang TLS libraries. Profiling the JA3 hash of the client connection and the JA3S hash of the server response can identify Golang-based C2 infrastructure, even over custom ports.

## Real-World Attack Scenario

### Phase 1: Infrastructure Setup
An APT group is preparing for a long-term espionage campaign. They deploy a Sliver Team Server inside an AWS VPC. They do not expose the server to the internet. Instead, they deploy three disposable DigitalOcean droplets to act as redirectors (one for HTTP, one for DNS, one for mTLS). 

### Phase 2: Multiplayer Configuration
The lead operator connects to the AWS instance via a private VPN and generates operator profiles for three specialized team members: an Initial Access Broker, a Lateral Movement specialist, and a Data Exfiltration specialist. 

### Phase 3: Execution and Evasion
The Initial Access Broker compromises the perimeter and drops an implant that connects back through the HTTP redirector. Once stable, they announce the successful connection in the team chat. The Lateral Movement specialist connects their local `sliver-client` to the Team Server, takes control of the active session, and begins exploring the internal network, while the backend database transparently records all command histories for post-engagement reporting.

## Chaining Opportunities

- **Cobalt Strike Integration**: Sliver can act as a sophisticated backup C2. An operator might establish an initial foothold with Sliver via DNS, and then spawn a Cobalt Strike beacon over HTTPS once the environment is deemed safe, utilizing the multiplayer console to manage the handoff.
- **Automated Deployments**: Using Infrastructure as Code (Terraform/Ansible) to automatically deploy the Team Server, configure UFW, spin up redirectors, and generate operator `.cfg` files, ensuring repeatable and consistent OPSEC.
