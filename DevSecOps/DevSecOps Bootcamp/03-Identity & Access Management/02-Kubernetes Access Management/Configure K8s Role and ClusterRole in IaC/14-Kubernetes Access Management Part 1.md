## Kubernetes Access Management

### Background Theory

Kubernetes access management is a critical aspect of securing your Kubernetes clusters. It ensures that only authorized users and processes can perform specific actions within the cluster. This is achieved through a combination of authentication, authorization, and resource management mechanisms.

#### Authentication

Authentication is the process of verifying the identity of a user or service. In Kubernetes, authentication can be performed using various methods such as:

- **X.509 client certificates**: These are digital certificates that verify the identity of a client connecting to the API server.
- **Username and password**: Basic authentication using a username and password.
- **Token-based authentication**: Using bearer tokens, typically generated by an external identity provider.
- **Service accounts**: Predefined accounts used by pods and services within the cluster.

#### Authorization

Authorization is the process of determining whether an authenticated user or service is allowed to perform a specific action. Kubernetes uses Role-Based Access Control (RBAC) to manage authorization. RBAC allows you to define roles and bindings that specify which users or groups can perform certain actions.

### Configuring Roles and ClusterRoles in Infrastructure as Code (IaC)

In the context of Infrastructure as Code (IaC), you can use tools like Terraform to manage and configure Kubernetes resources. This includes defining roles and role bindings to control access to the cluster.

#### Users and Roles

Let's start by defining two users: `admin` and `developer`. These users will be mapped from external AWS roles to Kubernetes users.

```yaml
# aws_users.tf
resource "aws_iam_role" "external_admin" {
  name = "external_admin"
}

resource "aws_iam_role" "external_developer" {
  name = "external_developer"
}
```

These AWS roles will be mapped to Kubernetes users. The mapping is done using a `Subject` in Kubernetes RBAC.

```yaml
# kubernetes_users.yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: User
metadata:
  name: admin
users:
- name: admin

apiVersion: rbac.authorization.k8s.io/v1
kind: User
metadata:
  name: developer
users:
- name: developer
```

### Creating Roles and Role Bindings

Next, we need to create roles and role bindings for these users. A `Role` defines a set of permissions within a namespace, while a `ClusterRole` defines permissions across the entire cluster.

#### Role Definition

A `Role` can be defined as follows:

```yaml
# kubernetes_roles.yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: default
  name: developer-role
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "list", "watch"]
```

This role grants the `developer` user the ability to list and watch pods in the `default` namespace.

#### Role Binding

To bind this role to the `developer` user, we create a `RoleBinding`:

```yaml
# kubernetes_role_bindings.yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: developer-binding
  namespace: default
subjects:
- kind: User
  name: developer
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: developer-role
  apiGroup: rbac.authorization.k8s.io
```

#### ClusterRole Definition

For the `admin` user, we might want to grant broader permissions across the entire cluster. This is done using a `ClusterRole`.

```yaml
# kubernetes_clusterroles.yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: admin-clusterrole
rules:
- apiGroups: [""]
  resources: ["*"]
  verbs: ["*"]
```

This `ClusterRole` grants the `admin` user full access to all resources in the cluster.

#### ClusterRole Binding

To bind this `ClusterRole` to the `admin` user, we create a `ClusterRoleBinding`:

```yaml
# kubernetes_clusterrole_bindings.yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: admin-binding
subjects:
- kind: User
  name: admin
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole
  name: admin-clusterrole
  apiGroup: rb
```

### Pitfalls and Best Practices

#### Common Mistakes

1. **Overly Permissive Roles**: Avoid granting more permissions than necessary. For example, granting `*` to all resources and verbs can lead to security vulnerabilities.
2. **Incorrect Role Binding**: Ensure that the role binding correctly maps the role to the intended user or group.
3. **Missing Namespace Scope**: Remember that `Role` is scoped to a namespace, while `ClusterRole` is cluster-wide. Misconfiguring this can lead to unexpected behavior.

#### Best Practices

1. **Least Privilege Principle**: Grant users only the minimum permissions required to perform their tasks.
2. **Regular Audits**: Periodically review and audit roles and bindings to ensure they remain appropriate.
3. **Use Namespaces**: Utilize namespaces to scope permissions and isolate resources.

### Real-World Examples

#### Recent CVEs and Breaches

This vulnerability allowed attackers to bypass RBAC restrictions and gain unauthorized access to sensitive resources. The issue was due to a flaw in the Kubernetes API server that could be exploited to escalate privileges.

### How to Prevent / Defend

#### Detection

1. **Audit Logs**: Enable and monitor audit logs to track access attempts and identify unauthorized activities.
2. **Security Scanners**: Use tools like `kube-bench` to scan your cluster for misconfigurations and vulnerabilities.

#### Prevention

1. **Least Privilege**: Implement the least privilege principle by granting minimal permissions.
2. **Namespace Isolation**: Use namespaces to isolate resources and limit the scope of permissions.
3. **Regular Reviews**: Conduct regular reviews of roles and bindings to ensure they remain appropriate.
