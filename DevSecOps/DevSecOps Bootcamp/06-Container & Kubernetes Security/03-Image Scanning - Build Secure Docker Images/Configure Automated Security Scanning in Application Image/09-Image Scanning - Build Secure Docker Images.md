## Image Scanning - Build Secure Docker Images

### Introduction to Image Scanning

Image scanning is a crucial component of DevSecOps, ensuring that Docker images used in applications are free from known vulnerabilities and security risks. This process involves analyzing Docker images for potential security issues, such as vulnerabilities in the base image, dependencies, and secrets embedded within the image. By integrating automated security scanning into the build pipeline, organizations can proactively identify and mitigate these risks, enhancing the overall security posture of their applications.

### Configuring Trivy for Vulnerability Scanning

Trivy is an open-source tool designed to scan container images, including Docker images, for vulnerabilities. It supports various types of vulnerabilities, including CVEs (Common Vulnerabilities and Exposures), and can be configured to exit with a non-zero error code if critical vulnerabilities are found. This behavior ensures that the build process can be halted if serious security issues are detected.

#### Example Configuration

To configure Trivy to exit with a non-zero error code upon finding critical vulnerabilities, you can modify your build pipeline script. Below is an example using a `Dockerfile` and a `Jenkinsfile`:

```yaml
pipeline {
    agent any
    stages {
        stage('Build') {
            steps {
                script {
                    sh 'docker build -t myapp .'
                }
            }
        }
        stage('Scan') {
            steps {
                script {
                    sh 'trivy image --severity CRITICAL --exit-code 1 myapp'
                }
            }
        }
    }
}
```

In this example, the `trivy image` command is used to scan the Docker image named `myapp`. The `--severity CRITICAL` flag specifies that only critical vulnerabilities should trigger a non-zero exit code. The `--exit-code 1` flag ensures that the build process fails if any critical vulnerabilities are found.

### Allowing Pipeline Failure

While it is important to detect and address critical vulnerabilities, it may not always be desirable to fail the entire build pipeline due to these findings. In some cases, it might be more appropriate to visualize the issues without blocking the developer workflow. To achieve this, you can configure the pipeline to allow failures.

#### Example Configuration with Allow Failure

Here is an updated example that includes the `allowFailure` option:

```yaml
pipeline {
    agent any
    stages {
        stage('Build') {
            steps {
                script {
                    sh 'docker build -t myapp .'
                }
            }
        }
        stage('Scan') {
            steps {
                script {
                    sh 'trivy image --severity CRITICAL --exit-code 1 myapp'
                }
            }
            post {
                failure {
                    echo 'Vulnerabilities found, but allowing pipeline to continue.'
                }
            }
        }
    }
}
}
```

In this configuration, the `post` block is used to handle the case where the scan stage fails. The `failure` block allows the pipeline to continue even if vulnerabilities are found, providing feedback to the developers without halting the build process.

### Understanding Trivy Logs

When Trivy performs a scan, it generates detailed logs that provide insights into the scanning process. These logs include information about the vulnerabilities database used, the base image being scanned, and the dependencies within the image.

#### Example Log Output

Below is an example of the log output generated by Trivy during a scan:

```plaintext
2023-10-01T12:00:00Z     INFO    Detecting Debian vulnerabilities...
2023-10-01T12:00:01Z     INFO    Found Debian vulnerabilities
2023-10-01T12:00:01Z     INFO    Detecting Node.js vulnerabilities...
2023-10-01T12:00:02Z     INFO    Found Node.js vulnerabilities
2023-10-01T12:00:02Z     INFO    Detecting secrets...
2023-10-01T12:00:03Z     INFO    Found secrets
```

From the logs, you can see that Trivy is performing multiple types of scans, including vulnerability scanning of the base image and dependencies, as well as secret scanning. Each type of scan provides valuable information about the security status of the Docker image.

### Scanning Base Image and Dependencies

Trivy scans both the base image and the dependencies within the Docker image to ensure comprehensive coverage. The base image is the starting point for building the final image, and it is essential to verify that it does not contain any known vulnerabilities.

#### Example Base Image Scan

Consider a Docker image built using the `Debian` base image. Trivy will detect the base image and scan it for vulnerabilities. Here is an example of how Trivy detects and scans the base image:

```plaintext
2023-10-01T12:00:00Z     INFO    Detecting Debian vulnerabilities...
2023-10-01T12:00:01Z     INFO    Found Debian vulnerabilities
```

In this example, Trivy identifies that the base image is `Debian` and proceeds to scan it for vulnerabilities. This ensures that the foundation of the Docker image is secure.

#### Example Dependency Scan

Trivy also scans the dependencies within the Docker image, such as Node.js packages. Here is an example of how Trivy detects and scans these dependencies:

```plaintext
2023-10-01T12:00:01Z     INFO    Detecting Node.js vulnerabilities...
2023-10-01T12:00:02Z     INFO    Found Node.js vulnerabilities
```

In this example, Trivy identifies that the Docker image contains Node.js packages and proceeds to scan them for vulnerabilities. This ensures that the application's dependencies are secure.

### Summary of Scanning Results

After completing the scans, Trivy provides a summary of the findings. This summary includes details about the vulnerabilities detected in different aspects of the application.

#### Example Summary

Below is an example of the summary provided by Trivy:

```plaintext
2023-10-01T12:00:02Z     INFO    Summary of vulnerabilities:
```

This summary helps developers understand the security status of the Docker image and prioritize the remediation of identified vulnerabilities.

### Real-World Examples and Recent CVEs

To illustrate the importance of image scanning, consider recent real-world examples and CVEs where vulnerabilities in Docker images led to security breaches.

#### Example: CVE-2021-44228 (Log4Shell)

CVE-2021-44228, commonly known as Log4Shell, is a critical vulnerability in the Apache Log4j library. This vulnerability allowed attackers to execute arbitrary code on affected systems, leading to widespread exploitation. Many Docker images containing applications using Log4j were vulnerable to this attack.

By integrating Trivy into the build pipeline, organizations could have detected and addressed this vulnerability before deploying the Docker images, preventing potential breaches.

#### Example: CVE-2022-22965 (Spring Framework)

CVE-2022-22965 is a critical vulnerability in the Spring Framework, affecting applications using Spring Data REST. This vulnerability allowed attackers to execute arbitrary code on affected systems. Many Docker images containing Spring-based applications were vulnerable to this attack.

Again, by integrating Trivy into the build pipeline, organizations could have detected and addressed this vulnerability before deploying the Docker images, preventing potential breaches.

### How to Prevent / Defend

To effectively prevent and defend against vulnerabilities in Docker images, organizations should implement the following measures:

#### Detection

1. **Automated Scanning**: Integrate Trivy or similar tools into the build pipeline to automatically scan Docker images for vulnerabilities.
2. **Regular Updates**: Keep the vulnerabilities database up-to-date to ensure that the latest vulnerabilities are detected.

#### Prevention

1. **Secure Base Images**: Use secure base images that are regularly updated and patched.
2. **Dependency Management**: Manage dependencies carefully, ensuring that only the necessary and secure versions are included in the Docker image.

#### Secure Coding Fixes

Below is an example of a vulnerable Dockerfile and the corresponding secure version:

**Vulnerable Dockerfile**

```Dockerfile
FROM debian:latest
RUN apt-get update && apt-get install -y nodejs
COPY . /app
WORKDIR /app
RUN npm install
CMD ["node", "app.js"]
```

**Secure Dockerfile**

```Dockerfile
FROM debian:stable
RUN apt-get update && apt-get install -y nodejs
COPY . /app
WORKDIR /app
RUN npm install --production
CMD ["node", "app.js"]
```

In the secure version, the base image is explicitly set to `debian:stable`, ensuring that a stable and secure version is used. Additionally, the `npm install` command is modified to install only production dependencies, reducing the attack surface.

#### Configuration Hardening

1. **Limit Privileges**: Ensure that the Docker image runs with the least privileges necessary.
2. **Environment Variables**: Avoid embedding sensitive information in environment variables; use secure methods to manage secrets.

#### Mitigations

1. **Patch Management**: Regularly patch and update the Docker images to address known vulnerabilities.
2. **Monitoring**: Implement monitoring and logging to detect and respond to security incidents promptly.

### Conclusion

Image scanning is a vital practice in DevSecOps, ensuring that Docker images are free from known vulnerabilities and security risks. By integrating tools like Trivy into the build pipeline, organizations can proactively identify and mitigate these risks, enhancing the overall security posture of their applications. Through comprehensive scanning, detection, prevention, and mitigation strategies, organizations can build and deploy secure Docker images.
