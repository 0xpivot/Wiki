# RITA Real Intelligence Threat Analytics for C2

## Introduction to RITA
RITA (Real Intelligence Threat Analytics) is an open-source framework developed by Active Countermeasures. It is designed specifically to detect command and control (C2) communication, data exfiltration, and lateral movement by statistically analyzing network connection logs. Unlike traditional Intrusion Detection Systems (IDS) like Snort or Suricata that rely on signature matching against known malicious packet payloads, RITA takes a purely behavioral and mathematical approach. It does not look at payloads; it looks at the metadata of the communication patterns over time.

RITA is designed to ingest connection logs generated by Zeek (formerly Bro). By processing these logs, typically in 24-hour batches, RITA applies advanced statistical models, including k-means clustering and Fast Fourier Transforms (FFT), to identify anomalous behaviors that human analysts or traditional SIEM rules would easily miss, such as highly jittered beacons or low-and-slow DNS tunneling.

## Architecture and Workflow

RITA operates offline or in batch-processing mode rather than real-time streaming. This is because mathematical models like clustering require a complete dataset over a period (usually a day) to accurately baseline behavior and calculate standard deviations.

## Core Analytical Modules in RITA

RITA analyzes the ingested Zeek logs across several distinct dimensions:

### 1. Beaconing Detection (The Core Feature)
Adversaries use beacons to have compromised internal hosts check in with external C2 servers for new commands. To evade basic threshold alerting, attackers use "jitter"—randomizing the sleep time between check-ins. 
RITA detects beacons by analyzing:
- **Inter-Arrival Time (IAT):** The time difference between connections. RITA looks for consistency in these gaps.
- **Connection Duration:** How long the connections last. Consistent durations imply automated scripts.
- **Payload Size (Bytes In/Out):** Beacons often have a highly uniform size for the request and response.
- **K-Means Clustering:** RITA groups the connections and calculates a score from 0.000 to 1.000. A score above 0.8 typically indicates a highly mathematical, machine-driven beacon, even if jitter is applied.

### 2. Long Connections
Data exfiltration or interactive reverse shells often rely on long-duration, persistent TCP connections. RITA flags connections that remain open for abnormal amounts of time (e.g., hours or days), especially if they originate from user workstations to external unclassified IP addresses.

### 3. DNS Tunneling (Exploded DNS)
Attackers sometimes tunnel C2 traffic or exfiltrate data by encoding the payload into the subdomains of DNS queries (e.g., `A3f9B...data.malicious-domain.com`). RITA analyzes `dns.log` to find internal hosts making an unusually high volume of requests to a single domain or resolving excessively long, randomly generated subdomains.

### 4. Strobes (Port Scanning)
A strobe occurs when a single internal IP connects to multiple internal systems on a specific port (Horizontal Scanning) or scans multiple ports on a single system (Vertical Scanning). This is a prime indicator of lateral movement and internal reconnaissance.

### 5. Blacklist Checking
RITA cross-references the external IP addresses found in the Zeek logs against known threat intelligence blacklists, providing immediate context if an internal host is communicating with known malicious infrastructure.

## Threat Hunting with RITA: Interpreting the Score

When using RITA, the primary metric a threat hunter looks at is the **Score**. 
- **Score 1.000:** Absolute mathematical perfection. This is almost certainly an automated machine process. It could be malware, or it could be a legitimate but poorly configured NTP or update service.
- **Score 0.800 - 0.990:** Highly suspicious. Often represents a C2 beacon with moderate jitter configured by an adversary.
- **Score below 0.500:** Normal human web browsing behavior. Humans are erratic; they click links at random intervals, generating low scores.

**Hunting Workflow:**
1. Generate the RITA report for the last 24 hours.
2. Navigate to the `Beacons` section.
3. Filter out known legitimate services (e.g., Microsoft telemetry, WSUS servers, internal DNS polling). This is known as "whitelisting" or "baselining."
4. Investigate the remaining high-scoring source-destination pairs. Pivot to the firewall or SIEM to examine the specific ports, User-Agents, or JA3 hashes associated with the traffic.

## Real-World Attack Scenario

### The Scenario: Evasion via Jittered Cobalt Strike
An adversary successfully phished an employee and dropped a Cobalt Strike payload. Knowing the organization used basic SIEM rules that alerted on "more than 10 connections per hour to the same IP," the attacker configured their malleable C2 profile with a 45-minute sleep timer and 30% jitter. 

### The Attack Flow
1. **Execution:** The Cobalt Strike beacon executes and begins communicating with `198.51.100.99` over TCP port 443.
2. **Evasion:** Because of the 45-minute sleep and 30% jitter, check-ins occur randomly between roughly 31 minutes and 58 minutes. Over a 24-hour period, there are only about 30 total connections. This completely bypasses the SIEM threshold alerts.

### How RITA Detected the Activity
1. The security team ran their daily `rita import` and `rita analyze` cron jobs on the previous day's Zeek logs.
2. The threat hunter opened the RITA HTML report.
3. In the Beaconing tab, an internal IP (`10.0.50.112`) was flagged with a Beacon Score of `0.875` communicating with `198.51.100.99`.
4. The hunter clicked into the details. RITA's scatter plot visualization clearly showed that despite the jitter, the connections clustered tightly around a distinct mathematical mean (the 45-minute baseline), and the byte sizes (`bytes_out: 154`, `bytes_in: 322`) were identical across all 30 connections. 
5. The mathematical evidence of automation prompted the hunter to isolate the endpoint and pull the memory forensics, revealing the Cobalt Strike beacon.

## Using the RITA CLI

While the HTML report is visually appealing, advanced hunters often use the CLI for automation and integration.

**Importing Data:**
```bash
# Import a directory of Zeek logs and assign it a dataset name
rita import /opt/zeek/logs/2026-06-09/ MyCompany_June9
```

**Analyzing Data:**
```bash
# Run the analytics engine on the imported dataset
rita analyze MyCompany_June9
```

**Querying Beacons via CLI:**
```bash
# Display the top beacons, perfect for grepping or piping into other tools
rita show-beacons MyCompany_June9 | head -n 20
```
