# 15 - YARA Scanning over Memory Images

## 1. Introduction to Memory-Based YARA Scanning

YARA is often described as the "pattern-matching Swiss army knife for malware researchers." While it is extraordinarily effective at scanning files on disk, its utility is massively amplified when applied to physical memory (RAM). 

Scanning memory is fundamentally different from scanning files. On disk, malware is often packed, encrypted, or obfuscated. It may not exhibit any recognizable strings or byte sequences. However, for a CPU to execute code, the malware *must* eventually unpack or decrypt itself in memory. By scanning a memory dump, analysts can apply YARA rules directly to the decrypted, executing payload.

## 2. Challenges of Memory Scanning vs. Disk Scanning

Applying standard file-based YARA rules directly to a raw memory dump (`.raw` or `.vmem` file) often yields sub-optimal results due to the nature of operating system memory management.

### 2.1 Page Boundaries and Fragmentation
Physical memory is divided into pages (typically 4KB in size). A contiguous file on disk might be loaded into memory across dozens of fragmented, non-contiguous physical pages. 

If a YARA rule looks for a 16-byte signature that happens to straddle the boundary of two different pages, and those pages are mapped to entirely different physical addresses, a raw string scan will fail.

### 2.2 PE Alignment vs. Memory Alignment
When a Windows Executable (PE file) is loaded from disk into memory, its layout changes. On disk, sections (`.text`, `.data`) are aligned tightly based on the `FileAlignment` header (often 512 bytes). In memory, these sections are stretched and aligned based on the `SectionAlignment` header (often 4096 bytes).

YARA rules that rely on exact physical offsets (e.g., "String X must be exactly 1024 bytes after String Y") will match on disk but completely fail in memory. Memory-optimized YARA rules must use wildcards (`[0-4096]`) or logical distances to account for section stretching.

## 3. Using Volatility for YARA Scanning

To overcome fragmentation and alignment issues, investigators use Volatility plugins rather than scanning the raw `.raw` file with the standalone YARA engine.

Volatility's `windows.yarascan` (Vol3) parses the Operating System's memory structures (like the Virtual Address Descriptor - VAD, and Page Tables). It reconstructs the continuous virtual memory space of each process on-the-fly and applies the YARA rule to that reconstructed, contiguous space.

### 3.1 Constructing Memory-Optimized Rules

When writing rules specifically for memory, threat hunters focus on artifacts generated by memory-injection techniques (Reflective DLL Injection, Process Hollowing, Cobalt Strike Beacons).

```yara
rule CobaltStrike_Memory_Beacon {
    meta:
        description = "Detects unencrypted Cobalt Strike beacon configs in memory"
        author = "Threat Hunting Team"
    strings:
        // Hunting for the decrypted config block, often found in heap or private VADs
        $config_header = { 00 01 00 01 00 02 00 01 00 03 00 02 }
        $user_agent = "Mozilla/5.0 (Windows NT 10.0; Win64; x64)" ascii wide
        
        // Reflective loader stub often leaves traces of the MZ header or specific opcodes
        $reflective_stub = { 4D 5A 41 52 55 48 89 E5 48 83 EC 20 }
    condition:
        $config_header and $user_agent or $reflective_stub
}
```

## 4. Advanced Memory Hunting Techniques

### 4.1 Targeting Specific Memory Regions (VADs)
Scanning the entire 32GB RAM of a modern server takes time. Advanced hunters narrow their scope by targeting specific types of memory allocations.

Injected malware usually resides in memory regions with specific protection flags, particularly `PAGE_EXECUTE_READWRITE` (RWX). Volatility can filter YARA scans to only look inside VADs with RWX permissions, drastically speeding up the scan and reducing false positives from cache or unallocated memory.

### 4.2 Hunting Fileless Malware and Beacons
Fileless malware, like Cobalt Strike or Metasploit Meterpreter, never touches the disk. It exists purely as a thread executing inside a legitimate process (like `explorer.exe` or `svchost.exe`). 

By scanning the process memory of `svchost.exe` with YARA rules targeting the decrypted payload, analysts can pinpoint the exact thread and memory address containing the malicious C2 beacon.

## 6. Real-World Attack Scenario

### 6.1 The Breach
An organization's SIEM alerts on suspicious outbound DNS traffic, indicative of a Cobalt Strike DNS beacon. EDR scans of the entire disk turn up completely clean. The attackers utilized a PowerShell "living off the land" technique to reflectively inject a fileless beacon directly into memory.

### 6.2 The Investigation
The incident response team captures a full physical memory dump of the suspected workstation. They know the malware is memory-resident.

They craft a highly specific YARA rule targeting the decrypted Cobalt Strike beacon configuration block (looking for specific Malleable C2 profiles and sleep time variables).

```bash
# Running Volatility 3 with the YARA plugin across all processes
vol -f WIN10-MEM.raw windows.yarascan --yara-file ./cobalt_strike.yar
```

### 6.3 The Extraction
The Volatility output indicates a positive YARA match inside `PID 4012`, which is a legitimate instance of `notepad.exe` running in the background.

```
Process: notepad.exe (4012)
Offset: 0x000000000A4B0000
Rule: CobaltStrike_Memory_Beacon
Hexdump:
0x0A4B0000  4d 5a 90 00 03 00 00 00 04 00 00 00 ff ff 00 00   MZ..............
0x0A4B0010  ...
```

The YARA scan points exactly to the base address (`0x0A4B0000`) of the injected payload. The investigators then pivot and use `windows.vaddump` or `windows.malfind` to dump the PE file starting at that exact memory address. By extracting the binary, they reverse-engineer the configuration, revealing the attacker's full command-and-control infrastructure and encryption keys, neutralizing the threat.

## 7. Extended Rule Capabilities: Thread Injection and Hollow Processes

### 7.1 Detecting API Unhooking and Syscall Stubs
Advanced malware often unhooks EDR monitored APIs (like `NtReadVirtualMemory`) in memory. A YARA rule can be designed to scan memory for the original syscall stubs, comparing them against hooked stubs.
```yara
rule EDR_Syscall_Unhook {
    meta:
        description = "Detects manually mapped clean syscall stubs in memory"
    strings:
        // 'mov r10, rcx; mov eax, <syscall_number>; syscall' pattern
        $syscall_pattern = { 4C 8B D1 B8 ?? ?? ?? ?? 0F 05 }
    condition:
        $syscall_pattern
}
```
By deploying this rule via Volatility across memory, analysts can identify hollowed processes or heavily obfuscated malware frameworks that bring their own clean copy of `ntdll.dll` into memory.

### 7.2 Writing Resilient YARA Memory Rules
A highly resilient rule does not just look for strings; it combines byte constraints, entropy requirements, and structural checks (like identifying valid PE export directory headers mapped dynamically in the heap). 
Memory rules should leverage the `math.entropy` module within YARA to identify blocks of highly compressed or encrypted staging shellcode adjacent to recognizable reflective loader stubs.

## 8. Massive YARA Rule Repository for Memory Hunts
When deploying YARA at scale across terabytes of enterprise memory dumps, maintaining an optimized rule repository is critical. The following rules represent the foundation of memory-centric hunting:

```yara
rule Generic_Reflective_Loader_Memory {
    meta:
        description = "Hunts for common Reflective DLL injection prologue in memory"
        impact = "High - indicates memory injection"
    strings:
        // push rbp; mov rbp, rsp; and rsp, -0x10; sub rsp, 0x20 
        $prologue_64 = { 55 48 89 E5 48 83 E4 F0 48 83 EC 20 }
        // "MZ" header offset dynamically calculated
        $mz = "MZ"
    condition:
        $mz in (0..1024) and $prologue_64
}

rule Hidden_Kernel_Module_Memory {
    meta:
        description = "Searches for unlinked Linux kernel module structures"
        os = "Linux"
    strings:
        // Module init state magic bytes and common names
        $magic = { 7F 45 4C 46 02 01 01 00 }
        $diamorphine = "diamorphine" ascii
        $reptile = "reptile_module" ascii
    condition:
        $magic at 0 and any of ($diamorphine, $reptile)
}
```
Applying these rules directly with `yarascan` allows defenders to sweep the entire environment for the exact state of execution.

## 9. Memory Alignment Mathematical Proofs
To understand why disk YARA fails in memory, consider the mathematics of PE Alignment.
- **Disk:** `FileAlignment = 0x200` (512 bytes). Section `.data` starts at offset `0x400` on disk.
- **Memory:** `SectionAlignment = 0x1000` (4096 bytes). Section `.data` starts at offset `0x2000` in virtual memory.

If a YARA rule specifies `$string1 in (0x400..0x600)`, it mathematically *cannot* match in memory because the OS loader has padded the PE sections with null bytes to fit the 4KB memory page granularity. The string has shifted to `0x2000`. Memory YARA rules must calculate distances based on `SectionAlignment` multiples or rely exclusively on wildcard jumps between sections.
