# Correlating Cloud Identity with Network Activity

In traditional on-premises networks, threat hunting relies heavily on analyzing IP addresses, MAC addresses, network segments, and perimeter firewall logs. In modern cloud environments, this paradigm is entirely broken. IP addresses are highly ephemeral, workloads scale dynamically in and out of existence, and the true perimeter is no longer the network firewall—it is Identity and Access Management (IAM). 

Therefore, isolating a sophisticated threat in the cloud requires correlating Cloud Control Plane identity telemetry (e.g., AWS CloudTrail, Azure AD Sign-in logs, GCP Audit Logs) with Cloud Data Plane network telemetry (e.g., VPC Flow Logs, NSG Flow Logs, DNS logs).

Without this deep correlation, a threat hunter merely sees a malicious IP connection but has absolutely no context regarding which user, service account, or specific workload triggered it, rendering response impossible.

## The Correlation Challenge

Cloud infrastructure is defined by extreme abstraction. An EC2 instance, an Azure Virtual Machine, or a GCP Compute Engine instance possesses an IP address. However, it also inherently possesses an associated IAM Role (via an Instance Profile or Managed Identity) which defines what it is authorized to do within the cloud fabric.

When an advanced attacker compromises a workload, they inherit the identity (the IAM privileges) of that workload. To hunt effectively, we must link three distinct dimensions:
1. **The Network Entity**: Source IP, Destination IP, Port, Protocol, TCP Flags, Bytes Transferred.
2. **The Cloud Resource**: Instance ID, Network Interface ID (ENI), Subnet ID, VPC ID, Security Group.
3. **The Identity**: The IAM Role attached to the resource, the Principal ID, and the specific, temporary session credentials generated by the Security Token Service (STS).

## Architecture of Cloud Telemetry Correlation

## Threat Hunting Methodologies

### 1. Hunting for Exfiltration by Compromised Workloads

**Hypothesis:** A cloud workload has been compromised (e.g., via a zero-day RCE) and is actively exfiltrating sensitive data to an external command-and-control (C2) server or a public data dump site.

**Hunting Workflow:**
1. **Identify Anomalous Network Spikes:** Query VPC Flow Logs for "top talkers" sorting by `bytes_out` directed to public IP ranges. Filter out known, legitimate SaaS/PaaS endpoints (like Microsoft 365, AWS APIs, Datadog agents).
2. **Resolve to Cloud Resource:** Map the offending source IP address to a specific Elastic Network Interface (ENI), and subsequently to an EC2 Instance, a container node, or a Lambda function `[[12 - Serverless Function Lambda Abuse Detection]]`.
3. **Identify Attached Role:** Query the cloud inventory to determine the IAM role attached to that instance profile at the time of the network event.
4. **Correlate with API Activity:** Query CloudTrail for data plane operations (like S3 `GetObject`, DynamoDB `Scan`, or RDS snapshot exports) executed by that specific IAM role during the exact same time window.

**Resulting Hunt Conclusion:** You discover that internal IP `10.0.2.14` transferred 50GB over port 443 to an unknown ASN. The instance belongs to the `AppServer-Role`. CloudTrail shows that `AppServer-Role` downloaded 50GB from the `customer-database-backup` bucket just minutes prior. This confirms deliberate data exfiltration, rather than just heavy application network usage.

### 2. Spotting SSRF to Instance Metadata Service (IMDS)

Server-Side Request Forgery (SSRF) is the most common technique used by attackers to access the local Instance Metadata Service (IMDS at `169.254.169.254`) and steal the temporary IAM credentials assigned to the workload.

**Hunting Workflow:**
1. **Network Anomaly:** While standard VPC Flow logs often do not log traffic to the IMDS link-local address, Host-based logs (Syslog/Windows Event Logs), WAF logs, or proxy logs might show incoming HTTP requests with a destination URI containing `/latest/meta-data/iam/security-credentials/`.
2. **Identity Anomaly:** Monitor CloudTrail for the usage of these STS tokens. The critical hunt is looking for the *execution* of stolen credentials outside the perimeter.
3. **The Correlation Rule:** When an STS token (stolen via SSRF) is used from an external IP (the attacker's machine), CloudTrail logs will show the `userIdentity.arn` as the internal Instance Role, but the `sourceIPAddress` will NOT match the instance's public Elastic IP or the VPC's NAT Gateway IP.

**Splunk SPL Detection Logic for Stolen Token Usage:**
```spl
index=aws_cloudtrail user_type=AssumedRole 
| search user_arn="*assumed-role/EC2-Instance-Role*"
| join type=outer sourceIPAddress [
    search index=aws_vpc_flow_logs 
    | stats count by nat_gateway_ip 
    | rename nat_gateway_ip as sourceIPAddress 
    | eval is_internal=1
]
| fillnull value=0 is_internal
| where is_internal==0 AND NOT match(sourceIPAddress, "^.*\.amazonaws\.com$")
| table _time, eventName, user_arn, sourceIPAddress, userAgent
```
*(This query alerts if an assumed role is used from an IP that is neither a recognized internal NAT IP nor an internal AWS service IP).*

### 3. Detecting Cryptomining and Lateral Movement

**Hypothesis:** An attacker has deployed a cryptominer and is aggressively scanning the internal VPC for other vulnerable hosts (Lateral Movement).

**Hunting Workflow:**
1. **Network Anomaly:** VPC Flow logs show a single instance making tens of thousands of connections on port 22 (SSH), 3389 (RDP), or 445 (SMB) to adjacent internal subnets. Look specifically for `REJECT` actions in the Flow Logs, which strongly denote aggressive network scanning against closed ports or blocked security groups.
2. **Network Protocol Anomaly:** Outbound connections on common cryptomining ports (3333, 14444) or DNS logs showing queries for known mining pools (`minexmr.com`).
3. **Identity Correlation:** Investigate the identity lifecycle. Did the identity associated with this instance recently get modified? Look in CloudTrail for `AttachRolePolicy` or Azure Activity Log for `Role Assignment` changes that granted the instance new permissions just prior to the massive network scanning activity. The attacker may have escalated privileges before pivoting.

## Real-World Attack Scenario

### Scenario: The Stolen Token Pivot via AKS

1. **Initial Access**: An attacker exploits a vulnerable Apache Struts application running on an Azure Kubernetes Service (AKS) cluster node `[[13 - Hunting in Kubernetes Cluster Audit Logs]]`.
2. **Credential Theft**: They extract the Azure Managed Identity token from the pod's environment variables or local volume mounts.
3. **Network Activity**: The attacker establishes an encrypted reverse proxy (using a tool like Chisel) tunneling out of the pod to their C2 server.
4. **Action on Objectives**: From their local machine, the attacker utilizes the stolen Azure Managed Identity token to run `Get-AzStorageAccountKey` against an administrative Azure Storage account, subsequently dumping files `[[11 - Identifying Anomalous Cloud Storage Access Buckets]]`.

**Hunter's Response:**
- The network hunting dashboard detects an anomalous, long-lived, high-bandwidth TCP connection originating from the AKS node subnet outbound to a foreign IP with no historical reputation.
- The identity hunting dashboard detects `ListKeys` operations in the Azure Activity log originating from a non-Azure, residential IP address.
- By correlating the exact timestamps and matching the specific Managed Identity assigned to that AKS node pool, the threat hunter definitively links the outbound C2 network connection to the external API abuse, mapping out the complete kill chain.

## Tooling and Best Practices

To make this complex correlation automated and scalable, organizations must build a robust threat hunting pipeline `[[15 - Building a Cloud Native Threat Hunting Pipeline]]`.
- **Enrichment at Ingestion**: When raw VPC Flow Logs are ingested into the SIEM, use a Lambda function or Logstash pipeline to dynamically query the cloud API and enrich the log payload with the `InstanceId`, `IAM_Role`, and `Environment_Tags`. Do this *before* it hits the storage layer.
- **Graph Databases**: Use graph visualization tools (like AWS Detective, Azure Sentinel Investigation Graph, or custom Neo4j setups) to visually map and explore the complex, non-linear relationships between IPs, ENIs, Instances, Roles, and human Users.
