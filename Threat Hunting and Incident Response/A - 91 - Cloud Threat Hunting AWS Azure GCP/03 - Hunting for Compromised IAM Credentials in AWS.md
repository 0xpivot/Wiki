# Hunting for Compromised IAM Credentials in AWS

## 1. Introduction and Executive Summary

In the cloud, credentials are the keys to the kingdom. Unlike traditional networks where compromising a VPN might require multifactor authentication and network traversal, compromised AWS Identity and Access Management (IAM) credentials often provide immediate, direct access to the management plane from anywhere on the internet. 

Credential compromise is consistently ranked as the top vector for cloud breaches. Attackers actively scrape public GitHub repositories, exploit Server-Side Request Forgery (SSRF) vulnerabilities, and utilize phishing campaigns to steal Access Keys, Secret Access Keys, and Session Tokens. This document explores the anatomy of AWS credentials, how they leak, and deep-dive methodologies for hunting compromised credentials using AWS logs.

## 2. Anatomy of AWS Credentials

Understanding the structure of AWS credentials helps in identifying and triaging leaks.

1. **Long-term Credentials (IAM Users):**
   - **Access Key ID:** Starts with `AKIA` (e.g., `AKIAIOSFODNN7EXAMPLE`).
   - **Secret Access Key:** A 40-character base64-encoded string.
   - *Characteristics:* These never expire unless manually rotated or deleted. They are extremely dangerous if leaked.

2. **Temporary Credentials (STS / IAM Roles):**
   - **Access Key ID:** Starts with `ASIA` (e.g., `ASIAIOSFODNN7EXAMPLE`).
   - **Secret Access Key:** A 40-character base64-encoded string.
   - **Session Token:** A long base64 string that must be passed alongside the keys.
   - *Characteristics:* These are generated by the AWS Security Token Service (STS). They have a set expiration time (from 15 minutes up to 36 hours). They are dynamically assigned to EC2 instances (via Instance Profiles), AWS Lambda, or users assuming a role.

## 3. How Credentials Leak

To hunt effectively, one must understand the ingestion vectors for attackers:
- **Source Code Repositories:** Developers hardcode `AKIA` keys into scripts and push them to public GitHub repos. Attackers use automated scanners (like Gitrob or TruffleHog) to pull these within seconds of a commit.
- **SSRF (Server-Side Request Forgery):** As seen in the Capital One breach, an attacker queries the EC2 Instance Metadata Service (IMDS) at `http://169.254.169.254` to steal the `ASIA` temporary credentials of the instance.
- **Local File Inclusion (LFI) / Path Traversal:** Attackers read the `~/.aws/credentials` file on a compromised developer workstation or server.
- **Environment Variables:** Application vulnerability leading to the disclosure of environment variables where `AWS_ACCESS_KEY_ID` is stored.

## 5. Hunting Methodologies and Indicators of Compromise (IoCs)

When an attacker obtains a credential, they rarely know what permissions it holds. Their first actions will invariably be reconnaissance, which is highly visible in CloudTrail.

### 5.1 The "Whoami" of AWS
The very first API call an attacker makes is almost always `sts:GetCallerIdentity`. This API tells the caller their ARN, User ID, and Account ID. While legitimate tools occasionally call this, a sudden spike, or a call from an unusual IP address, is a massive red flag.

### 5.2 Enumeration and Error Spikes (AccessDenied)
Because the attacker doesn't know their limits, they will use tools like `Pacu`, `Enumerate IAM`, or `Cloudsplaining`. These tools blast the AWS API with `List*` and `Describe*` calls across multiple regions.
Since the compromised credential likely lacks permissions for *everything*, this generates a storm of `AccessDenied` or `UnauthorizedOperation` errors in CloudTrail.

**Hunting Metric:** Monitor the ratio of successful API calls to `AccessDenied` errors per IAM Principal. A sudden spike in the error ratio indicates automated enumeration.

### 5.3 Geographic and ASN Anomalies (Impossible Travel)
If an `AKIA` key belongs to a developer who works in New York, and that key suddenly makes an API call from a VPN exit node in Amsterdam, this is an anomaly.
Similarly, if an `ASIA` key belongs to an EC2 instance in `us-west-2` (Oregon), the source IP of API calls using that key *must* belong to the AWS IP space for that region. If an `ASIA` key assigned to an EC2 instance makes an API call from an external ISP (e.g., Comcast, DigitalOcean), it means the key was exfiltrated via SSRF/RCE.

## 6. Real-World Attack Scenario

### The Scenario: The GitHub Leak to Ransomware
1. **The Leak:** A junior data scientist writes a Python script to pull data from S3. They hardcode their `AKIA` credentials into the script and push it to a public repository to ask for help on StackOverflow.
2. **The Discovery:** Within 4 seconds, a bot scraping GitHub via webhooks detects the `AKIA` prefix. The bot automatically sends the key to the attacker's C2 server.
3. **Reconnaissance:** The attacker runs `aws sts get-caller-identity --profile stolen`. They discover they are the `data-sci-user`.
4. **Enumeration:** The attacker runs Pacu to enumerate permissions. They receive hundreds of `AccessDenied` errors but discover they have `s3:ListAllMyBuckets` and `s3:GetObject`.
5. **Impact:** The attacker downloads sensitive customer data, deletes the original S3 buckets (if they have `DeleteObject` permission), and leaves a ransom note via a new S3 object.

## 7. Operationalizing the Hunt: Advanced Queries

### Athena SQL: Hunting for EC2 Role Exfiltration (SSRF Detection)
This is arguably the most important query for AWS threat hunting. It detects when temporary credentials meant for an EC2 instance are used from outside the AWS network.

```sql
SELECT 
    eventTime,
    userIdentity.arn,
    eventName,
    sourceIPAddress,
    userAgent
FROM cloudtrail_logs
WHERE userIdentity.type = 'AssumedRole'
-- Look for role sessions that look like EC2 instances (typically i-xxxxxx)
AND userIdentity.principalId LIKE '%:i-%'
-- Check if the IP is outside of known AWS IP ranges (requires a custom UDF or join table of AWS IPs, simplified here)
AND sourceIPAddress NOT LIKE '10.%'
AND sourceIPAddress NOT LIKE '172.16.%'
AND sourceIPAddress NOT LIKE '192.168.%'
-- Assuming external IP format implies exfiltration if it doesn't match your NAT Gateway IPs
AND sourceIPAddress NOT IN ('<YOUR_NAT_GATEWAY_IP_1>', '<YOUR_NAT_GATEWAY_IP_2>')
ORDER BY eventTime DESC;
```

### Splunk: Hunting for Enumeration Error Spikes
This query identifies users who are suddenly generating a massive amount of authorization failures, indicative of an attacker blindly probing.

```splunk
index=aws_cloudtrail errorCode=AccessDenied OR errorCode=UnauthorizedOperation
| bucket _time span=1h
| stats count as ErrorCount by userIdentity.arn, sourceIPAddress, _time
| eventstats avg(ErrorCount) as AvgErrors, stdev(ErrorCount) as StdevErrors by userIdentity.arn
| eval AnomalyThreshold = AvgErrors + (3 * StdevErrors)
| where ErrorCount > AnomalyThreshold AND ErrorCount > 50
| sort - ErrorCount
```

## 8. Mitigation and Remediation
- **Disable Long-Term Credentials:** Shift entirely to SSO (AWS IAM Identity Center / Single Sign-On). Humans should not have `AKIA` keys.
- **IMDSv2:** Enforce Instance Metadata Service Version 2 (IMDSv2) on all EC2 instances. IMDSv2 requires a `PUT` request with a specific header to retrieve a session token, which breaks 99% of SSRF attack vectors.
- **AWS Secrets Manager:** Developers should retrieve secrets programmatically at runtime rather than storing them in code or environment variables.
- **Git Secrets Scanning:** Implement pre-commit hooks (e.g., `git-secrets`) and pipeline checks to prevent credentials from ever being pushed to a repository.
