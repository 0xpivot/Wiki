# AWS CloudTrail Analysis for Persistence

## 1. Introduction to CloudTrail
AWS CloudTrail is the foundational logging service for the AWS ecosystem. It provides a comprehensive history of AWS API calls for an account, including API calls made via the AWS Management Console, AWS SDKs, command-line tools, and higher-level AWS services. For a threat hunter, CloudTrail is the equivalent of the Windows Security Event Log, but for the entire datacenter infrastructure.

When adversaries breach an AWS environment, their primary objective after initial reconnaissance is often to establish persistence. They want to ensure that even if the initially compromised credential is revoked, or the vulnerable EC2 instance is terminated, they retain access to the control plane. 

CloudTrail is the exact telemetry source required to detect these persistence mechanisms.

## 2. CloudTrail Log Types
To hunt effectively, one must understand the types of events CloudTrail records:
1. **Management Events (Control Plane Operations):** These track operations that are performed on resources in your AWS account. Examples include configuring security groups, registering devices, routing tables, and setting up IAM rules. By default, CloudTrail records management events.
2. **Data Events (Data Plane Operations):** These track resource operations performed *on or within* a resource. Examples include S3 object-level API activity (e.g., `GetObject`, `PutObject`) or AWS Lambda function execution (`Invoke`). These are high-volume and are NOT enabled by default due to cost.
3. **Insights Events:** Anomaly detection events generated by AWS when unusual API call volumes or error rates occur.

For persistence hunting, **Management Events** are the primary focus.

## 3. Persistence Mechanisms in AWS
Adversaries can persist via Identity and Access Management (IAM) or by backdooring compute resources. 

### 3.1 IAM-Based Persistence
- **Creating new IAM Users:** `CreateUser`
- **Creating new Access Keys:** `CreateAccessKey` (for an existing, legitimate user, bypassing password resets).
- **Console Passwords:** `CreateLoginProfile` or `UpdateLoginProfile` (adding console access to a programmatic-only user).
- **Modifying Trust Policies:** `UpdateAssumeRolePolicy` (modifying a role's trust relationship so the attacker's external AWS account can assume it).
- **Inline/Attached Policies:** `PutUserPolicy`, `AttachUserPolicy` (escalating privileges of a compromised low-privileged user).

### 3.2 Infrastructure/Resource-Based Persistence
- **EC2 UserData:** `ModifyInstanceAttribute` (injecting a reverse shell payload into the UserData of an existing, stopped EC2 instance, which executes upon the next reboot).
- **Lambda Backdoors:** `UpdateFunctionCode` (injecting malicious code into an existing legitimate Lambda function) or `AddPermission` (allowing external invocation of a Lambda).
- **AMI Sharing:** `ModifyImageAttribute` (sharing a snapshot or AMI containing sensitive data with an external, attacker-controlled AWS account).
- **Security Group Modifications:** `AuthorizeSecurityGroupIngress` (opening port 22 or 3389 to `0.0.0.0/0` for SSH/RDP access).

## 5. Deep Dive into Specific API Calls

### 5.1 CreateAccessKey
When an attacker compromises a developer's AWS CLI credentials, their first action is often to generate a *second* set of access keys for that same user. If the SOC detects the breach and revokes the *first* set of keys, the attacker still maintains access via the second set.

**Detection Strategy:** Look for `CreateAccessKey` events where the user generating the key is different from the target user, or when multiple access keys are generated in a short timeframe. AWS allows a maximum of 2 active access keys per IAM user.

### 5.2 UpdateAssumeRolePolicy (The Cross-Account Backdoor)
IAM Roles do not have permanent credentials. They are assumed by entities defined in their "Trust Policy" (AssumeRolePolicyDocument). An attacker can modify an existing, highly privileged role to trust their own personal AWS account. 

**Log Snippet (CloudTrail):**
```json
{
  "eventName": "UpdateAssumeRolePolicy",
  "requestParameters": {
    "roleName": "Production-Admin-Role",
    "policyDocument": "{\"Version\":\"2012-10-17\",\"Statement\":[{\"Effect\":\"Allow\",\"Principal\":{\"AWS\":\"arn:aws:iam::999999999999:root\"},\"Action\":\"sts:AssumeRole\"}]}"
  }
}
```
*Notice the `999999999999` account ID. This is the attacker's AWS account. They can now assume the `Production-Admin-Role` at any time from their own account.*

## 6. Real-World Attack Scenario

### 6.1 The Scenario: The Lambda Persistence Backdoor
An attacker successfully phishes an engineering manager, obtaining an AWS session token via a malicious OAuth app or AiTM attack. The token is valid for 12 hours.

Knowing the token will expire, the attacker seeks persistence. Instead of creating a noisy IAM user (which GuardDuty might flag), the attacker looks for an existing AWS Lambda function that runs on a schedule (e.g., a daily cleanup script).

1. The attacker calls `GetFunction` to download the legitimate Lambda code.
2. The attacker modifies the Python code locally, adding a backdoor that checks for a specific HTTP header and executes arbitrary commands via `os.system()`.
3. The attacker calls `UpdateFunctionCode` to deploy the modified code back to AWS.
4. The attacker calls `AddPermission` to make the Lambda function publicly invocable via Function URLs or API Gateway.

### 6.2 The Hunting Perspective
A threat hunter analyzing CloudTrail should look for anomalous modifications to Lambda functions, especially by users who do not normally deploy code (or outside of standard CI/CD pipelines).

**Athena Query for Lambda Backdoor modifications:**
```sql
SELECT 
    eventTime, 
    userIdentity.arn, 
    eventName, 
    requestParameters, 
    sourceIPAddress
FROM cloudtrail_logs
WHERE eventSource = 'lambda.amazonaws.com'
AND eventName IN ('UpdateFunctionCode', 'AddPermission', 'CreateFunctionUrlConfig')
AND userIdentity.arn NOT LIKE '%Jenkins-Role%' -- Exclude legitimate CI/CD
ORDER BY eventTime DESC;
```

## 7. Operationalizing the Hunt: Advanced Queries

### Hunting for IAM Backdoors (Athena SQL)
This query identifies critical IAM modifications that establish persistence.

```sql
SELECT 
    eventTime,
    userIdentity.type,
    userIdentity.arn AS actor,
    eventName,
    requestParameters,
    sourceIPAddress,
    userAgent
FROM cloudtrail_logs
WHERE eventName IN (
    'CreateAccessKey',
    'CreateLoginProfile',
    'CreateUser',
    'UpdateAssumeRolePolicy',
    'AttachUserPolicy',
    'PutUserPolicy',
    'CreateVirtualMFADevice'
)
-- Filter out known acceptable automation
AND userIdentity.arn NOT LIKE '%Terraform-Service-Role%'
ORDER BY eventTime DESC;
```

### Hunting for Infrastructure Sharing (Splunk)
This searches for an attacker attempting to share EC2 snapshots or AMIs with external accounts to exfiltrate data persistently.

```splunk
index=aws_cloudtrail eventName=ModifySnapshotAttribute OR eventName=ModifyImageAttribute
| spath requestParameters.createVolumePermission.add.items{}.userId output=target_account
| search target_account="*"
| eval is_internal_account=if(match(target_account, "^(111111111111|222222222222)$"), "True", "False")
| search is_internal_account="False"
| table _time, eventName, userIdentity.arn, target_account, sourceIPAddress
```

## 8. Mitigation and Remediation
- **Least Privilege:** Ensure users and CI/CD roles do not have `iam:CreateAccessKey` or `iam:UpdateAssumeRolePolicy` unless strictly necessary.
- **Service Control Policies (SCPs):** Use AWS Organizations SCPs to explicitly deny the ability to modify critical roles or share resources externally, regardless of the IAM user's policy.
- **Continuous Monitoring:** Alert immediately on `CreateAccessKey` events and verify them against an internal ticketing system.
