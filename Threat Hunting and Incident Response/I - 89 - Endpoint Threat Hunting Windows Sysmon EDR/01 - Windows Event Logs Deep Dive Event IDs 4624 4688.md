# Windows Event Logs Deep Dive: Event IDs 4624 & 4688

## 1. Introduction

Windows Event Logs remain the absolute cornerstone of endpoint threat hunting, digital forensics, and incident response (DFIR). Among the hundreds of event IDs generated by the Windows Security Log, Event IDs 4624 (Successful Logon) and 4688 (Process Creation) provide the most critical and fundamental telemetry for tracking adversary movement, execution, and persistence across an enterprise environment. 

By analyzing these two events in tandem, threat hunters can reconstruct precise attack timelines, trace lateral movement across a network, and uncover unauthorized code execution that bypasses traditional antivirus solutions. This note provides an extensive and comprehensive deep dive into the inner workings of Event IDs 4624 and 4688. We will highlight their specific fields, telemetry nuances, how attackers attempt to manipulate or bypass these logs, and advanced hunting strategies to identify malicious behavior.

## 2. Event ID 4624: An Account Was Successfully Logged On

Event ID 4624 is generated by the Local Security Authority (LSA) every time an account successfully authenticates to a Windows machine. It contains critical details about the user, the origin of the authentication request, the protocol used, and the type of logon performed.

### 2.1 Key Fields in Event ID 4624

The telemetry provided by Event 4624 is vast. The most critical fields for threat hunting include:

1. **Logon Type**: This is arguably the single most important field in a 4624 event. It specifies exactly *how* the user logged on.
    - **Type 2 (Interactive)**: A user physically logged on at the keyboard and screen of the system.
    - **Type 3 (Network)**: A user or computer logged on from the network (e.g., accessing a shared folder, RPC, WMI, or PsExec). Attackers frequently use Type 3 for lateral movement via Pass-the-Hash (PtH).
    - **Type 4 (Batch)**: Used by scheduled tasks. Attackers establishing persistence via Windows Scheduled Tasks will trigger Type 4 logons when the task executes.
    - **Type 5 (Service)**: Used by services starting up. Malicious services created by attackers (e.g., via Service Control Manager) will trigger this.
    - **Type 7 (Unlock)**: The workstation was unlocked from a locked state.
    - **Type 8 (NetworkCleartext)**: Logon with network credentials sent in the clear (often seen with basic IIS authentication or some legacy, insecure protocols).
    - **Type 9 (NewCredentials)**: Used when a user clones their token to run a process as another user without authenticating immediately (e.g., `RunAs /netonly`). This is a massive indicator of Pass-the-Hash (PtH) attacks using tools like Mimikatz.
    - **Type 10 (RemoteInteractive)**: Terminal Services, Remote Desktop Protocol (RDP). Crucial for tracking RDP-based lateral movement.
    - **Type 11 (CachedInteractive)**: A user logged on with cached credentials because the domain controller was unreachable.

2. **Logon Process**: Indicates the trusted process that submitted the logon request (e.g., User32, Advapi).
3. **Authentication Package**: The protocol used (e.g., Kerberos, NTLM, Negotiate). If NTLM is seen in an environment where Kerberos is enforced, it may indicate malicious fallback or PtH.
4. **TargetUserName / TargetDomainName**: The specific account that was logged into.
5. **IpAddress / IpPort**: The source IP and port from which the network logon originated.
6. **Logon GUID**: Can be correlated with other events (like Kerberos TGT requests in Event 4768) if Kerberos authentication is used.

### 2.2 Threat Hunting Focus for 4624

When hunting through 4624 events, defenders should look for:
- **Anomalous Logon Types**: Correlate Logon Type 10 (RDP) occurring outside of normal business hours, or Type 3 (Network) originating from unusual jump hosts or VPN IP spaces.
- **Service Accounts Performing Interactive Logons**: Service accounts typically use Logon Type 5. If a service account exhibits a Type 2 or Type 10 logon, it strongly suggests the account credentials have been compromised and are being used interactively by an attacker.
- **Type 9 Logons**: Extremely rare in normal administration unless specific tools are used. Highly indicative of credential injection and Pass-the-Hash (PtH).

## 3. Event ID 4688: A New Process Has Been Created

Event 4688 records every time a new process starts on the system. It tracks what was executed, by whom, from where, and (if properly configured) exactly what command line arguments were passed.

### 3.1 Key Fields in Event ID 4688

1. **New Process Name**: The full path to the executable (e.g., `C:\Windows\System32\cmd.exe`). Attackers often run processes from unusual or writeable directories like `C:\PerfLogs\`, `C:\Users\Public\`, or `C:\ProgramData\`.
2. **Creator Process Name (Parent Process)**: The process that spawned the new process. Parent-Child relationships are vital for identifying malicious chains (e.g., `winword.exe` spawning `powershell.exe`).
3. **Process Command Line**: (Must be enabled via GPO). Shows the exact arguments passed to the executable. This is the single most valuable piece of forensic data for understanding attacker intent (e.g., seeing `powershell.exe -nop -w hidden -enc...`).
4. **SubjectUserName**: The account context under which the new process was spawned.
5. **Token Elevation Type**: Indicates if the process was launched with standard privileges, elevated privileges (UAC bypass), or as the default administrator.
    - **Type 1**: Default administrator or UAC is disabled.
    - **Type 2**: Elevated via UAC prompt (Full token).
    - **Type 3**: Limited token (Standard user context).

### 3.2 Threat Hunting Focus for 4688

- **Parent-Child Anomalies**: `services.exe` should only spawn known, registered service binaries. If `services.exe` spawns `cmd.exe` or `powershell.exe`, it strongly indicates a malicious service execution (like PsExec or a Cobalt Strike service payload).
- **Suspicious Command Lines**: Look for Base64 encoded strings, network download commands (`Invoke-WebRequest`, `certutil -urlcache`), or execution originating from user temp folders.
- **Masquerading**: A process named `svchost.exe` running from `C:\Temp\` instead of `C:\Windows\System32\` is a classic masquerading technique.

## 5. Real-World Attack Scenario

### Lateral Movement via PsExec and Pass-the-Hash
An attacker compromises a workstation (Workstation A), bypasses UAC, dumps credentials using Mimikatz, and obtains the NTLM hash for a Domain Administrator. They use PsExec to move laterally to the Domain Controller (DC) to dump the NTDS.dit file.

1. **On Workstation A**:
   - **Event 4688**: The attacker executes `mimikatz.exe` to run `sekurlsa::pth`.
   - **Event 4624**: A Logon Type 9 (NewCredentials) is logged on Workstation A. This event indicates that a new authentication token was created with the injected NTLM hash, ready for network use.
2. **Network Connection to DC**:
   - **Event 4624**: A Logon Type 3 (Network) is logged on the Domain Controller. The Authentication Package is recorded as NTLM (a strong indicator since Kerberos should be the default). The source IP points back to Workstation A.
3. **Service Creation on DC**:
   - **Event 7045**: The Service Control Manager logs the creation of a new service, typically named `PSEXESVC` (though attackers can rename this).
4. **Execution on DC**:
   - **Event 4688**: `services.exe` (Parent) spawns `PSEXESVC.exe` (Child).
   - **Event 4624**: Logon Type 5 (Service) is logged for the PsExec service starting up.
   - **Event 4688**: `PSEXESVC.exe` spawns `cmd.exe` running as `NT AUTHORITY\SYSTEM`, which then executes the attacker's final payload to dump AD credentials.

## 6. Deep Technical Concepts and Evidentiary Nuances

### Process Command Line Logging Limitations
While enabling Process Command Line logging is crucial, it has inherent limitations. Command lines are logged precisely as they are passed to the Operating System. If an attacker uses obfuscation techniques (e.g., environment variable substitution like `cmd.exe /c set a=pow && set b=ershell && call %a%%b%`), the 4688 event will only capture the obfuscated string. Defenders must rely on supplemental telemetry like script block logging (Event 4104) to see the deobfuscated payload in memory.

Furthermore, attackers can employ advanced evasion techniques like **Command Line Spoofing**. Using Windows APIs like `CreateProcess` in a suspended state, an attacker can supply a benign command line (e.g., `ping 8.8.8.8`), which is dutifully logged by Event 4688. Before resuming the process thread, the attacker modifies the Process Environment Block (PEB) memory to point to the real, malicious payload. The 4688 event will only show the benign command, blinding defenders relying solely on this event.

### Kerberos vs NTLM in 4624 Type 3
When hunting for anomalous lateral movement, inspecting the Authentication Package in a 4624 Type 3 logon is essential. In modern Active Directory environments, Kerberos is the default for network authentication. The presence of NTLM authentication should be treated as highly suspicious, especially between two servers, or from a workstation to a server using privileged accounts. Attackers frequently default to NTLM for Pass-the-Hash, or because exploitation frameworks like Impacket often utilize NTLM authentication natively unless explicitly configured for Kerberos.

## 7. Advanced Detection and Hunting Queries

### KQL Query: Suspicious Parent-Child Relationships
This query looks for common system binaries acting as parents to shells, which often indicates malicious injection or service abuse.
```kusto
SecurityEvent
| where EventID == 4688
| where ParentProcessName endswith "spoolsv.exe" 
     or ParentProcessName endswith "svchost.exe"
     or ParentProcessName endswith "services.exe"
| where ProcessName in~ ("cmd.exe", "powershell.exe", "certutil.exe", "wmic.exe", "rundll32.exe")
| project TimeGenerated, Computer, Account, ParentProcessName, ProcessName, CommandLine
| sort by TimeGenerated desc
```

### KQL Query: Anomalous Type 9 Logons (Pass-the-Hash Indicator)
```kusto
SecurityEvent
| where EventID == 4624
| where LogonType == 9
| where AuthenticationPackageName =~ "Negotiate"
| where LogonProcessName =~ "seclogo"
| project TimeGenerated, Computer, Account, TargetUserName, LogonType, IpAddress
```

### KQL Query: RDP Logons Outside Business Hours
```kusto
SecurityEvent
| where EventID == 4624
| where LogonType == 10
| extend HourOfDay = hourofday(TimeGenerated)
| where HourOfDay < 6 or HourOfDay > 20
| project TimeGenerated, Computer, TargetUserName, IpAddress
```

## 8. Mitigation and Hardening
To fully leverage these event IDs, environments must ensure proper audit policies are applied via Group Policy Objects (GPO):
1. **Enable Audit Process Creation**: Ensure success logging is enabled.
2. **Include Command Line in Process Creation Events**: This requires enabling the specific administrative template in GPO (`Administrative Templates > System > Audit Process Creation > Include command line in process creation events`).
3. **Audit Logon Events**: Ensure both Success and Failure are logged for Logon/Logoff categories.
