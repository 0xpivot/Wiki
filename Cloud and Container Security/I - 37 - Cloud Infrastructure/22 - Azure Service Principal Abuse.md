# 22 - Azure Service Principal Abuse

## Introduction to Azure Service Principals

Azure Service Principals (SPs) are the core non-human identities used in Azure Active Directory (Azure AD). When an application needs to access Azure resources, it doesn't use a standard user account. Instead, a Service Principal is created in the specific Azure AD tenant where the application will operate. Think of a Service Principal as a user account for a machine, service, or application. It is the local representation, or application instance, of a global application object in a single tenant or directory.

Understanding how Service Principals are created, managed, and authenticated is a foundational requirement for any cloud penetration tester. Compromising a Service Principal often leads directly to privilege escalation, lateral movement, or data exfiltration.

In Azure, there are three types of Service Principals:
1. **Application**: The local representation of a global application object.
2. **Managed Identity**: A special type of Service Principal used to provide an identity for Azure resources. These are managed by Azure, eliminating the need for developers to manage credentials.
3. **Legacy**: Service principals that represent legacy applications.

## Architecture and Core Concepts

Before diving into the abuse mechanics, we must distinguish between the **Application Object** and the **Service Principal Object**.

- **Application Object**: The global representation of your application. It defines the application's configuration globally. It lives in the tenant where the application was registered (the home tenant).
- **Service Principal Object**: The local representation of the application in a specific tenant. It dictates what the application can actually do in that tenant (its permissions, roles, etc.).

When an application is granted access to a resource, the role assignment is made to the Service Principal, not the Application Object.

### Authentication Mechanisms

Service Principals typically authenticate using one of two methods:
1. **Client Secrets (Passwords)**: A string generated by Azure AD.
2. **Certificates (Asymmetric Keys)**: A more secure method using X.509 certificates.

## Enumeration of Service Principals

Once initial access to an Azure environment is achieved (even with a low-privileged user), the next step is enumeration. By default, any user in Azure AD can read the details of all other users, groups, and applications (unless restricted by the administrator).

### Using Azure CLI
```bash
# List all Service Principals
az ad sp list --all --output table

# Get details of a specific Service Principal
az ad sp show --id <app-id>
```

### Using AzureAD PowerShell Module
```powershell
# Get all service principals
Get-AzureADServicePrincipal -All $true

# Find SPs with password credentials (secrets)
Get-AzureADServicePrincipal | Where-Object { $_.PasswordCredentials.Count -gt 0 }
```

### Using ROADtools
ROADtools is a framework to interact with Azure AD. It is highly effective for enumerating and analyzing SPs.
```bash
roadrecon auth --username <user> --password <pass>
roadrecon gather
roadrecon plugin gui
```

## Attack Vectors and Exploitation

### 1. Hardcoded Credentials

The most common vulnerability associated with Service Principals is hardcoded credentials. Developers often embed SP client IDs and secrets directly into source code, ARM templates, configuration files, or CI/CD pipelines (like GitHub Actions or Azure DevOps).

**Discovery**:
- Scanning public or private GitHub repositories for secrets (e.g., using TruffleHog or Gitleaks).
- Analyzing extracted application binaries.
- Checking environment variables on compromised internal machines or containers.

**Exploitation**:
Once the Tenant ID, Application ID (Client ID), and Client Secret are obtained, an attacker can authenticate:
```bash
az login --service-principal -u <App-ID> -p <Client-Secret> --tenant <Tenant-ID>
```

### 2. Over-Privileged Service Principals

Often, SPs are granted the `Contributor` or even `Owner` role at the subscription level, rather than being scoped to the specific resource group or resource they actually need to access.

If an attacker compromises an SP with the `Contributor` role on a subscription, they can:
- Run commands on any VM in the subscription using `RunCommand`.
- Extract secrets from Azure Key Vaults.
- Create new resources for cryptomining.
- Modify network security groups (NSGs) to allow lateral movement.

### 3. Service Principal Owner Abuse (Credential Addition)

If a compromised user account is listed as an "Owner" of an Application Object, they have the ability to add new credentials (secrets or certificates) to that application. This is a critical privilege escalation path.

Even if the user doesn't have rights over the resources the SP can access, adding a secret to the SP allows the attacker to log in as the SP and inherit its permissions.

**Steps to Exploit**:
1. Identify applications owned by the compromised user.
2. Add a new client secret to the application.
3. Authenticate as the Service Principal.

```powershell
# Using Az PowerShell module
$App = Get-AzADApplication -ObjectId <App-ObjectId>
$Secret = New-AzADAppCredential -ObjectId $App.Id -EndDate (Get-Date).AddYears(1)
```
*Note: This creates a new password for the app. The attacker logs in, while legitimate operations using the original secret continue uninterrupted, maintaining stealth.*

### 4. Illicit Consent Grant Attacks

An attacker can create a malicious multi-tenant application and trick a victim (often an administrator) into granting it consent. Once consent is granted, a Service Principal for the malicious app is created in the victim's tenant, granting the attacker's app access to the victim's data (like reading emails, accessing SharePoint, etc.).

## Privilege Escalation Scenarios

### Key Vault Extraction
If the SP has access to an Azure Key Vault, attackers can extract other credentials, certificates, or keys to move laterally.
```bash
az keyvault secret list --vault-name <VaultName>
az keyvault secret show --name <SecretName> --vault-name <VaultName>
```

### Automation Accounts and Runbooks
SPs often have permissions to execute Azure Automation Runbooks. An attacker can modify a runbook to execute malicious code under the context of the Automation Account's identity.

## Persistence Mechanisms

Attackers frequently use Service Principals to establish persistence in an Azure environment.
1. **Creating a rogue SP**: If an attacker has sufficient privileges (e.g., Global Admin or Application Admin), they can create a new SP and assign it the `Owner` role on the subscription.
2. **Adding backup credentials**: Adding an extra secret or certificate to a legitimate, highly privileged SP with a long expiration date (e.g., 10 years).
3. **Federated Identity Credentials**: Configuring federated credentials allows an attacker to access the SP without needing to store a secret in Azure AD, instead using OIDC tokens from external providers (like a GitHub repository they control).

## Mitigations and Defenses

1. **Least Privilege**: Strictly scope Role-Based Access Control (RBAC) assignments. Never grant an SP `Owner` or `Contributor` on a whole subscription if it only needs access to a single storage account.
2. **Use Managed Identities**: Whenever possible, replace traditional Service Principals with Managed Identities. Managed Identities do not have credentials that can be leaked; Azure manages them automatically.
3. **Restrict App Registration**: Disable the ability for standard users to register applications in Azure AD. This prevents regular users from creating unmonitored SPs.
4. **Certificate over Secret**: Use certificates for SP authentication instead of client secrets. They are harder to accidentally leak in source code.
5. **Credential Rotation**: Implement aggressive rotation policies for SP secrets.

## Detection and Logging

Monitoring for SP abuse involves analyzing Azure AD Sign-in logs and Audit logs.

**High-Fidelity Alerts to configure in Sentinel/Defender:**
- **Multiple failed logins**: Followed by a successful login for an SP.
- **Unusual IP addresses**: An SP authenticating from an IP address not associated with your corporate network or cloud environment.
- **New credential added**: Event ID `Update application – Certificates and secrets management` in Azure AD Audit logs. This is highly suspicious if done on a highly privileged application outside of a standard deployment window.
- **First-time access**: An SP accessing a resource (like a Key Vault) it has never accessed before.

## Chaining Opportunities

- **Initial Access**: Can be combined with credential stuffing or repository leaks to gain initial access via SP secrets.
- **Privilege Escalation**: Adding a secret to an app object allows lateral movement to the permissions held by the SP, often chaining into `[[05 - Privilege Escalation in Azure]]`.
- **SSRF**: Using `[[27 - Cloud SSRF to Credential Theft — Full Chain]]` to steal tokens from an environment, effectively acting as the SP.

## Deep Dive: Advanced Real-World Scenarios and Case Studies

In advanced penetration testing engagements, simplistic vulnerabilities are rarely found in isolation. Instead, attackers must chain multiple low-severity issues to achieve critical impact. The complexity of modern cloud architectures often obscures these attack paths from defenders, while providing numerous opportunities for patient adversaries.

Consider a scenario where an organization implements strict IAM policies but neglects network-level egress controls. An attacker might exploit a minor Server-Side Request Forgery (SSRF) vulnerability that, due to strict IAM, yields a token with seemingly useless permissions. However, by thoroughly enumerating the environment using tools discussed previously, the attacker discovers an obscure, legacy API endpoint internal to the VPC. This endpoint, trusting any request originating from within the network, allows the attacker to manipulate database records.

This illustrates a fundamental principle in cloud security: identity is the new perimeter, but network controls still provide critical defense-in-depth. A failure in either domain can lead to a complete compromise.

Furthermore, the operational tempo of cloud deployments—where Infrastructure as Code (IaC) pipelines deploy changes multiple times a day—frequently introduces transient vulnerabilities. A permission granted temporarily for debugging might be accidentally committed to the main branch, exposing a highly privileged role for just a few hours. Advanced adversaries automate their reconnaissance to detect and exploit these fleeting windows of opportunity.

To combat this, defensive teams must adopt an "assume breach" mentality. This means implementing continuous monitoring of control plane logs (like AWS CloudTrail or Azure Activity Logs), utilizing anomaly detection to spot unusual API call patterns, and conducting regular red team exercises to validate the effectiveness of security controls. The notes in this module provide the offensive perspective necessary to design these robust, resilient cloud architectures.
