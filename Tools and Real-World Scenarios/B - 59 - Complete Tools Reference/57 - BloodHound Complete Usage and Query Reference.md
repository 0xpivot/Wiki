# BloodHound: Complete Usage and Custom Query Reference

BloodHound uses graph theory to reveal the hidden and often unintended relationships within an Active Directory or Azure environment. Attackers use BloodHound to easily identify complex attack paths that would otherwise be impossible to quickly identify. Defenders use BloodHound to identify and eliminate those same attack paths.

BloodHound relies on a Neo4j database to store the graph data and an ingestor (like SharpHound or AzureHound) to collect the data.

## Architecture and Components

BloodHound consists of three primary components:
1.  **Data Collector (Ingestor):** Tools like SharpHound (C#), BloodHound.py (Python), or AzureHound used to poll AD via LDAP, RPC, and SMB to gather users, groups, computers, ACLs, and sessions.
2.  **Database (Neo4j):** A graph database optimized for traversing highly interconnected data.
3.  **GUI Application:** An Electron-based interface that allows users to query Neo4j using Cypher queries and visualizes the results as nodes and edges.

## Neo4j Setup and Management

BloodHound requires a running instance of Neo4j.

**Installation (Kali/Debian):**
```bash
sudo apt update
sudo apt install bloodhound neo4j
```

**Starting Neo4j:**
```bash
sudo systemctl start neo4j
sudo neo4j console
```

**Accessing Neo4j Console (Web Interface):**
Navigate to `http://localhost:7474`.
Default credentials are `neo4j` / `neo4j`. You will be forced to change the password upon first login.

**Clearing the Database:**
Before importing a new dataset or when switching between engagements, it is crucial to wipe the database to prevent data contamination.
- Via BloodHound GUI: Database Info -> Clear Database.
- Via Cypher Query: `MATCH (n) DETACH DELETE n`

## Ingesting Data

Data ingestion is primarily performed using SharpHound or BloodHound-Python.

- **Importing via GUI:** Drag and drop the `.zip` file generated by SharpHound directly into the BloodHound GUI.
- **Importing via API (BloodHound CE):** Enterprise and CE versions allow direct API ingestion.

For full details on data collection, refer to [[58 - SharpHound Data Collection for BloodHound]].

## Node and Edge Types

Understanding the graph structure is essential for writing custom queries.

### Nodes (Entities)
- `User`: AD User accounts.
- `Computer`: Joined machines.
- `Group`: Security and Distribution groups.
- `Domain`: The AD domain itself.
- `GPO`: Group Policy Objects.
- `OU`: Organizational Units.
- `Container`: Active Directory containers.

### Edges (Relationships/Permissions)
Edges define how nodes interact. Common edges include:
- `MemberOf`: Node is a member of a Group.
- `HasSession`: A user has an active session on a computer.
- `AdminTo`: Node has local admin rights on a computer.
- `GenericAll`: Full control over the target object.
- `GenericWrite`: Ability to write properties to the target object.
- `WriteOwner`: Ability to take ownership of the target object.
- `ForceChangePassword`: Ability to reset a user's password without knowing the current one.
- `AddMember`: Ability to add users to a group.
- `CanRDP`: Allowed to connect via RDP.
- `ExecuteDCOM`: Can execute COM objects remotely.
- `AllowedToDelegate`: Constrained delegation allowed.
- `SyncLAPSPassword`: Can read LAPS passwords.

## Cypher Query Language Reference

Cypher is the query language used by Neo4j. It uses ASCII-art style syntax to represent graph patterns.

Syntax Basics:
- `(n:User)` represents a Node `n` with the label `User`.
- `[r:MemberOf]` represents a Relationship (edge) `r` with the type `MemberOf`.
- `->` indicates directionality.

Example structure: `MATCH (SourceNode)-[Relationship]->(TargetNode) RETURN Node`

### Pre-Built Queries

BloodHound includes excellent pre-built queries in the "Analysis" tab:
- Find all Domain Admins
- Find shortest paths to Domain Admins
- Find principals with DCSync rights
- Find computers with unsupported operating systems

### Advanced Custom Cypher Queries

When standard queries aren't enough, custom Cypher queries unlock BloodHound's true power.

#### 1. Enumerating High-Value Targets
**Find all users with an SPN (Kerberoastable users):**
```cypher
MATCH (u:User {hasspn:true}) RETURN u
```

**Find all users that do not require Kerberos Pre-Authentication (AS-REP Roastable):**
```cypher
MATCH (u:User {dontreqpreauth:true}) RETURN u
```

**Find all Kerberoastable users with paths to High Value Targets (Domain Admins):**
```cypher
MATCH (u:User {hasspn:true})
MATCH (g:Group) WHERE g.name CONTAINS 'DOMAIN ADMINS'
MATCH p = shortestPath((u)-[*1..]->(g))
RETURN p
```

#### 2. Analyzing Lateral Movement Paths
**Find all computers where a specific user is a local admin:**
```cypher
MATCH (u:User {name:"ATTACKER@DOMAIN.LOCAL"})-[r:AdminTo]->(c:Computer)
RETURN u,r,c
```

**Find all active sessions for members of a specific group (e.g., Helpdesk):**
```cypher
MATCH (g:Group {name:"HELPDESK@DOMAIN.LOCAL"})<-[:MemberOf*1..]-(u:User)-[:HasSession]->(c:Computer)
RETURN u,c
```

**Identify computers where multiple Domain Admins have active sessions:**
```cypher
MATCH (c:Computer)<-[:HasSession]-(u:User)-[:MemberOf*1..]->(g:Group {name:'DOMAIN ADMINS@DOMAIN.LOCAL'})
WITH c, collect(u) as sessions, count(u) as count
WHERE count > 1
RETURN c, sessions, count
```

#### 3. Analyzing Abuse of Active Directory ACLs
**Find all users that can force reset the password of another user:**
```cypher
MATCH (u1:User)-[r:ForceChangePassword]->(u2:User) RETURN u1,r,u2
```

**Find objects that have GenericAll (Full Control) over a target GPO:**
```cypher
MATCH (n)-[r:GenericAll]->(g:GPO) RETURN n,r,g
```

**Find all principals with DCSync rights (GetChangesAll):**
```cypher
MATCH (n)-[r:GetChangesAll]->(d:Domain) RETURN n,r,d
```

#### 4. Delegation and Trust Mapping
**Find users trusted for Unconstrained Delegation:**
```cypher
MATCH (u:User {unconstraineddelegation:true}) RETURN u
```

**Find computers trusted for Unconstrained Delegation (excluding Domain Controllers):**
```cypher
MATCH (c:Computer {unconstraineddelegation:true})
MATCH (dc:Group) WHERE dc.name CONTAINS 'DOMAIN CONTROLLERS'
WHERE NOT (c)-[:MemberOf*1..]->(dc)
RETURN c
```

**Map Constrained Delegation paths:**
```cypher
MATCH (u:User)-[r:AllowedToDelegate]->(c:Computer) RETURN u,r,c
```

#### 5. Defense and auditing
**Identify stale user accounts (Password last set > 90 days ago):**
```cypher
MATCH (u:User)
WHERE u.pwdlastset < (timestamp() - 7776000) AND u.enabled = true
RETURN u.name, u.pwdlastset
```

**Find Local Administrator accounts that share the same password across multiple machines (LAPS absent):**
```cypher
MATCH (c1:Computer)-[:AdminTo]->(u:User)<-[:AdminTo]-(c2:Computer)
WHERE c1 <> c2
RETURN u, c1, c2
```

## Practical Analysis Methodology

1.  **Mark the target:** Right-click the Domain Admin group (or an equivalent high-value group) and set it as the target.
2.  **Mark the owned nodes:** As you compromise users and computers, mark them as "Owned" (Skull icon).
3.  **Calculate paths:** Use the query `Shortest Paths from Owned Principals` to dynamically calculate the shortest attack path from your currently compromised assets to Domain Admin.
4.  **Evaluate edge abuse:** Analyze the specific edges connecting your owned nodes to the target. For instance, if the path relies on an `AdminTo` edge, verify you can access SMB/RPC on the target machine. If it relies on `GenericWrite`, prepare tools like PowerView or Impacket to abuse the ACL.
5.  **Iterate:** As new systems are compromised or new data is ingested, repeat the process.
