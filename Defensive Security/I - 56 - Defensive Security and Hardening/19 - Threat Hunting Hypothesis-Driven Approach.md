# 19 - Threat Hunting: Hypothesis-Driven Approach

Traditional Security Operations Center (SOC) operations are predominantly reactive. They rely on established signatures, rules, and alerts generated by SIEMs, EDRs, or Firewalls. If a rule triggers, an analyst investigates.

**Threat Hunting** is the proactive, iterative, and human-centric process of actively searching through networks and datasets to detect malicious activity that has evaded existing security defenses. It assumes the environment is already compromised (Assume Breach mindset) and seeks to find the hidden adversary.

The most effective and mature form of Threat Hunting is the **Hypothesis-Driven Approach**.

## The Philosophy of Threat Hunting

Threat hunting is not randomly digging through logs looking for "bad stuff." It requires structure, methodology, and a deep understanding of both the organization's architecture and the adversary's tradecraft.

*   **Proactive, not Reactive:** Hunters don't wait for alerts.
*   **Human-Driven:** Relies heavily on the analyst's intuition, experience, and knowledge of attacker behavior, augmented by automation.
*   **Iterative:** The hunt process is continuous. Findings from one hunt feed into the next and are used to improve automated defenses.

## The Threat Hunting Lifecycle (The Loop)

A successful threat hunt follows a structured lifecycle to ensure hunts are focused, measurable, and produce actionable results.

### 1. Hypothesis Generation

This is the most critical phase. A hypothesis is a structured question or a testable statement about a potential threat in the environment.

*   **Bad Hypothesis:** "Are there any hackers in the network?" (Un-testable, too broad).
*   **Good Hypothesis:** "Due to recent reporting by CISA regarding APT29 utilizing compromised Service Principals, I hypothesize that an adversary might be utilizing dormant Azure AD Service Principal accounts to issue anomalous Graph API calls for email exfiltration."

**Sources for Hypotheses:**
*   **Intelligence-Driven:** Driven by Cyber Threat Intelligence (CTI) reports, OSINT, or industry trends. (e.g., "A new zero-day in MS Exchange is actively exploited; let's hunt for post-exploitation web shells").
*   **Situational Awareness / Domain-Driven:** Based on internal knowledge of the environment. (e.g., "We recently acquired a new company; their legacy domain might be used as a pivot point").
*   **Analytics-Driven (Data-Driven):** Based on deviations from established baselines identified through machine learning or statistical analysis.

### 2. Hunt Execution

Once the hypothesis is formed, the hunter determines what data sources are required to test it and executes the technical aspect of the hunt.

*   **Data Sources:** Firewall logs, Proxy logs, Windows Event Logs (specifically 4688, 4624), EDR telemetry, DNS queries, PCAPs.
*   **Techniques:**
    *   **Searching:** Querying for specific IOCs (Hashes, IPs).
    *   **Stacking (Frequency Analysis):** Taking a large dataset and counting the occurrences of specific values. Anomalies often hide in the "long tail" (items that appear very infrequently). *Example: Stacking User-Agents in proxy logs to find a unique agent string used by a single host.*
    *   **Clustering:** Grouping similar data points together based on specific characteristics to identify patterns.
    *   **Grouping:** Analyzing events within a specific time window.

### 3. Discovery & Triage

The hunt execution will return data. The hunter must now analyze this data to confirm or refute the hypothesis.

*   **Refutation:** The data shows no evidence of the hypothesized activity. This is still a success! It confirms a specific defense is working or a specific technique is not currently being used.
*   **Confirmation:** The hunter identifies anomalous or malicious activity.
*   **Triage:** The malicious activity is immediately handed off to the Incident Response (IR) team for containment, eradication, and recovery. The hunt officially pauses here.

### 4. Enrichment & Automation

A hunt is only truly valuable if its findings are used to improve the organization's overall security posture.

*   **Rule Creation:** If a hunt successfully identified a new attacker technique, the queries used to find it should be codified into a new automated SIEM rule or EDR alert. The goal is to turn today's successful proactive hunt into tomorrow's reactive automated alert.
*   **Visibility Gaps:** If the hunt failed because the necessary logs were missing (e.g., "We couldn't test the hypothesis because command-line logging isn't enabled on servers"), this gap must be documented and addressed by IT engineering.

## Practical Hypothesis-Driven Scenarios

### Scenario 1: Hunting for Lateral Movement via WMI

*   **Hypothesis:** An attacker who has gained initial access is using Windows Management Instrumentation (WMI) to move laterally and execute commands on remote endpoints to evade detection by standard antivirus.
*   **Data Sources Required:** Windows Security Event Logs, Sysmon (specifically Network connections and Process creation).
*   **Execution (Stacking):** Query the SIEM for all network connections over port 135 (RPC/WMI) originating from non-administrative subnets. Correlate this with process execution logs (Event ID 4688) on the destination hosts looking for `WmiPrvSE.exe` spawning unusual child processes (like `cmd.exe` or `powershell.exe`).
*   **Discovery:** A developer workstation is observed making WMI connections to 50 different servers within 5 minutes, followed immediately by `WmiPrvSE.exe` executing an encoded PowerShell command on those servers.
*   **Action:** Initiate IR.

### Scenario 2: Hunting for C2 Beaconing

*   **Hypothesis:** Malware on a compromised internal host is communicating with an external Command and Control (C2) server using HTTPS, hiding its traffic within legitimate web browsing.
*   **Data Sources Required:** Proxy logs, Firewall logs, DNS logs.
*   **Execution (Statistical Analysis):** Attackers often program their malware to "beacon" (check in) at regular intervals (e.g., every 60 seconds) with slight randomization (jitter) to avoid detection. Query proxy logs for outbound connections from internal hosts to external IPs. Use mathematical functions within the SIEM to calculate the variance or standard deviation of the time delta between connections to the same IP.
*   **Discovery:** Find a host connecting to an unknown, un-categorized domain exactly every 302 to 308 seconds, 24 hours a day, regardless of user activity.
*   **Action:** Block the domain, isolate the host, and investigate the process responsible for the connections.

## Framework Integration: MITRE ATT&CK

The MITRE ATT&CK framework is indispensable for Threat Hunting. It provides a common taxonomy for describing adversary tactics and techniques. Hunters use ATT&CK to:
1.  Map CTI reports to specific techniques to generate hypotheses.
2.  Track which techniques they have hunted for, identifying gaps in their coverage.
3.  Communicate findings clearly to other teams (e.g., "We found evidence of T1055: Process Injection").
