# 24.04 — Open Redirect via Referer Header

## What is it?
While most Open Redirects happen via URL parameters, they can also occur implicitly via HTTP headers—most notably the `Referer` (or `Referrer`) header.

When a user browses a web application, their browser automatically sends the `Referer` header indicating the page they just came from. Developers often use this header to create a "Go Back" button or to automatically return the user to their previous page after completing an action (like logging in, switching a language, or changing a theme).

If the application reads the `Referer` header and places it directly into a `Location:` header without validation, an attacker can exploit this. While an attacker cannot easily force a victim's browser to send a fake `Referer` header across domains, they *can* host a malicious page that links to the vulnerable site. When the victim clicks the link on the attacker's site, the browser sets the `Referer` to the attacker's site. The vulnerable application then bounces the user right back (or somewhere else malicious).

Think of it like a package return policy. You buy a TV and the store policy is "If it's broken, return to sender." If you put a fake return address on the box, the store blindly ships the TV to the fake address without checking if it makes sense.

## How to Find It
- **Manual steps:**
  1. Identify actions in the web application that return you to your previous page (e.g., logging in, clicking "Cancel" on a form, changing the site's language).
  2. Intercept the request for that action in Burp Suite.
  3. Locate the `Referer` header. Change its value to `http://evil.com`.
  4. Submit the request.
  5. Check the HTTP response. If it contains `Location: http://evil.com`, the application is vulnerable.

- **Tool commands with flags explained:**
  Using `curl` to manually inject the Referer header:
  ```bash
  # Send a POST request to a language toggle endpoint, injecting the Referer
  curl -i -X POST "https://target.com/setLanguage?lang=fr" \
       -H "Referer: https://evil.com"
  # Look for 'Location: https://evil.com' in the output
  ```

## How to Exploit It
- **Step-by-step walkthrough:**
  1. Confirm the vulnerability using Burp Suite or `curl`.
  2. To create a weaponized exploit, you must host a web page.
  3. Build an HTML page on your server (`evil.com`) that contains a link to the vulnerable endpoint (`https://target.com/setLanguage?lang=fr`).
  4. Ensure your server does *not* have a stringent Referrer-Policy that drops the header. (Usually, navigating from HTTPS to HTTPS will send the full Referer).
  5. Trick the victim into visiting your page and clicking the link. The browser automatically sends your domain in the `Referer` header to the target, which executes the action and redirects the victim back into your trap.

- **Actual payloads:**
  **Attacker HTML Page (`index.html` on `evil.com`):**
  ```html
  <!DOCTYPE html>
  <html>
  <head>
      <!-- Ensure the browser sends the full URL as the Referer -->
      <meta name="referrer" content="unsafe-url">
  </head>
  <body>
      <h1>You have won a prize!</h1>
      <p>Click here to claim: <a href="https://target.com/login">Claim Prize</a></p>
  </body>
  </html>
  ```

- **Real HTTP request/response examples:**
  **Victim Request (Auto-generated by browser):**
  ```http
  POST /preferences/theme?set=dark HTTP/1.1
  Host: target.com
  Referer: https://attacker.com/trap
  ```
  **Vulnerable Response:**
  ```http
  HTTP/1.1 302 Found
  Location: https://attacker.com/trap
  ```

## Real-World Example
A security researcher was testing a forum application. When a user clicked "Logout", the forum used the `Referer` header to redirect the user back to the thread they were just reading. The researcher set up a malicious domain (`hacked-forum.com`) that loaded the real forum's `/logout` endpoint in a hidden `<iframe>`. When victims visited `hacked-forum.com`, their browsers fetched the iframe, sending `Referer: https://hacked-forum.com`. The forum logged them out and redirected the iframe back to `hacked-forum.com`. The researcher combined this with a fake login page, successfully phishing credentials from logged-out users.

## How to Fix It
- **Developer remediation:**
  The `Referer` header is entirely untrusted client input and must be treated as such. Never use it directly for routing or redirection without strict validation. If you must use it to return a user to a previous page, you must validate that the domain in the `Referer` header exactly matches your own application's domain.
  
  Alternatively, avoid using the `Referer` header for navigation. Instead, rely on server-side session state to track the user's journey.

- **Code snippet:**
  **Python (Django - Validating Referer before Redirecting):**
  ```python
  from urllib.parse import urlparse
  from django.shortcuts import redirect
  from django.conf import settings

  def safe_referer_redirect(request):
      referer = request.META.get('HTTP_REFERER')
      
      if referer:
          parsed_referer = urlparse(referer)
          # Only redirect if the referer exactly matches our domain
          if parsed_referer.netloc == settings.ALLOWED_HOST:
              return redirect(referer)
              
      # Fallback to a safe, known path
      return redirect('/home')
  ```
