# 14 - Parsing NTDS.dit Secretsdump

## 1. Introduction and Context

Once an attacker has successfully extracted the `NTDS.dit` file and the `SYSTEM` registry hive from a compromised Domain Controller (see [[13 - NTDS.dit Extraction via VSSAdmin and NTDSUtil]]), the next critical phase in the attack lifecycle is offline extraction and parsing. 

The `NTDS.dit` file is an Extensible Storage Engine (ESE) database. It is not plain text and cannot be read with standard text editors. Furthermore, the sensitive data within it—such as NTLM hashes, LM hashes, Kerberos keys, cleartext passwords (if reversible encryption is enabled), and password history—is heavily encrypted. 

Parsing `NTDS.dit` offline is highly advantageous for an attacker because it generates zero network traffic and zero event logs on the target infrastructure once the files have been exfiltrated.

### The Active Directory Encryption Architecture

To understand how parsing tools work, one must understand how Active Directory protects these secrets cryptographically:
1. **Password Hash (NTLM):** The user's plaintext password is mathematically hashed using the MD4 algorithm (NTLM).
2. **PEK (Password Encryption Key):** The NTLM hash is not stored plainly in the database. It is encrypted using the PEK. The PEK is a global, 16-byte RC4 key generated by Active Directory to protect all secrets.
3. **BootKey (SYSKEY):** The PEK itself is encrypted using the BootKey (historically known as SYSKEY).
4. **SYSTEM Hive:** The BootKey is stored within the `SYSTEM` registry hive of the Domain Controller.

Therefore, to parse `NTDS.dit` successfully, a tool must sequentially:
1. Extract the BootKey from the `SYSTEM` hive.
2. Use the BootKey to decrypt the PEK stored inside the `NTDS.dit` database metadata.
3. Traverse the ESE database tables, extracting the encrypted user objects.
4. Use the decrypted PEK to RC4-decrypt the NTLM hashes and Kerberos keys of the users.

## 2. Parsing Tooling: Impacket's SecretsDump

The industry standard for parsing `NTDS.dit` offline is `secretsdump.py` from the Impacket suite, developed by Fortra (formerly Core Security). It handles the complex ESE database interactions and cryptographic derivations automatically.

### 2.1 Execution Syntax

Assuming the attacker has exfiltrated both files (`ntds.dit` and `SYSTEM`) to their local Kali Linux machine or attack rig, the command structure is straightforward:

```bash
impacket-secretsdump -system SYSTEM -ntds ntds.dit LOCAL
```

**Parameters Explained:**
- `-system SYSTEM`: Specifies the local path to the extracted SYSTEM registry hive.
- `-ntds ntds.dit`: Specifies the local path to the extracted NTDS database file.
- `LOCAL`: Tells secretsdump to parse local files rather than attempting a remote extraction via network protocols (like DCSync).

### 2.2 Advanced Options and Optimization

`secretsdump.py` provides several flags to optimize the output, especially for massive enterprise domains where the database might contain hundreds of thousands of objects, which could take hours to parse.

- `-outputfile <filename>`: Saves the parsed output directly to a file prefix, generating `<filename>.ntds`, `<filename>.sam`, etc.
- `-just-dc-user <username>`: Extracts data for a specific user only (e.g., `krbtgt` or `Administrator`). This significantly speeds up the parsing process if the attacker only needs a specific token.
- `-pwd-last-set`: Includes the timestamp of when the password was last changed, useful for determining if an account is stale.
- `-history`: Dumps password history hashes (useful for credential stuffing or understanding password patterns).
- `-just-dc-ntlm`: Only dumps NTLM hashes, skipping Kerberos keys and AES keys, which saves processing time and disk space.

## 3. Analyzing the Output

A successful parsing operation yields an output heavily based on the standard PWDump format:
`Username:RID:LMHash:NTLMHash:::`

**Example Output:**
```text
Administrator:500:aad3b435b51404eeaad3b435b51404ee:31d6cfe0d16ae931b73c59d7e0c089c0:::
Guest:501:aad3b435b51404eeaad3b435b51404ee:31d6cfe0d16ae931b73c59d7e0c089c0:::
krbtgt:502:aad3b435b51404eeaad3b435b51404ee:7a288bd1cb8eb85be16ba58742b0df2b:::
s.connor:1105:aad3b435b51404eeaad3b435b51404ee:8846f7eaee8fb117ad06bdd830b7586c:::
```

### 3.1 Understanding the Output Structure

1. **Username:** The sAMAccountName of the user.
2. **RID (Relative Identifier):** The unique identifier for the user within the domain. `500` is always the built-in Administrator, `502` is `krbtgt`.
3. **LMHash:** The LAN Manager hash. If it shows `aad3b435b51404eeaad3b435b51404ee`, this is the "empty" or "blank" LM hash, indicating LM hashing is disabled (which is standard and highly recommended on modern Windows).
4. **NTLMHash:** The actual NTLM hash of the user's password.

## 5. Alternative Parsing Tools

While Impacket is the most common, attackers and red teamers may use other tools, particularly in Windows-centric environments where Python is unavailable or when dealing with multi-gigabyte database files.

### 5.1 DSInternals (PowerShell)

The `DSInternals` PowerShell module is a powerful native Windows capability for parsing NTDS files. It is highly optimized and integrates seamlessly with other PowerShell automation.

```powershell
Import-Module DSInternals
# Extract BootKey first
$key = Get-BootKey -SystemHiveFilePath 'C:\Temp\SYSTEM'
# Parse database using the extracted key
Get-ADDBAccount -All -DBPath 'C:\Temp\ntds.dit' -BootKey $key | Format-Custom -View Hash
```

### 5.2 NTDSDumpEx

A standalone C++ executable that can rapidly parse the database. It is often significantly faster than Python-based equivalents for incredibly massive databases (e.g., databases over 10GB).

## 6. Post-Parsing Exploitation Actions

Once the hashes are extracted into a text file, the attacker's capabilities increase exponentially, achieving full domain dominance.

1. **Offline Password Cracking:**
   The hashes are fed into `Hashcat` to crack the plaintext passwords.
   ```bash
   hashcat -m 1000 ntds_hashes.txt rockyou.txt -r rules/dive.rule
   ```
   Cracking is critical for accessing external systems that do not accept Pass-the-Hash (e.g., VPN portals, Office 365, RDP without Restricted Admin mode).

2. **Pass-the-Hash (PtH):**
   Using the NTLM hashes directly to authenticate to other servers in the domain without needing the plaintext password, typically using tools like `wmiexec.py` or `CrackMapExec`.

3. **Golden Ticket Generation:**
   The NTLM hash of the `krbtgt` account is utilized to forge a Golden Ticket, providing catastrophic, persistent domain dominance even if the Domain Admin passwords are later changed.

## 7. Defensive Considerations and Detection

### Detection Strategy

Detecting the actual parsing of `NTDS.dit` is practically impossible from a network or endpoint telemetry perspective if the parsing is done offline on an attacker-controlled machine (which is almost always the case). 

The primary detection focus MUST be on the extraction phase (detecting `vssadmin`, `ntdsutil`, or DCSync network traffic) and the exfiltration phase (detecting large file transfers). Once the file leaves the network, the data must be considered fully compromised.

### Mitigation Strategy

1. **Password Complexity & Auditing:** Since the hashes will inevitably be subject to offline cracking if stolen, enforce strong, complex passwords (minimum 15 characters, utilizing passphrases) to make brute-forcing mathematically infeasible.
2. **Limit Data Value:** Rotate the `krbtgt` password periodically (every 180 days, done twice sequentially) to invalidate older Golden Tickets in case an old backup of `NTDS.dit` is stolen.
3. **Data Loss Prevention (DLP):** Implement egress filtering and DLP solutions that might detect massive database files or chunks of hex/Base64 data leaving the network perimeter.

## Real-World Attack Scenario

Following a successful extraction of `NTDS.dit` and the `SYSTEM` hive using `ntdsutil` during a ransomware readiness assessment, the red team exfiltrated the files to their external, GPU-optimized cracking rig. The target organization had over 15,000 active directory objects, meaning parsing this data live on a compromised host would be unacceptably slow and risky.

Once the files were locally accessible on the Kali Linux attack machine, the attacker utilized Impacket's `secretsdump.py` to parse the database offline. To maintain an organized workspace and avoid terminal clutter, the attacker directed the output to a specific prefix:
```bash
impacket-secretsdump -system SYSTEM_hive_dump -ntds ntds.dit LOCAL -outputfile corp_domain_hashes
```

The script successfully extracted the BootKey from the `SYSTEM` hive, decrypted the PEK within the `NTDS.dit` file, and proceeded to dump the NTLM hashes of all 15,000 users. Because the attacker specifically needed the Kerberos Ticket Granting Ticket (krbtgt) account to establish long-term persistence, they verified its presence in the output:
```bash
grep krbtgt corp_domain_hashes.ntds
```
The output displayed: `krbtgt:502:aad3b435b51404eeaad3b435b51404ee:f47a618e4726487e35ab196df576a0d2:::`. Armed with this NTLM hash, the attacker immediately forged a Golden Ticket to guarantee domain access even if the initial compromised accounts were discovered and reset.

Simultaneously, the attacker took the entire `corp_domain_hashes.ntds` file and fed it into Hashcat against the `rockyou.txt` dictionary combined with a custom rule set tailored to the company's name and local sports teams:
```bash
hashcat -m 1000 corp_domain_hashes.ntds /opt/wordlists/rockyou.txt -r /opt/hashcat/rules/OneRuleToRuleThemAll.rule
```
Within four hours, the cracking rig had successfully cracked the plaintext passwords for over 45% of the domain users, exposing widespread non-compliance with the organization's stated password policies and providing the red team with hundreds of credentials capable of accessing the external VPN without relying on Pass-the-Hash.
