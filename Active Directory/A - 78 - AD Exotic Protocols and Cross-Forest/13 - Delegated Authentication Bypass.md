# 78.13 Delegated Authentication Bypass

## Introduction to Kerberos Delegation
Kerberos Delegation is a feature in Active Directory that allows a service to impersonate a user to access other network resources on that user's behalf. For example, a frontend web server might need to access a backend database using the identity of the user browsing the website.
While conceptually useful, delegation is one of the most historically abused and misunderstood configurations in Active Directory. Misconfigured delegation paths allow attackers to forge service tickets, steal Ticket Granting Tickets (TGTs), and laterally move across administrative tiers, often resulting in complete domain or cross-forest compromise.

There are three primary types of Kerberos Delegation:
1.  **Unconstrained Delegation:** The most dangerous. The service extracts the user's actual TGT and uses it to request any service ticket.
2.  **Constrained Delegation (KCD):** The service requests a ticket on behalf of the user, but only for specific, predefined backend services.
3.  **Resource-Based Constrained Delegation (RBCD):** The target backend service determines who is allowed to delegate to it, shifting the access control model.

## Unconstrained Delegation Exploitation
When a server has Unconstrained Delegation enabled (the UserAccountControl flag `TRUSTED_FOR_DELEGATION`), any user authenticating to that server will have a copy of their TGT saved in the server's LSASS memory space.
An attacker who compromises a server with Unconstrained Delegation merely needs to wait for a highly privileged user (e.g., Domain Admin) to authenticate to the server. The attacker can then extract the DA's TGT and Pass-the-Ticket to compromise the domain.

### Coercion Techniques
Attackers do not have to wait passively. They can actively coerce high-privileged accounts (like Domain Controllers) to authenticate to the compromised Unconstrained Delegation server using RPC exploits like `PrintNightmare`, `PetitPotam`, or `SpoolSample`.
If a Domain Controller is coerced into authenticating, its Machine Account TGT is captured, immediately granting the attacker DCSync capabilities.

## Constrained Delegation (KCD) and Protocol Transition
Constrained Delegation restricts the impersonation to a list of allowed services, defined in the `msDS-AllowedToDelegateTo` attribute.
However, if a service account has Constrained Delegation configured with **Protocol Transition** (the `TRUSTED_TO_AUTHENTICATE_FOR_DELEGATION` UAC flag), it can leverage two Kerberos extensions:
-   **S4U2Self (Service for User to Self):** Allows the service to ask the KDC for a service ticket to *itself* on behalf of *any* arbitrary user.
-   **S4U2Proxy (Service for User to Proxy):** Allows the service to take the ticket generated by S4U2Self and exchange it for a ticket to the backend service specified in `msDS-AllowedToDelegateTo`.

### The KCD Attack Path
If an attacker compromises the hash or password of an account with KCD and Protocol Transition, they can use S4U2Self to generate a ticket for the `Administrator` account to themselves, and then use S4U2Proxy to exchange it for an `Administrator` ticket to the target backend service.
Crucially, the KDC does not care about the port/service class in the SPN. If the attacker has KCD to `cifs/target.corp.local`, they can use the resulting ticket for `ldap/target.corp.local` or `host/target.corp.local` (the Alternate Service Name or SPN translation attack), granting full control over the target.

## Resource-Based Constrained Delegation (RBCD)
RBCD modifies the delegation flow by placing the configuration on the *target* object rather than the *delegating* object. It uses the `msDS-AllowedToActOnBehalfOfOtherIdentity` attribute.
To exploit RBCD, an attacker only needs `GenericWrite` or `WriteDacl` permissions over the target computer object.

## Step-by-Step Execution Mechanics

### Phase 1: Exploiting RBCD using Impacket
Assuming the attacker has compromised an account with `GenericWrite` over `Victim-Server$`.
```bash
# 1. Create a fake machine account using MachineAccountQuota
impacket-addcomputer corp.local/attacker:Password123 -computer-name EvilPC$ -computer-pass EvilPass123

# 2. Modify the RBCD attribute on the target server
impacket-rbcd corp.local/attacker:Password123 -delegate-to Victim-Server$ -delegate-from EvilPC$ -action write

# 3. Request the service ticket using S4U2Self and S4U2Proxy
impacket-getST -spn cifs/Victim-Server.corp.local -impersonate Administrator -dc-ip 10.0.0.10 corp.local/EvilPC$:EvilPass123

# 4. Use the exported ticket to access the server
export KRB5CCNAME=Administrator.ccache
impacket-psexec -k -no-pass Victim-Server.corp.local
```

## Cross-Forest Delegation Exploitation
Delegation becomes exceptionally complex across forest trusts. By default, Microsoft introduced `EnableTGTDelegation = False` across forest trusts, meaning Unconstrained Delegation cannot extract TGTs from users crossing the trust boundary.
However, **Constrained Delegation and RBCD can still function across trusts** under specific conditions. If an attacker compromises a user in Domain A that has `GenericWrite` over a machine in Domain B (the trusting domain), they can configure RBCD on the Domain B machine and execute the S4U2 flow across the inter-realm trust, effectively bypassing the forest security boundary.

### Shadow Credentials (msDS-KeyCredentialLink)
A modern alternative to RBCD is the Shadow Credentials attack. Instead of modifying delegation attributes, the attacker modifies the `msDS-KeyCredentialLink` attribute of the target, injecting an attacker-controlled public key (Windows Hello for Business / Key Trust model). The attacker then uses the corresponding private key to request a PKINIT Kerberos TGT for the target machine or user, bypassing delegation entirely.

## Indicators of Compromise (IoCs) and Telemetry
1.  **Machine Account Creation:** Event ID 4741 (A computer account was created) by a non-administrative user.
2.  **Delegation Modifications:** Event ID 5136 (A directory service object was modified) looking specifically at changes to `msDS-AllowedToDelegateTo` or `msDS-AllowedToActOnBehalfOfOtherIdentity`.
3.  **S4U Anomalies:** High volumes of Event ID 4769 (TGS Request) where the requesting account and the target service account are the same (S4U2Self), immediately followed by an anomalous service request (S4U2Proxy).
4.  **KeyCredentialLink Edits:** Event ID 5136 for `msDS-KeyCredentialLink`.

## Defensive Mitigations and Engineering
1.  **Account Sensitive and Cannot Be Delegated:** Enable this Active Directory flag for all Tier 0 and administrative accounts. This prevents their credentials from being delegated via any method (Unconstrained, Constrained, or RBCD).
2.  **Protected Users Group:** Add highly privileged users to the Protected Users security group. This implicitly disables delegation for these accounts and enforces Kerberos-only authentication.
3.  **Disable Unconstrained Delegation:** Hunt for and eliminate all instances of Unconstrained Delegation, replacing them with strictly scoped KCD or RBCD.
4.  **Set MachineAccountQuota to 0:** Prevent standard users from creating up to 10 machine accounts, breaking the first step of standard RBCD and Shadow Credential attacks.
