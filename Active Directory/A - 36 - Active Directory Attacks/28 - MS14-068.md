# 36.28 MS14-068 (Forged PAC Vulnerability)

## 1. Introduction
**MS14-068 (CVE-2014-6324)** is a legendary privilege escalation vulnerability in Microsoft's Kerberos Key Distribution Center (KDC) implementation. Disclosed in late 2014, it allowed any standard authenticated domain user to forge a Kerberos Ticket Granting Ticket (TGT) that falsely claimed they were a member of highly privileged groups, such as `Domain Admins` or `Enterprise Admins`.

Because the Domain Controller accepted this forged ticket, the attacker could instantly elevate their privileges from a standard low-level user to total Domain Administrator in a single step, without needing any exploits against memory corruption or application logic—it was purely a cryptographic logic flaw.

## 2. Understanding the PAC (Privilege Attribute Certificate)
To understand MS14-068, one must understand how Microsoft adapted Kerberos.
Native Kerberos only handles authentication (who you are), not authorization (what you can access). To add authorization, Microsoft added a proprietary extension called the **Privilege Attribute Certificate (PAC)**.

The PAC is a data structure embedded inside Kerberos tickets (TGTs and TGSs). It contains the user's SIDs, their group memberships (e.g., Domain Users, Domain Admins), and user profile information. Without the PAC, Kerberos would not function properly within Windows environments.

### 2.1 PAC Signatures
Because the PAC contains highly sensitive authorization data, it must be protected against tampering. The KDC signs the PAC with two signatures:
1. **Server Signature**: Signed with the password hash of the target service. This verifies the integrity of the PAC for the target application.
2. **KDC Signature**: Signed with the `krbtgt` account hash. This verifies that the KDC actually issued the PAC and prevents attackers from forging it.

When a user requests a Service Ticket (TGS), they present their TGT (which contains their PAC). The KDC unpacks the TGT, reads the PAC, and uses it to build the subsequent service ticket.

## 3. The Vulnerability Mechanics

The MS14-068 vulnerability is a perfect storm of three separate logic flaws within how the Windows KDC handles Kerberos messages.

### Flaw 1: Disabling the PAC
The Kerberos specification allows a client to explicitly request that the KDC *not* include a PAC in the TGT. 
An attacker sends an `AS-REQ` (Authentication Service Request) setting the `include-pac` flag to `False`. The KDC complies and returns a TGT containing no PAC. This seems innocuous but sets the stage for the attack.

### Flaw 2: The "Include PAC" feature in TGS-REQ
When the client requests a Service Ticket (TGS-REQ), they must prove their authorization. If the TGT they present doesn't have a PAC, Microsoft allows the client to manually supply a PAC in the `authenticator` field of the request. 
The attacker crafts a malicious PAC locally. In this PAC, they put their own username, but they add the SIDs for `Domain Admins`, `Enterprise Admins`, and `Administrators`.

### Flaw 3: Cryptographic Downgrade (The Fatal Flaw)
The KDC must verify the signatures on this manually supplied PAC. Normally, PACs are signed using HMAC-MD5 or HMAC-SHA1 using the `krbtgt` key. The attacker does not possess the `krbtgt` key, so they cannot forge a valid KDC signature.

However, the KDC code had a catastrophic logic flaw: it allowed the PAC signature algorithm to be determined by the *client*, and it accepted non-keyed hash algorithms, specifically **MD5**.
The attacker hashes their forged PAC using standard MD5, completely bypassing the need for the `krbtgt` secret key. The KDC receives the PAC, sees the signature type is MD5, performs an MD5 hash of the data, and matches it with the signature. The KDC blindly trusts the maliciously crafted PAC!

### The Result
The KDC accepts the forged PAC, packages it into a brand new TGT, signs it properly with the actual `krbtgt` key, and sends it back to the attacker. The attacker now possesses a completely valid, cryptographically sound TGT that asserts they are a Domain Administrator.

## 4. Exploitation Step-by-Step

To exploit MS14-068, you need:
1. A valid domain username.
2. The user's cleartext password, NTLM hash, or Kerberos AES keys.
3. The user's exact SID.
4. The Domain Controller's IP and Domain Name.

### 4.1 Gathering Prerequisites
Using a tool like `rpcclient` or `Get-ADUser`, obtain the user's SID:
```bash
rpcclient -U "john" 192.168.1.10 -c "lookupnames john"
# Output: john S-1-5-21-123456-789012-345678-1001
```

### 4.2 Running the Exploit (Impacket / GoldenPac)
Impacket contains a module specifically designed to automate MS14-068 called `goldenPac.py`.

```bash
# syntax: goldenPac.py domain.local/user:password@targetDC
python3 goldenPac.py corp.local/john:'Password123'@dc01.corp.local
```
What `goldenPac.py` does:
1. Requests TGT without PAC.
2. Forges a PAC with Domain Admin SIDs (S-1-5-21-...-512).
3. Signs it with MD5.
4. Requests TGS for the `cifs` service on the DC, providing the forged PAC.
5. Injects the returned ticket and drops an SMB execution shell (PsExec-style).

You are dropped into a `C:\Windows\system32>` shell as `NT AUTHORITY\SYSTEM`.

### 4.3 Using PyKEK
Alternatively, the original Python Kerberos Exploitation Kit (PyKEK) can be used to generate a `.ccache` ticket file.
```bash
python ms14-068.py -u john@corp.local -s S-1-5-21-123456-789012-345678-1001 -d dc01.corp.local -p Password123
```
This generates a file like `TGT_john@corp.local.ccache`.
You export this ticket into your environment variables:
```bash
export KRB5CCNAME=/path/to/TGT_john@corp.local.ccache
```
You can now use `wmiexec.py` or `smbclient` with Kerberos authentication (`-k`) to access the DC as an admin.

## 5. Detection Mechanisms

### 5.1 Event Logs
Because MS14-068 manipulation occurs during the Kerberos AS and TGS exchanges, advanced AD logging can catch anomalies.
- **Event ID 4769 (Kerberos Service Ticket Requested)**: Pay attention to the `Ticket Options` field. A ticket requested without a PAC will have specific bits set.
- **Event ID 4768 (TGT Request)**: Watch for standard users requesting TGTs without PACs.
- **Event ID 4624 (Logon)**: A logon event where the user is a standard user, but the access token contains `Domain Admins` SIDs.
- Anomaly: A user accessing Domain Controller network shares (`C$`, `ADMIN$`) who is mathematically not part of the Domain Admins group in the AD database.

### 5.2 Network IDS / IPS
Network monitoring can inspect Kerberos traffic on Port 88. Snort rules exist that specifically look for PACs where the signature algorithm is defined as MD5 (Algorithm ID `0xFFFFFF76` or similar non-standard values).

### 5.3 Behavioral Analytics
Monitor for sudden changes in privilege levels without corresponding group membership modifications. A user suddenly performing DCSync operations without being in a privileged group is highly indicative of ticket manipulation.

## 6. Mitigation
The fix for MS14-068 was released in November 2014 via patch `KB3011780`. 
The patch ensures the KDC strictly validates the signature algorithm. If a PAC is supplied by a client, the KDC enforces that it must be signed by the `krbtgt` account using a keyed HMAC algorithm, totally nullifying the MD5 downgrade attack.

Because this vulnerability is over a decade old, it is rarely seen in modern, well-maintained environments. However, it frequently appears in CTF environments (like HackTheBox or TryHackMe), legacy OT networks, or unpatched staging environments.

## 7. Conclusion
MS14-068 is a masterclass in how complex protocols can harbor devastating logical flaws. It proved that cryptographic algorithms are only as secure as their implementation. The decision to allow the client to dictate the cryptographic algorithm to the server is an anti-pattern that led directly to one of the easiest and most lethal AD privilege escalations in history.

## Real-World Attack Scenario

During a penetration test for a local municipality utilizing a heavily outdated legacy SCADA network, the red team obtained standard credentials for a maintenance user, `m.worker`. After mapping the network, the attacker realized the legacy domain's primary Domain Controller, a Windows Server 2008 R2 machine, lacked a decade's worth of updates, making it vulnerable to MS14-068.

The attacker's objective was to quickly elevate privileges from the standard `m.worker` account to Domain Admin without tripping legacy AV heuristics with memory injection or dropping binaries. They first queried the domain for their own user's SID using `rpcclient`:
```bash
rpcclient -U m.worker -c "lookupnames m.worker" 192.168.10.5
```
This returned `S-1-5-21-3333-4444-5555-1123`.

Next, the attacker used the Python script `goldenPac.py` from the Impacket suite. They passed the cleartext password and the target Domain Controller's IP:
```bash
python3 goldenPac.py 'scada.local/m.worker:Winter2014!'@dc01.scada.local
```

Under the hood, the script sent a Kerberos AS-REQ to the DC requesting a TGT *without* a Privilege Attribute Certificate (PAC). It then forged a custom PAC containing the SID of the `Domain Admins` group (`-512`), signed it locally using the unkeyed MD5 algorithm, and submitted it via a TGS-REQ. The unpatched KDC accepted the MD5 signature and returned a valid Service Ticket containing Domain Admin privileges.

The `goldenPac` script instantly leveraged this forged ticket to authenticate to the `ADMIN$` share of the Domain Controller, dropping a Python-based PsExec shell. The attacker was immediately presented with a `C:\Windows\system32>` command prompt running as `NT AUTHORITY\SYSTEM`, achieving total domain compromise from a standard user account in less than ten seconds via a pure cryptographic logic flaw.
