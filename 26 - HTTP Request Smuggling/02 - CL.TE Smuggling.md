# 26.02 — CL.TE Smuggling

## What is it?
**CL.TE Smuggling** stands for **Content-Length / Transfer-Encoding**. It is a specific type of HTTP Request Smuggling where the **Front-End server uses the `Content-Length` (CL)** header to determine the size of the request, and the **Back-End server uses the `Transfer-Encoding` (TE)** header.

Because the front-end trusts the `Content-Length`, it reads the entire payload (including your hidden smuggled request) and forwards the whole block to the back-end over the shared TCP connection.

Because the back-end trusts `Transfer-Encoding: chunked`, it parses the incoming data as chunks. As soon as it encounters a chunk of size `0` (the zero-length chunk, which means "end of message" in chunked encoding), it stops reading. Any data sitting in the TCP stream *after* that `0` is ignored for the current request. The back-end assumes that leftover data must be the beginning of the *next* legitimate HTTP request coming down the pipeline.

Think of it like a train. The front-end (Ticket Inspector) counts the number of train cars (Content-Length = 5 cars) and lets the train through. The back-end (Station Manager) doesn't count cars; they just look for the red caboose (Transfer-Encoding: chunked ending with '0'). If the attacker puts a red caboose on the 3rd car, the Station Manager detaches cars 4 and 5, leaving them sitting on the track. When the *next* train arrives, it crashes right into cars 4 and 5.

## How to Find It
- **Manual steps:**
  1. Send a request to the target with both headers.
  2. To detect CL.TE, you use a **Time-Delay payload**.
  3. You send a payload where the CL is short, but the TE chunk implies more data is coming.
  4. If the server hangs (waits for data), it confirms the back-end is using TE.

- **The CL.TE Timeout Probe:**
  ```http
  POST / HTTP/1.1
  Host: vulnerable.com
  Transfer-Encoding: chunked
  Content-Length: 4
  
  1
  A
  X
  ```
  *Explanation:* The Front-End reads 4 bytes (`1\r\nA\r\n`). It forwards them. The Back-End sees chunked encoding. It reads chunk size `1` (which is 'A'). It then expects the next chunk size. But the Front-End only sent 4 bytes and stopped! The Back-End hangs, waiting for the rest of the chunks, causing a noticeable timeout (e.g., 10 seconds). This confirms CL.TE.

## How to Exploit It
- **Step-by-step walkthrough:**
  1. Once CL.TE is confirmed via the timeout probe, construct your malicious prefix.
  2. Your prefix is the HTTP request you want the *next* user to execute.
  3. Wrap the prefix in a CL.TE payload.
  4. Ensure your `Content-Length` exactly matches the length of the body (from the `0` chunk down to the end of your smuggled prefix). Every `\r\n` counts as 2 bytes.
  5. Send the payload. Immediately send a normal request from a different browser to act as the "victim". If your normal request returns the response intended for the smuggled request, you successfully exploited the pipeline.

- **Actual payloads:**
  **Smuggling a 404 Request (Verification):**
  ```http
  POST / HTTP/1.1
  Host: vulnerable.com
  Content-Type: application/x-www-form-urlencoded
  Content-Length: 49
  Transfer-Encoding: chunked
  
  0
  
  GET /does_not_exist HTTP/1.1
  X-Ignore: X
  ```
  *(Note: The `X-Ignore: X` header at the end is crucial. When the victim's request is appended, it will look like `X-Ignore: XGET /home HTTP/1.1`. This absorbs the victim's request line into a useless header, ensuring your smuggled `GET /does_not_exist` is the only valid HTTP method the back-end processes).*

## Real-World Example
An attacker found a CL.TE vulnerability on a banking application. The bank had an endpoint `POST /api/transfer` that required a CSRF token. The attacker couldn't guess the CSRF token of other users. However, they smuggled a request: `POST /api/transfer HTTP/1.1\r\nHost: bank.com\r\nAmount: 1000\r\nTo: Attacker\r\nContent-Length: 100\r\nX-Ignore: `. 
When a legitimate logged-in user navigated to the homepage, their browser sent a `GET /` request containing their Session Cookie and their CSRF token. The back-end concatenated this onto the attacker's smuggled prefix. The back-end parsed it as a valid transfer request, authorized by the victim's appended cookies and tokens, stealing $1,000 without the victim ever clicking a malicious link.

## How to Fix It
- **Developer remediation:**
  Configure the front-end proxy (e.g., HAProxy, Nginx) to strictly enforce RFC 7230: "If a message is received with both a Transfer-Encoding header field and a Content-Length header field, the latter MUST be ignored." 
  Furthermore, the proxy should normalize the request: it should either reject the request entirely (400 Bad Request) or parse the chunked body, strip the `Transfer-Encoding` header, calculate the true `Content-Length`, and forward it to the back-end as a standard, non-chunked request.
