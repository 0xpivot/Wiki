# 26.03 — TE.CL Smuggling

## What is it?
**TE.CL Smuggling** stands for **Transfer-Encoding / Content-Length**. It is the inverse of CL.TE smuggling. In this scenario, the **Front-End server uses the `Transfer-Encoding` (TE)** header, while the **Back-End server uses the `Content-Length` (CL)** header.

Because the front-end trusts `Transfer-Encoding: chunked`, it reads the payload chunk by chunk until it hits the `0` chunk (the end of the message). It then forwards this chunked stream to the back-end.

Because the back-end trusts `Content-Length`, it reads exactly the number of bytes specified in the CL header. If the attacker sets the CL header to a very small number, the back-end reads only the first few bytes of the chunked stream and stops. The rest of the chunked payload (the smuggled request) is left sitting dead in the TCP socket, waiting to attach itself to the next user's legitimate request.

Think of it like a conveyer belt. The front-end (TE) reads the box sizes and says "Okay, here's a size 10 box, here's a size 5 box, and here's a size 0 box. Done!" and pushes them all onto the belt to the backend. The backend (CL) looks at the shipping manifest and says "I am only supposed to pick up 4 pounds of boxes." It picks up the first 4 pounds and goes on break. The remaining boxes sit on the belt. When the next worker turns on the belt, those leftover boxes crash into the new shipment.

```text
[2. Front-End Logic (TE)]
"Transfer-Encoding chunked. I read the 5c chunk, then the 0 chunk. The request is complete. Forwarding everything."

[3. Back-End Logic (CL)]
"Content-Length is 4. I will read exactly 4 bytes."
Reads: "5c\r\n" (Exactly 4 bytes!).
"Okay, my request is done! I will process it."
"Wait, there is a giant `POST /admin/add_user` sitting in my TCP socket. I will save it for the next user."

[4. Next Legitimate User sends request]
GET / HTTP/1.1
Cookie: admin=true

[5. Back-End combines them!]
POST /admin/add_user HTTP/1.1
Host: vulnerable.com
...
{"user":"hax"}
0
GET / HTTP/1.1
Cookie: admin=true

[Result: The smuggled POST request is executed with the victim's admin cookie!]
================================================================================
```

## How to Find It
- **Manual steps:**
  1. Send a request to the target with both headers.
  2. To detect TE.CL, you use a **Time-Delay payload**.
  3. You send a payload where the Front-End (TE) thinks the request is over, but the Back-End (CL) is told to wait for a massive amount of data that never comes.
  4. If the server hangs, it confirms the back-end is using CL.

- **The TE.CL Timeout Probe:**
  ```http
  POST / HTTP/1.1
  Host: vulnerable.com
  Transfer-Encoding: chunked
  Content-Length: 6
  
  0
  
  X
  ```
  *Explanation:* The Front-End (TE) sees the `0` chunk and forwards the request immediately. The Back-End (CL) sees `Content-Length: 6`. It receives the `0\r\n\r\n` (which is 5 bytes). It is waiting for the 6th byte! Because the Front-End already finished sending and closed its stream, the Back-End hangs, waiting for 1 more byte that will never arrive. This timeout confirms TE.CL.

## How to Exploit It
- **Step-by-step walkthrough:**
  1. Construct your malicious prefix (the smuggled request).
  2. Calculate the exact length of your prefix in hexadecimal.
  3. Put the hex length on the line above your prefix (creating a valid chunk).
  4. Ensure your `Content-Length` header is set to exactly the length of the string before the smuggled chunk (usually the hex length + `\r\n` = 3 or 4 bytes).
  5. End the entire payload with a `0\r\n\r\n` chunk so the Front-End knows the request is over.

- **Actual payloads:**
  **Smuggling a POST request to an internal endpoint:**
  ```http
  POST / HTTP/1.1
  Host: vulnerable.com
  Content-Length: 4
  Transfer-Encoding: chunked
  
  5e
  POST /internal/reset_password HTTP/1.1
  Host: vulnerable.com
  Content-Length: 100
  
  user=admin
  0
  
  ```
  *(Note: The `Content-Length` of the carrier request is 4 because it only reads `5e\r\n` before stopping. The smuggled request starts exactly at the 'P' in POST).*

## Real-World Example
During a bug bounty on a massive SaaS platform, a researcher discovered the Cloudflare WAF (Front-End) processed `Transfer-Encoding`, but the internal HAProxy (Back-End) relied on `Content-Length` due to a misconfiguration in HTTP/1.0 downgrading. The researcher crafted a TE.CL payload that smuggled a request to the `/api/cache/purge` endpoint. Because the internal HAProxy saw the smuggled request as originating from the trusted Cloudflare IP address over the established TCP socket, it bypassed the IP whitelists and purged the entire global cache of the SaaS platform, causing a massive denial of service.

## How to Fix It
- **Developer remediation:**
  1. **Consistent Parsing:** Ensure all servers in the proxy chain use exactly the same HTTP parsing library or configuration.
  2. **Reject Ambiguity:** The Front-End must reject (400 Bad Request) any request containing both headers. 
  3. If rejecting is impossible, the Front-End MUST rewrite the request. It should consume the chunked body, calculate the total size, remove the `Transfer-Encoding` header entirely, and inject a new, hardcoded `Content-Length` header before passing it to the Back-End.
