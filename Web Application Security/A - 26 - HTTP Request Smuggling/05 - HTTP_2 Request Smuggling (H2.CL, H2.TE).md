# 26.05 — HTTP/2 Request Smuggling (H2.CL, H2.TE)

## What is it?
HTTP/2 was supposed to fix Request Smuggling. HTTP/2 uses a binary framing mechanism where the exact length of the data is explicitly defined in the frame header. There is no `Content-Length` string to parse, and `Transfer-Encoding: chunked` is strictly forbidden in the HTTP/2 specification. 

So how does smuggling happen? **HTTP/2 Downgrading**.

Most architectures are not HTTP/2 from the user all the way to the backend database. Usually, the Front-End proxy (like Cloudflare or AWS API Gateway) accepts HTTP/2 from the user's browser, but to communicate with older internal services, it *downgrades* the request to HTTP/1.1. 

**H2.CL** and **H2.TE** smuggling occur when the attacker injects standard HTTP/1.1 headers (`Content-Length` or `Transfer-Encoding`) *inside* an HTTP/2 header block. The Front-End proxy accepts them because in HTTP/2, headers are just arbitrary key-value pairs. But when the Front-End translates the request into HTTP/1.1 to send to the Back-End, it blindly writes those malicious headers into the new HTTP/1.1 stream. 

The Back-End server receives an HTTP/1.1 request containing both a valid `Content-Length` (generated by the proxy) and the attacker's injected `Transfer-Encoding` header, triggering a classic CL.TE or TE.CL smuggling attack!

Think of it like translating a letter from English to French. In English (HTTP/2), you write a paragraph that says "PS: Read only the first half of this letter." The translator (Front-End) doesn't understand the context; they just translate it word-for-word into French (HTTP/1.1). The French reader (Back-End) reads the translated instruction and stops halfway through, leaving the rest of the letter to be attached to the next person's mail.

## How to Find It
- **Manual steps:**
  1. Ensure the target supports HTTP/2 (most modern sites do).
  2. Use Burp Suite Professional. Go to Project Options -> HTTP -> HTTP/2 and ensure "Enable HTTP/2" is checked.
  3. Send a request to Repeater. Go to the Inspector panel on the right side.
  4. Under "Request Attributes", manually add a header. Name: `transfer-encoding`, Value: `chunked`.
  5. Add a smuggled payload to the body (e.g., the `0\r\n\r\nSMUGGLED` payload).
  6. Send the request. If the backend hangs, or if you receive a weird response on a subsequent request, the downgrade process is vulnerable.
  7. **H2.CL:** Instead of injecting `transfer-encoding`, inject a custom `content-length` header that conflicts with the actual size of your data frame.

## How to Exploit It
- **Step-by-step walkthrough:**
  1. Identify if the backend prefers CL or TE when receiving conflicting HTTP/1.1 headers.
  2. **If Backend prefers TE (H2.TE):** Inject `transfer-encoding: chunked` into your HTTP/2 headers. Write your body data formatted as chunked encoding, containing your smuggled request.
  3. **If Backend prefers CL (H2.CL):** Inject `content-length: [small_number]` into your HTTP/2 headers. The proxy will forward your entire HTTP/2 data frame, but the backend will only read `[small_number]` bytes.

- **Actual payloads:**
  **H2.TE Attack via Burp Repeater (Inspector):**
  *(Headers added via Inspector UI, not text editor)*
  ```text
  :method: POST
  :path: /
  :authority: target.com
  transfer-encoding: chunked
  ```
  *(Body)*
  ```text
  0
  
  GET /admin/delete HTTP/1.1
  Host: target.com
  Cookie: session=attacker
  Content-Length: 10
  
  x=
  ```

## Real-World Example
In 2021, James Kettle revealed HTTP/2 Request Smuggling. He targeted Netflix. Netflix's Front-End was handling HTTP/2, but downgrading to HTTP/1.1 for internal microservices. The Front-End did not strip the `transfer-encoding` header from the HTTP/2 stream. Kettle injected `transfer-encoding: chunked` and successfully smuggled requests into Netflix's backend. He used this to achieve **Response Queue Poisoning**, redirecting JavaScript files meant for other users to his own server, effectively gaining stored XSS on Netflix's homepage for random users.

## How to Fix It
- **Developer remediation:**
  1. **End-to-End HTTP/2:** Do not downgrade to HTTP/1.1. If the internal microservices support HTTP/2, use it the entire way down the chain.
  2. **Strict Protocol Translation Validation:** The Front-End proxy MUST strip or reject any HTTP/2 request that contains HTTP/1.1 connection-specific headers like `Transfer-Encoding`, `Connection`, `Keep-Alive`, or `Upgrade`. 
  3. **Content-Length Validation:** If downgrading, the proxy must ensure it generates a singular, accurate `Content-Length` header based on the HTTP/2 DATA frames, and strictly reject any user-supplied `content-length` header in the HTTP/2 stream.
