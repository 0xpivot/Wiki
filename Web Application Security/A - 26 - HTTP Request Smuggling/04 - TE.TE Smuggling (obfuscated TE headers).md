# 26.04 — TE.TE Smuggling (Obfuscated TE headers)

## What is it?
**TE.TE Smuggling** occurs when the attacker sends a request containing *two* `Transfer-Encoding` headers (or one heavily obfuscated TE header). Both the Front-End and the Back-End server technically support `Transfer-Encoding: chunked`. However, because of how the header is obfuscated, one of the servers fails to recognize it, while the other server parses it successfully.

If the Front-End gets confused by the obfuscation and decides to ignore the TE header, it falls back to using the `Content-Length` header (resulting in a TE.CL attack). If the Back-End gets confused and ignores the TE header, it falls back to the `Content-Length` header (resulting in a CL.TE attack).

The root cause is a discrepancy in how different HTTP parsers handle malformed syntax (spaces, tabs, capitalization, null bytes, or unexpected characters). 

Think of it like a bouncer (Front-End) and a bartender (Back-End) looking at a fake ID. The fake ID has the birth year written in strange handwriting. The bouncer says, "I can't read that, I'm just going to look at the physical description" (Falling back to CL). The bartender says, "Ah, it says 1990, here's your drink" (Successfully parsing TE). The discrepancy in interpretation allows the attacker to slip the smuggling payload through.

## How to Find It
- **Manual steps:**
  1. Identify the target application.
  2. You must fuzz the `Transfer-Encoding` header with various obfuscation techniques to find a permutation that one server processes but the other ignores.
  3. Send the obfuscated payload using the Time-Delay probes (from CL.TE and TE.CL). If the server times out, you have found a working obfuscation.

- **Fuzzing Payloads for TE.TE:**
  *These must be sent exactly as written, preserving spaces and casing.*
  ```http
  Transfer-Encoding: xchunked
  Transfer-Encoding : chunked
  Transfer-Encoding: chunked
  Transfer-Encoding: x
  Transfer-Encoding:[tab]chunked
  [space]Transfer-Encoding: chunked
  X: X[\n]Transfer-Encoding: chunked
  Transfer-Encoding
  : chunked
  ```

## How to Exploit It
- **Step-by-step walkthrough:**
  1. Use Burp Suite's "HTTP Request Smuggler" extension or manually send obfuscated headers to find which server drops the TE header.
  2. If the Front-End drops the TE header and uses CL, but the Back-End processes it, you have a **CL.TE** scenario. Build a CL.TE payload (See [[02 - CL.TE Smuggling]]) but replace the normal TE header with your working obfuscated TE header.
  3. If the Front-End processes the TE header, but the Back-End drops it and uses CL, you have a **TE.CL** scenario. Build a TE.CL payload (See [[03 - TE.CL Smuggling]]) using your obfuscated header.

- **Actual payloads:**
  **Example Payload (Resulting in TE.CL because Backend fails to parse the space before the colon):**
  ```http
  POST / HTTP/1.1
  Host: vulnerable.com
  Content-length: 4
  Transfer-Encoding : chunked

  5c
  GET /admin HTTP/1.1
  Host: vulnerable.com
  Content-Length: 15

  x=1
  0

  ```

## Real-World Example
A Bug Bounty hunter was testing a large payment gateway. They sent standard CL.TE and TE.CL probes, but the front-end WAF blocked requests containing both headers. The hunter bypassed the WAF by sending `Transfer-Encoding: chunked\r\nTransfer-Encoding: x`. The WAF's regex engine got confused by the second invalid header and decided to strip *both* TE headers from the request, falling back to `Content-Length`. However, before forwarding, the WAF re-wrote the request and accidentally re-inserted a clean `Transfer-Encoding: chunked` header because it noticed the body was chunked! The back-end parsed the chunked body, while the WAF used the Content-Length. This TE.TE obfuscation completely bypassed the WAF's smuggling protections.

## How to Fix It
- **Developer remediation:**
  1. **HTTP/2:** Use HTTP/2 end-to-end to eliminate chunked encoding ambiguity entirely.
  2. **Reject Multiple Headers:** If an HTTP request contains more than one `Transfer-Encoding` header, or if it contains both `Content-Length` and `Transfer-Encoding`, the Front-End proxy MUST immediately close the connection and return a 400 Bad Request. It should not attempt to "guess" or "normalize" malformed headers.
