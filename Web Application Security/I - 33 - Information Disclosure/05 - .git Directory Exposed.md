# .git Directory Exposed

## 1. Executive Summary and Definition

A `.git` directory exposure occurs when a web application's root directory or one of its subdirectories contains a globally accessible `.git` folder. Git is the most widely used version control system, and when a developer initializes a repository (using `git init` or `git clone`), a hidden directory named `.git` is created. This directory contains the complete history of the repository, including all commits, branches, configuration settings, and object representations of every file that has ever been tracked by Git.

When an application is deployed simply by copying or cloning a Git repository directly into the web server's document root (e.g., `/var/www/html/`) without restricting access to hidden directories, the web server inadvertently makes the `.git` directory available over HTTP/HTTPS. 

This is a critical information disclosure vulnerability. An attacker who discovers an exposed `.git` directory can reconstruct the entire application source code, effectively turning a black-box environment into a white-box environment. Even worse, the attacker gains access to the entire commit history, allowing them to extract sensitive information (like hardcoded passwords, API keys, and database credentials) that might have been accidentally committed and later "removed" in subsequent commits.

## 2. Technical Deep Dive into the .git Directory Structure

To understand how this vulnerability is exploited, one must understand the internal architecture of the `.git` directory. It is not merely a backup of files; it is a complex database of objects and references.

Key components of the `.git` directory:
- **`HEAD`**: A text file pointing to the current branch reference (e.g., `ref: refs/heads/master`).
- **`config`**: A configuration file that contains repository-specific settings, including the URL of the remote repository (which might contain credentials in the URL, e.g., `https://user:pass@github.com/org/repo.git`).
- **`index`**: A binary file that tracks the staging area. It contains a list of all tracked files, their paths, permissions, and SHA-1 hashes (object IDs).
- **`objects/`**: The core database of Git. It stores commits, trees (directory structures), and blobs (file contents). They are stored in subdirectories named after the first two characters of their SHA-1 hash, and the filename is the remaining 38 characters.
- **`refs/`**: Contains pointers to specific commit objects for branches (`refs/heads/`) and tags (`refs/tags/`).
- **`logs/`**: Contains the `HEAD` history, which can be useful to find orphaned commits.

Because web servers generally allow the downloading of static files, an attacker can simply send HTTP GET requests to read these components one by one.

### 2.1 The Mechanism of Exposure

The most common root cause is flawed deployment practices:
1. **Direct cloning to webroot:** A developer connects to the production server and runs `git clone https://... /var/www/html`.
2. **Improper rsync/FTP synchronization:** A developer syncs their entire local project folder—including the hidden `.git` directory—to the production server via FTP or an unconfigured `rsync` script.
3. **Misconfigured static site generators:** Some SSGs output the build folder, but if the deployment script copies the entire repository tree instead of just the `build/` or `public/` directory, the `.git` folder is exposed.
4. **Lack of Web Server Access Controls:** The web server (Apache/Nginx/IIS) lacks specific rules to block access to directories starting with a dot (`.`).

## 4. Exploitation and Tooling

### 4.1 Identification
During the reconnaissance phase, penetration testers and automated scanners will attempt to access standard Git paths. 

```bash
curl -s http://target.com/.git/HEAD
curl -s http://target.com/.git/config
```
If the server responds with a `200 OK` and content like `ref: refs/heads/master` or `[core]`, the vulnerability is confirmed.

### 4.2 Exploitation and Extraction
Because directory listing is usually disabled, an attacker cannot simply browse the `.git` folder and click "download." Instead, they must systematically request the known files and deduce the locations of objects.

Tools like **GitTools** (specifically `git-dumper.sh`) or **git-dumper (Python)** automate this complex process:
1. Download `/.git/HEAD`, `/.git/config`, `/.git/index`.
2. Parse the binary `index` file to extract the SHA-1 hashes of all currently tracked files.
3. For each SHA-1 hash (e.g., `d670460b4b4aece5915caf5c68d12f560a9fe3e4`), issue a request to download the corresponding object file: `/.git/objects/d6/70460b4b4aece5915caf5c68d12f560a9fe3e4`.
4. Download pack files (`.pack` and `.idx`) if they exist in `/.git/objects/pack/`.

**Example command using git-dumper:**
```bash
git-dumper http://target.com/.git/ website_repo/
```

### 4.3 Reconstructing and Analyzing the Repository
Once the `.git` directory is fully downloaded, the attacker navigates into the local folder and runs:
```bash
cd website_repo/
git checkout .
```
This command restores the working tree, placing all the application source code files in the directory.

To hunt for secrets, the attacker can use tools like **TruffleHog** or **Gitleaks** to scan the entire commit history:
```bash
trufflehog git file:///path/to/website_repo/
```
Alternatively, an attacker can manually search through the commit log for sensitive changes:
```bash
git log -p | grep -iE "password|secret|api_key|token|bearer"
```

## 5. Impact and Risk Assessment

The impact of an exposed `.git` directory is classified as **Critical**.

1. **Complete Source Code Disclosure:** Access to the full source code allows attackers to perform white-box analysis, uncovering SQL injections, IDORs, authentication bypasses, and logic flaws that would be difficult or impossible to find otherwise.
2. **Exposure of Historical Secrets:** Developers frequently commit sensitive data (AWS keys, DB passwords) by mistake, realize the error, and delete the data in a subsequent commit. However, the secret remains forever in the Git history. An exposed `.git` folder compromises these historical secrets.
3. **Repository Hijacking Potential:** If the `.git/config` file contains access tokens or embedded credentials in the remote origin URL, the attacker could use those credentials to push malicious code back to the central repository (e.g., GitHub, GitLab), executing a supply chain attack.

## 6. Remediation and Prevention Strategies

Preventing this vulnerability requires robust deployment procedures and strict server configurations.

### 6.1 Web Server Access Control
The most immediate fix is to configure the web server to block access to all hidden files and directories (those starting with a dot).

**Apache Configuration (`.htaccess` or `httpd.conf`):**
```apache
<IfModule mod_rewrite.c>
    RewriteEngine On
    RewriteRule "(^|/)\.(?!well-known)" - [F,L]
</IfModule>
```
Or simply:
```apache
<DirectoryMatch "^\.|\/\.">
    Require all denied
</DirectoryMatch>
```

**Nginx Configuration (`nginx.conf`):**
```nginx
server {
    ...
    location ~ /\.(?!well-known).* {
        deny all;
        access_log off;
        log_not_found off;
    }
}
```

### 6.2 Secure CI/CD Deployment Pipelines
Do not use `git clone` or `git pull` directly in the document root of the production server. Instead, use a build artifact system:
1. A CI/CD platform (e.g., Jenkins, GitHub Actions) pulls the code.
2. It compiles/builds the application.
3. It packages *only* the necessary executable files and static assets (excluding `.git`, `.env`, tests, and markdown files) into a `.zip` or `.tar.gz` archive.
4. The archive is securely transferred to the production server via SSH/SCP and extracted into the webroot.

## 7. Chaining Opportunities

- **.git Exposure -> Remote Code Execution (RCE):** Discovering an insecure file upload implementation or insecure deserialization flaw in the source code allows the attacker to craft a payload for RCE.
- **.git Exposure -> Database Takeover:** Extracting database credentials from old commits allows the attacker to connect directly to the database if it is exposed to the internet, or leverage an internal Server-Side Request Forgery (SSRF) to interact with it.
- **.git Exposure -> Supply Chain Attack:** Utilizing credentials found in `.git/config` to compromise the master repository and insert backdoors into future releases of the application.
