# Origin Header Spoofing

## 1. Introduction to Origin Header Spoofing
Origin Header Spoofing is a technique used to exploit misconfigurations in Cross-Origin Resource Sharing (CORS) implementations. The `Origin` HTTP header is automatically added by web browsers to cross-origin requests, indicating the scheme, hostname, and port of the document that initiated the request. Servers rely on this header to determine whether the requesting origin is authorized to access the requested resources, usually by reflecting it in the `Access-Control-Allow-Origin` (ACAO) response header.

When backend servers implement flawed validation logic for the `Origin` header—such as weak regular expressions or trusting arbitrary inputs—attackers can spoof or manipulate this header to bypass CORS restrictions, leading to severe data exfiltration or unauthorized actions on behalf of authenticated users.

## 2. The Mechanics of CORS and Origin
CORS is a browser security mechanism that relaxes the Same-Origin Policy (SOP). Under SOP, a web page can only request resources from the exact same origin (scheme, domain, and port). To allow legitimate cross-domain requests (e.g., a frontend on `app.com` fetching API data from `api.com`), CORS must be explicitly configured.

When `app.com` makes an AJAX request to `api.com`, the browser sends:
`Origin: https://app.com`

The server at `api.com` checks this Origin. If authorized, it responds with:
`Access-Control-Allow-Origin: https://app.com`
`Access-Control-Allow-Credentials: true`

This tells the browser: "It is safe to expose this response to the JavaScript running on app.com."

## 4. Vulnerability Mechanics in Detail

### 4.1 Regex Vulnerabilities and Prefix/Suffix Matches
Developers often use regular expressions to dynamically generate the ACAO header to support multiple subdomains. Poorly written regexes are a primary source of Origin spoofing.
- **Suffix matching flaw:** If the server checks if the origin *ends with* `example.com`, an attacker can register `notexample.com` or `maliciousexample.com`.
- **Prefix matching flaw:** Checking if it *starts with* `https://example.com` allows origins like `https://example.com.attacker.com`.
- **Unescaped dots:** The regex `^https://api\.example\.com` is secure. However, `^https://api.example.com` (where dots are unescaped) matches `https://apiXexample.com`, allowing an attacker to bypass the check by registering that domain.

### 4.2 The "null" Origin
Local HTML files, data URIs, or sandboxed iframes send the Origin header as `Origin: null`.
Some frameworks or developers explicitly whitelist the `null` origin to support local development or specific edge cases.
An attacker can force a request to have a `null` origin by initiating it from an `iframe` with the `sandbox` attribute.
```html
<iframe sandbox="allow-scripts allow-top-navigation allow-forms" src="data:text/html,<script>
  var req = new XMLHttpRequest();
  req.onload = reqListener;
  req.open('get','https://vulnerable.com/api/data',true);
  req.withCredentials = true;
  req.send();
  function reqListener() { alert(this.responseText); }
</script>"></iframe>
```
If the server responds with `Access-Control-Allow-Origin: null`, the exploit succeeds.

### 4.3 Wildcard with Credentials
A server might respond with `Access-Control-Allow-Origin: *`. However, the CORS specification dictates that if `Access-Control-Allow-Credentials: true` is set (meaning cookies/sessions are sent), the ACAO header *cannot* be a wildcard `*`; it must specify an exact domain.
To bypass this, developers dynamically reflect whatever Origin is sent in the request back into the ACAO header:
```http
Request: Origin: https://evil.com
Response: Access-Control-Allow-Origin: https://evil.com
          Access-Control-Allow-Credentials: true
```
This complete trust in the Origin header is a critical vulnerability, leading to complete data exfiltration for authenticated users.

## 5. Exploitation Scenarios

### 5.1 Data Theft via CORS
The most direct impact. An attacker hosts a malicious page. A victim logs into the vulnerable application, then visits the attacker's page. The attacker's JavaScript issues an XHR/Fetch request to the vulnerable API. Because the CORS configuration reflects the spoofed Origin, the browser allows the attacker's script to read the sensitive JSON response (e.g., PII, API keys, CSRF tokens).

### 5.2 Bypassing CSRF Defenses
If an application uses CORS as its primary defense against Cross-Site Request Forgery (CSRF) by ensuring that requests only come from trusted Origins, Origin spoofing breaks this defense. By spoofing the Origin or registering a domain that bypasses the regex, the attacker can execute state-changing requests (like changing a password) on behalf of the victim.

### 5.3 Cross-Site WebSocket Hijacking (CSWSH)
WebSockets do not enforce the Same-Origin Policy. Instead, the server is responsible for verifying the `Origin` header during the initial HTTP handshake. If the server is vulnerable to Origin spoofing, an attacker can initiate a WebSocket connection from their malicious domain, read messages intended for the victim, and send messages as the victim.

## 6. Advanced Techniques

### 6.1 HTTP Header Injection
If the server has an HTTP Response Splitting or Header Injection vulnerability in how it handles the Origin, an attacker might inject CRLF sequences to manipulate other headers or the response body.
Example payload: `Origin: https://example.com%0d%0aSet-Cookie:%20session=hacked`

### 6.2 Proxy and Load Balancer Quirks
Sometimes the backend validates the Origin securely, but a reverse proxy in front alters the headers or caches responses improperly based on the Origin header, leading to cache poisoning scenarios.

## 7. Mitigation and Defense

### 7.1 Strict Allowlisting
Never use wildcards `*` in conjunction with `Access-Control-Allow-Credentials: true`. Do not reflect arbitrary Origins.
Maintain a strict, hardcoded allowlist of permitted origins.

### 7.2 Secure URL Parsing
When validating the Origin header, use robust URL parsing libraries rather than regular expressions. Extract the hostname and compare it exactly against the allowlist.
If regex must be used, ensure it is anchored correctly (`^` and `$`) and all literal dots are properly escaped (`\.`).

Example secure validation in Express.js:
```javascript
const allowedOrigins = ['https://trusted.com', 'https://api.trusted.com'];
app.use(cors({
  origin: function(origin, callback){
    if(!origin) return callback(null, true);
    if(allowedOrigins.indexOf(origin) === -1){
      return callback(new Error('CORS policy violation'), false);
    }
    return callback(null, true);
  }
}));
```

### 7.3 Avoid Whitelisting "null"
Never configure `Access-Control-Allow-Origin: null`. This is inherently insecure and bypasses the entire purpose of CORS.

### 7.4 Implement Defense in Depth
Do not rely solely on CORS for CSRF protection. Implement strong Anti-CSRF tokens for all state-changing operations.

## 8. Summary
Origin spoofing demonstrates the danger of relying on client-provided data for security boundaries. Robust validation, strict allowlists, and avoiding dynamic reflection are essential for securing CORS implementations.
