# ECB Mode Encryption — Block Boundary Manipulation

## 1. Introduction to Electronic Codebook (ECB) Mode

Electronic Codebook (ECB) is the simplest of the encryption modes of operation for a block cipher. A block cipher, such as AES (Advanced Encryption Standard), operates on fixed-length groups of bits called blocks. In the case of AES, the block size is consistently 128 bits (16 bytes).

When using ECB mode, the plaintext is divided into blocks of the cipher's block size, and each block is encrypted independently using the same cryptographic key. If the plaintext length is not a multiple of the block size, it must be padded (typically using PKCS#7 padding) before encryption.

### The Core Flaw of ECB
The fundamental weakness of ECB mode is its lack of diffusion. Because each block is encrypted entirely independently of the others, identical plaintext blocks are encrypted into identical ciphertext blocks. This means that data patterns in the plaintext are preserved in the ciphertext. This characteristic makes ECB highly vulnerable to pattern recognition (famously demonstrated by the "ECB Penguin" image) and block manipulation attacks, such as Block Boundary Manipulation, Cut-and-Paste attacks, and Byte-at-a-time decryption.

## 2. Visualizing the ECB Process and Vulnerability

### ASCII Diagram: ECB Mode Encryption

```text
Plaintext:
[ Block 1 (16 bytes) ]  [ Block 2 (16 bytes) ]  [ Block 3 (16 bytes) ]
          |                       |                       |
          v                       v                       v
    +-----------+           +-----------+           +-----------+
    | Block     |           | Block     |           | Block     |
    | Cipher    |<---Key--->| Cipher    |<---Key--->| Cipher    |
    | Encrypt   |           | Encrypt   |           | Encrypt   |
    +-----------+           +-----------+           +-----------+
          |                       |                       |
          v                       v                       v
[Ciphertext Block 1 ]   [Ciphertext Block 2 ]   [Ciphertext Block 3 ]
```

As visualized above, there is no chaining mechanism (like an Initialization Vector or feedback loop). If `Block 1` is identical to `Block 3`, then `Ciphertext Block 1` will be mathematically identical to `Ciphertext Block 3`. This simple property gives rise to extremely powerful cryptographic attacks when user input is concatenated with sensitive data and encrypted.

## 3. The Cut-and-Paste Attack

A Cut-and-Paste attack leverages the independent nature of ECB blocks. An attacker who controls part of the plaintext can craft inputs such that specific, desired ciphertext blocks are generated. They can then capture these ciphertext blocks and rearrange them, replacing blocks in the target ciphertext to manipulate the underlying decrypted plaintext, without ever knowing the encryption key.

### Scenario: Cookie Manipulation
Imagine a web application that issues a session cookie to users. The cookie is generated by concatenating user input with an internal role and encrypting it using AES-128-ECB.

Format: `email=user@test.com&uid=10&role=user`

If we provide `user@test.com`, the resulting plaintext is 38 bytes long. It gets split into 16-byte blocks and padded:

```text
Block 1: "email=user@test."
Block 2: "com&uid=10&role="
Block 3: "user\x0c\x0c\x0c\x0c\x0c\x0c\x0c\x0c\x0c\x0c\x0c\x0c" (PKCS#7 padding)
```

The application decrypts this cookie on the server-side, parses the key-value pairs, and grants access based on the `role` parameter. The goal of the attacker is to elevate their privileges by changing `role=user` to `role=admin`.

### Execution Steps
1. **Block Alignment:** The attacker registers an email address designed to align the target data perfectly on block boundaries. We want the string `admin` (padded to a full block) to be encrypted as a standalone block.
   Let's craft an email: `A` * 10 + `admin` + `\x0b` * 11
   Email: `AAAAAAAAAAadmin\x0b\x0b\x0b\x0b\x0b\x0b\x0b\x0b\x0b\x0b\x0b`
   
   The resulting plaintext becomes:
   ```text
   Block 1: "email=AAAAAAAAAA"
   Block 2: "admin\x0b\x0b\x0b\x0b\x0b\x0b\x0b\x0b\x0b\x0b\x0b"  <-- We capture this ciphertext block!
   Block 3: "&uid=10&role=use"
   Block 4: "r\x0f\x0f\x0f\x0f\x0f\x0f\x0f\x0f\x0f\x0f\x0f\x0f\x0f\x0f\x0f"
   ```
   We save `Ciphertext Block 2` locally because we know exactly what it decrypts to (`admin` plus valid padding).

2. **Pushing the target field into its own block:** Now we craft a new email to push the word `user` (at the end of `role=`) into the start of a new block.
   The prefix before our email is `email=`. Length = 6.
   We want the string before the role to be a multiple of 16.
   Prefix: `email=` (6) + `email_input` + `&uid=10&role=` (13)
   Total before role = 19 + `len(email_input)`.
   If we want this to be 32 (2 blocks), `len(email_input)` must be 13. Let's use `user@test.com`.
   19 + 13 = 32.

   Plaintext layout for this new email:
   ```text
   Block 1: "email=user@test."
   Block 2: "com&uid=10&role="
   Block 3: "user\x0c\x0c\x0c\x0c\x0c\x0c\x0c\x0c\x0c\x0c\x0c\x0c"
   ```

3. **The Paste:** We take the ciphertext generated from step 2, but we replace `Block 3` with the `Ciphertext Block 2` we captured in step 1.

   The manipulated ciphertext decrypts to:
   ```text
   Block 1: "email=user@test."
   Block 2: "com&uid=10&role="
   Block 3: "admin\x0b\x0b\x0b\x0b\x0b\x0b\x0b\x0b\x0b\x0b\x0b"
   ```
   When parsed by the backend, the parameters are `email=user@test.com`, `uid=10`, and `role=admin`. The attacker successfully escalated privileges without knowing the AES key!

## 4. Byte-at-a-Time ECB Decryption

Another devastating attack against ECB mode is the byte-at-a-time decryption attack. This is applicable when the attacker can prepend their own controlled input to an unknown secret string, and the combined string is encrypted using ECB.

### Scenario
An oracle function exists: `AES_128_ECB( attacker_input || unknown_secret_key, encryption_key )`.
The attacker can send arbitrary data as `attacker_input` and receive the resulting ciphertext. The attacker's goal is to extract the `unknown_secret_key` entirely, without ever knowing the `encryption_key`.

### Attack Mechanics

#### Step 1: Discover the Block Size
Feed the oracle with increasing lengths of identical characters (e.g., `A`, `AA`, `AAA`). Monitor the size of the returned ciphertext. When the ciphertext size suddenly jumps (e.g., from 32 bytes to 48 bytes), the difference between the new size and the old size is the block size (usually 16 bytes for AES).

#### Step 2: Discover the Secret Length
The initial ciphertext length with zero attacker input provides an upper bound for the secret length. By adding input characters one by one until the ciphertext size jumps, the exact length of the secret can be mathematically derived.

#### Step 3: Align the Target Byte
If the block size is 16 bytes, we craft an input of 15 bytes of padding (e.g., `AAAAAAAAAAAAAAA`).
The first 16-byte block of the plaintext will consist of our 15 'A's plus the very first byte of the unknown secret.

```text
[ A A A A A A A A A A A A A A A ? ] [ rest of secret ... ]
```
We send this to the oracle and store the first block of the resulting ciphertext. This block represents the encrypted state of our 15 'A's and the target unknown byte. Let's call this the `target_block`.

#### Step 4: Dictionary Attack on the Unknown Byte
Next, we generate a dictionary of all 256 possible bytes. For every possible byte `g` from `0x00` to `0xff`, we craft the following 16-byte string:
`AAAAAAAAAAAAAAA` + `g`

We send each of these 256 strings to the oracle and observe the first block of ciphertext. When the first block of ciphertext matches the `target_block` we stored in Step 3, we know that our guess `g` is equal to the first byte of the secret!

#### Step 5: Iteration
To find the second byte, we reduce our padding to 14 bytes (`AAAAAAAAAAAAAA`).
The first block of plaintext now looks like this:
```text
[ A A A A A A A A A A A A A A S1 ? ]
```
(Where `S1` is the first byte of the secret we just discovered).
We send this 14-byte padding to the oracle, save the first ciphertext block, and build a new dictionary using `AAAAAAAAAAAAAA` + `S1` + `g`.

This process is repeated, decreasing the padding length by one each iteration, until the entire secret is recovered. When the end of the first block is reached, the attacker simply shifts their focus to the second ciphertext block, and so on.

### Example Python Exploit Pseudocode

```python
import string

def ecb_oracle(plaintext):
    # This represents the remote server function
    # Returns AES_ECB(plaintext + secret, key)
    pass

block_size = 16
secret = b""
padding_length = 15

# Loop to find the first block (16 bytes)
for i in range(block_size):
    # 1. Craft the block boundary alignment
    payload = b"A" * padding_length
    target_ciphertext = ecb_oracle(payload)
    target_block = target_ciphertext[0:block_size]
    
    # 2. Brute force the last byte of the block
    for char in range(256):
        guess_byte = bytes([char])
        dictionary_payload = payload + secret + guess_byte
        
        test_ciphertext = ecb_oracle(dictionary_payload)
        test_block = test_ciphertext[0:block_size]
        
        # 3. Match found!
        if test_block == target_block:
            secret += guess_byte
            padding_length -= 1
            print(f"[+] Found byte: {guess_byte} | Secret so far: {secret}")
            break
```

## 5. Defense and Remediation

To mitigate Block Boundary Manipulation and other ECB-related attacks, developers and security architects must adopt the following practices:

1. **Never use ECB mode for sensitive data:** The core remediation is to completely deprecate the use of ECB mode in any context where the plaintext contains patterns or exceeds a single block in length.
2. **Use Authenticated Encryption with Associated Data (AEAD):** Modes like AES-GCM (Galois/Counter Mode) or ChaCha20-Poly1305 provide both confidentiality and integrity. They ensure that ciphertext cannot be tampered with (preventing Cut-and-Paste attacks entirely).
3. **Use securely generated IVs:** If an older mode like CBC must be used, ensure that a Cryptographically Secure Pseudo-Random Number Generator (CSPRNG) is used to generate a unique, unpredictable Initialization Vector (IV) for every encryption operation.

---

## 6. Chaining Opportunities

- **Authentication Bypass:** By manipulating encrypted session cookies or JWTs that mistakenly utilize ECB mode, an attacker can forge their identity or role, escalating privileges.
- **Data Exfiltration:** Exploiting a byte-at-a-time decryption oracle allows attackers to extract sensitive backend tokens, database connection strings, or administrative credentials that are appended to user inputs.
- **[[07 - Cryptographic Failures in Web Tokens]]:** If a custom token structure uses ECB, the lack of integrity checks enables forgery, leading to full application compromise.
