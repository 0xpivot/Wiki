# SPI Flash Dumping and Manipulation

## 1. Introduction

In the realm of IoT penetration testing, acquiring the device's firmware is the critical first step for static analysis, vulnerability discovery, and cryptographic key extraction. While firmware can sometimes be intercepted during an over-the-air (OTA) update or downloaded from a vendor's website, attackers often encounter devices where the firmware is heavily guarded, encrypted in transit, or simply unavailable online.

When network-based extraction fails, and software interfaces (like UART or JTAG) are locked down, hardware hackers employ a direct physical attack: **SPI Flash Dumping**. 

Almost all modern embedded systems store their bootloader, kernel, filesystem, and configuration data on a non-volatile memory chip. Serial Peripheral Interface (SPI) flash memory is the undisputed standard for this purpose due to its low cost, small physical footprint, and relatively high speed. This note details the advanced methodology of physically interfacing with SPI flash chips, extracting their contents, and potentially rewriting them.

## 2. Understanding SPI (Serial Peripheral Interface)

SPI is a synchronous serial communication interface specification used for short-distance communication, primarily in embedded systems. It operates in a Master-Slave architecture. The Microcontroller (CPU/SoC) acts as the Master, and the Flash memory chip acts as the Slave.

SPI communication relies on four primary logic signals (pins):
1.  **SCLK (Serial Clock):** Generated by the Master to synchronize data transmission.
2.  **MOSI (Master Out Slave In):** Data sent *from* the CPU *to* the flash chip (e.g., commands like "Read", "Write", or data to be written).
3.  **MISO (Master In Slave Out):** Data sent *from* the flash chip *to* the CPU (e.g., the actual contents of the flash memory).
4.  **CS/SS (Chip Select / Slave Select):** An active-low signal generated by the Master. When pulled low (0V), it wakes up the specific flash chip and tells it to listen to the bus.

Additionally, Flash chips have pins for Power (`VCC`), Ground (`GND`), Write Protect (`WP`), and Hold (`HOLD`).

## 3. The Hardware Tools

To read a flash chip independently of the device's CPU, the attacker's hardware must take over the role of the SPI Master. Common tools include:
*   **CH341A Programmer:** The most common, inexpensive, and widely used USB-to-SPI programmer. Extremely popular in the hardware hacking community.
*   **Bus Pirate:** A versatile multi-protocol tool that can speak SPI, I2C, UART, etc.
*   **Shikra:** A highly reliable, fast FT232H-based tool specifically designed for hardware hacking.
*   **Raspberry Pi:** The GPIO pins on a standard Raspberry Pi can easily be configured to act as an SPI Master, utilizing tools like `flashrom`.

## 4. Extraction Methodology

Dumping SPI flash involves a precise physical and software methodology to avoid corrupting the data or physically destroying the delicate PCB components.

### Step 1: Identification and Datasheet Acquisition

1.  Locate the flash chip on the PCB. They are typically 8-pin (SOIC-8) or 16-pin (SOIC-16) black ICs located near the main SoC. Common manufacturers include Winbond, Macronix (MXIC), Spansion, and GigaDevice.
2.  Read the microscopic part number etched into the plastic casing (e.g., `Winbond W25Q128JV`).
3.  Search the internet for the exact datasheet. The datasheet is the hardware hacker's bible. It provides the pinout, operating voltage (crucial: 3.3V vs. 1.8V), storage capacity (e.g., 128 Megabits = 16 Megabytes), and supported instruction sets.

### Step 2: Physical Interfacing Techniques

You must connect the SPI pins of your programmer to the pins on the flash chip. There are three primary methods, ranging from least to most invasive:

*   **In-Circuit (Test Clips):** Using an SOIC-8 or SOIC-16 Pomona test clip, you clamp directly onto the chip while it is still soldered to the board. 
    *   *Pros:* Fast, non-destructive.
    *   *Cons:* **Power Contention.** When you apply power (VCC) from your programmer to the flash chip, that power travels through the PCB traces and powers up the SoC and other components. The SoC will boot up and try to talk to the flash chip at the same time as your programmer, causing collisions on the MISO/MOSI lines and resulting in corrupt dumps. To bypass this, you must hold the SoC's reset pin low (if accessible) to keep it dormant.
*   **In-Circuit (Probes/Pogo Pins):** Using microscopic spring-loaded pins held in place by a mechanical arm to touch the chip's legs.
*   **Desoldering (Off-Board):** The most reliable method. Using a hot air rework station, carefully heat the solder pads and remove the chip from the PCB. Place the chip into a specialized ZIF (Zero Insertion Force) socket connected to your programmer.
    *   *Pros:* Guaranteed 100% clean read, no interference from the mainboard.
    *   *Cons:* Requires soldering skills; risk of lifting PCB pads or destroying the chip if overheated.

### Step 3: Software Extraction (Flashrom)

Once physically connected, the software of choice in the Linux environment is `flashrom`.

1.  **Probe the Chip:** Verify the connection and ensure `flashrom` can read the chip's internal ID.
    ```bash
    $ flashrom -p ch341a_spi
    flashrom v1.2 on Linux 5.15.0 (x86_64)
    flashrom is free software, get the source code at https://flashrom.org

    Calibrating delay loop... OK.
    Found Winbond flash chip "W25Q128.V" (16384 kB, SPI) on ch341a_spi.
    ```
2.  **Read the Contents:** Execute the read command and output it to a binary file.
    ```bash
    $ flashrom -p ch341a_spi -r firmware_dump_1.bin
    Reading flash... done.
    ```
3.  **Verification (The Golden Rule of Hardware Hacking):** *Never trust a single dump.* Always read the chip at least three times into separate files.
    ```bash
    $ flashrom -p ch341a_spi -r firmware_dump_2.bin
    $ flashrom -p ch341a_spi -r firmware_dump_3.bin
    ```
4.  Compare the hashes of the dumps. If they match, your connection is stable, and the dump is valid. If they differ, you have a loose wire, noise on the line, or power contention.
    ```bash
    $ md5sum firmware_dump*.bin
    2a1b3c4d5e6f7a8b9c0d1e2f3a4b5c6d  firmware_dump_1.bin
    2a1b3c4d5e6f7a8b9c0d1e2f3a4b5c6d  firmware_dump_2.bin
    2a1b3c4d5e6f7a8b9c0d1e2f3a4b5c6d  firmware_dump_3.bin
    ```

## 6. Post-Extraction Analysis and Modification

Once `firmware.bin` is acquired, it moves to the software analysis phase.
*   **Binwalk:** `binwalk -e firmware.bin` is used to identify and extract filesystems (SquashFS, JFFS2), compressed kernels (LZMA), and bootloader environments.
*   **Secret Extraction:** Attackers parse the filesystem looking for hardcoded credentials, API keys for cloud backend services, private RSA keys for SSH or TLS, and proprietary vendor algorithms.
*   **Firmware Modification (The "Write" Phase):**
    If the device lacks Secure Boot (which verifies cryptographic signatures of the firmware on every boot), an attacker can extract the filesystem, plant a backdoor (e.g., a reverse shell in `/etc/init.d/`), repack the filesystem, recreate the monolithic binary image, and use `flashrom -w modified_firmware.bin` to write it back to the chip. When the chip is soldered back onto the board and powered on, the device executes the attacker's backdoored operating system.

## 7. A Note on Voltage (The 3.3V vs 1.8V Danger)

The vast majority of SPI flash chips operate at 3.3 Volts. However, modern, low-power IoT devices increasingly use 1.8V flash chips. 
**CRITICAL WARNING:** Connecting a standard CH341A (which supplies 3.3V or sometimes 5V on its data lines) directly to a 1.8V chip will permanently fry the flash chip, destroying the device and the firmware. Always measure the operating voltage of the chip using a multimeter while the device is powered normally *before* attaching any programmer, or meticulously verify the datasheet. If it is a 1.8V chip, a logic level converter or a specialized 1.8V adapter is mandatory.
