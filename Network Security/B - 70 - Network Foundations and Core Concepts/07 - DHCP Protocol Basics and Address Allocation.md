# DHCP Protocol Basics and Address Allocation

## 1. Overview of DHCP
The Dynamic Host Configuration Protocol (DHCP) is a client/server protocol that automatically provides an Internet Protocol (IP) host with its IP address and other related configuration information such as the subnet mask and default gateway. Operating at the Application Layer (using UDP transport), DHCP evolved from the older BOOTP (Bootstrap Protocol) and drastically simplifies network administration by eliminating the need to manually configure IP settings on every device.

DHCP uses a leased-address mechanism. Instead of owning an IP address permanently, a client leases it from the server for a specific duration. Before the lease expires, the client must renew it, or the server will reclaim the address and place it back into the available pool.

## 2. The DORA Process (Detailed Breakdown)
The acquisition of an IP address via DHCP involves a four-step transaction known commonly by the acronym **DORA**. This process relies heavily on broadcast traffic since the client initially lacks a routable IP address.

*   **D - DISCOVER (DHCPDiscover):**
    *   **Source IP:** `0.0.0.0` (Client has no IP).
    *   **Destination IP:** `255.255.255.255` (Local network broadcast).
    *   **Source MAC:** Client's MAC address.
    *   **Destination MAC:** `FF:FF:FF:FF:FF:FF` (Broadcast).
    *   **Action:** The client broadcasts a message over UDP port 67 (to the server) from UDP port 68 (client) asking, "Are there any DHCP servers out there?"

*   **O - OFFER (DHCPOffer):**
    *   **Source IP:** DHCP Server's IP.
    *   **Destination IP:** `255.255.255.255` (Broadcast) or unicast to client MAC depending on the `Broadcast Flag`.
    *   **Action:** Any listening DHCP server responds with an offer. This message contains a proposed IP address (yiaddr), subnet mask, default gateway, lease duration, and the IP of the server making the offer.

*   **R - REQUEST (DHCPRequest):**
    *   **Source IP:** `0.0.0.0` (Client still doesn't officially own the IP).
    *   **Destination IP:** `255.255.255.255` (Broadcast).
    *   **Action:** The client selects one of the received offers (usually the first one). It broadcasts a formal request for that specific IP address. It is broadcasted to inform *all* other DHCP servers that it has accepted an offer, allowing them to return their proposed IPs to their respective pools. The packet includes the Server Identifier option of the chosen server.

*   **A - ACKNOWLEDGE (DHCPAck):**
    *   **Source IP:** DHCP Server's IP.
    *   **Destination IP:** Broadcast or Unicast.
    *   **Action:** The chosen server confirms the lease. It sends an acknowledgment containing the final configuration parameters. Once the client receives this, it applies the IP address to its interface.
    *   *(Note: The client usually performs a Gratuitous ARP to ensure no other device on the network is currently using the assigned IP address before fully committing to it).*

## 3. DHCP Packet Structure and Fields
DHCP messages are encapsulated in UDP datagrams. Key fields in the DHCP payload include:
*   **Opcode (op):** 1 for Request (Client to Server), 2 for Reply (Server to Client).
*   **Hardware Type (htype) & Length (hlen):** Usually 1 (Ethernet) and 6 (MAC address length).
*   **Hops:** Used by relay agents. Set to 0 by the client.
*   **Transaction ID (xid):** A random 32-bit number generated by the client to correlate messages and responses.
*   **Seconds Elapsed (secs):** Seconds since the client began the address acquisition process.
*   **Flags:** Contains the Broadcast bit. If set to 1, the client cannot receive unicast packets until configured, forcing the server to broadcast the Offer/Ack.
*   **Client IP Address (ciaddr):** Only filled during lease renewal, when the client already has a valid IP.
*   **Your IP Address (yiaddr):** The IP address offered by the server to the client.
*   **Server IP Address (siaddr):** Address of the next server to use in bootstrap (often a TFTP server for PXE boot).
*   **Gateway IP Address (giaddr):** The IP address of the DHCP Relay Agent, if used.
*   **Client Hardware Address (chaddr):** The client's MAC address.
*   **Magic Cookie:** The first four bytes of the options field, always set to `0x63825363`, distinguishing DHCP from BOOTP.

## 4. Critical DHCP Options
The Options field carries the actual configuration payload, organized as TLV (Type-Length-Value) structures.
*   **Option 1 (Subnet Mask):** Specifies the client's subnet mask.
*   **Option 3 (Router):** Provides the default gateway(s) for the client.
*   **Option 6 (Domain Name Server):** Provides a list of DNS servers.
*   **Option 15 (Domain Name):** The DNS suffix for the client.
*   **Option 53 (DHCP Message Type):** Specifies whether the packet is a Discover, Offer, Request, Ack, Nak, Release, or Inform.
*   **Option 82 (Relay Agent Information):** Inserted by DHCP Relay Agents (switches/routers) to provide the server with topological information about where the client is attached (e.g., switch port number, VLAN). This allows the server to assign IPs based on physical location and mitigates some spoofing attacks.

## 5. DHCP Relay Agents (IP Helper)
Because DHCP relies on broadcasts, a DHCP server must exist on the same subnet as the clients, as routers drop broadcast packets by default. To scale enterprise networks without placing a DHCP server on every VLAN, **DHCP Relay Agents** are used.

Configured on the router's interface (e.g., `ip helper-address` in Cisco IOS), the relay agent intercepts the client's local broadcast (`DHCPDiscover`). It then encapsulates this request into a unicast UDP packet and forwards it to the centralized DHCP server. It fills the `giaddr` field with its own interface IP address, allowing the DHCP server to know which subnet pool the client belongs to. The server responds unicast back to the relay agent, which then broadcasts the `DHCPOffer` onto the local subnet for the client.

## 6. DHCP in IPv6 (SLAAC and DHCPv6)
IPv6 introduces alternative mechanisms for address allocation.
*   **SLAAC (Stateless Address Autoconfiguration):** Devices generate their own IPv6 addresses using the network prefix advertised by local routers (via ICMPv6 Router Advertisements) and their MAC address (EUI-64) or random generation. SLAAC does not provide DNS info by default.
*   **DHCPv6:** A separate protocol from DHCPv4, using UDP ports 546 (client) and 547 (server). It uses a four-step process: **Solicit, Advertise, Request, Reply (SARR)**. It can operate in Stateful mode (providing both IP and options) or Stateless mode (providing only options like DNS, while SLAAC provides the IP).

## 7. DHCP Security Vulnerabilities
Because it was designed for trusted networks, native DHCP has no authentication mechanisms.
*   **Rogue DHCP Server:** An attacker connects a machine to the network running DHCP server software. When clients broadcast a `DHCPDiscover`, the attacker replies with a `DHCPOffer` faster than the legitimate server. The rogue offer provides a malicious default gateway and malicious DNS server, successfully establishing a Man-in-the-Middle (MITM) position.
*   **DHCP Starvation:** An attacker runs a script (e.g., Yersinia) that floods the network with thousands of `DHCPDiscover` messages, each using a randomly generated spoofed source MAC address. The legitimate DHCP server quickly exhausts its entire pool of available IP addresses. This creates a Denial of Service (DoS) for legitimate clients and perfectly sets the stage for a Rogue DHCP attack.
*   **Option Injection (WPAD Spoofing):** An attacker uses DHCP Option 252 to supply a malicious Proxy Auto-Configuration (PAC) file via the Web Proxy Auto-Discovery Protocol (WPAD), intercepting the victim's web browsing traffic.

## 8. Mitigations and Defensive Measures
*   **DHCP Snooping:** A Layer 2 switch security feature. The administrator configures specific switch ports (pointing to the real DHCP server) as "Trusted," and all edge ports connected to end-users as "Untrusted." The switch inspects all DHCP packets. It immediately drops any `DHCPOffer` or `DHCPAck` packets originating from an "Untrusted" port, entirely neutralizing Rogue DHCP servers.
*   **Port Security:** Limits the number of MAC addresses allowed on a single switch port. This stops DHCP Starvation dead in its tracks, as the attacker cannot spoof thousands of MAC addresses without the switch port shutting down in an err-disable state.
*   **Dynamic ARP Inspection (DAI):** Works hand-in-hand with DHCP Snooping. The switch builds a binding database mapping legitimate IP addresses to MAC addresses based on successful DHCP leases. It then blocks any ARP replies that do not match the database, preventing ARP spoofing.

## 10. Chaining Opportunities
*   **DHCP Starvation to Rogue DHCP to Responder/NTLM Relay:** By starving the real DHCP server, the attacker forces new clients to accept leases from the Rogue DHCP. The attacker sets the DNS server to their own machine, pointing all WPAD queries or file share queries to their rogue SMB/HTTP server running Responder, capturing NTLMv2 hashes or relaying them for remote code execution.
*   **WPAD Injection via Option 252:** Configuring the Rogue DHCP server to push an automatic proxy configuration, allowing the interception and modification of plain-text HTTP traffic or bypassing network access controls.
