# MSMQ — Unauthenticated Access

## 1. Executive Summary
Microsoft Message Queuing (MSMQ) is a legacy, asynchronous messaging protocol developed by Microsoft. It is designed to ensure reliable message delivery between distributed applications, even over unreliable networks. By temporarily storing messages in queues when a destination is unreachable, MSMQ guarantees delivery once connectivity is restored.

While largely superseded by modern message brokers like RabbitMQ or Apache Kafka, MSMQ remains deeply embedded in older enterprise environments, Microsoft Exchange server deployments, and specialized Windows services (such as banking and healthcare applications). The security model of MSMQ relies heavily on Windows Access Control Lists (ACLs). However, widespread misconfigurations frequently expose queues to unauthenticated access. Furthermore, critical vulnerabilities in the MSMQ parsing engine (like CVE-2023-21554, "QueueJumper") allow for unauthenticated Remote Code Execution (RCE) at the `SYSTEM` level, making MSMQ a high-value target during internal network penetration tests.

## 2. Technical Architecture: MSMQ Protocol
MSMQ operates using a combination of direct TCP connections and Microsoft Remote Procedure Call (MS-RPC).
- **TCP Port 1801:** The primary port used for direct MSMQ message transfer.
- **TCP Port 135 (RPC Endpoint Mapper):** Used to map dynamic high ports for queue management, remote reading, and Active Directory integration.
- **Older Ports:** UDP 3527 (Ping), TCP 2101, 2103, 2105 (Older MSMQ RPC services).

### 2.1 Queue Types
- **Public Queues:** Published in Active Directory. They can be located by any domain member.
- **Private Queues:** Not published in AD. They reside locally on the server (e.g., `.\private$\orders_queue`) and must be accessed via a direct format name (e.g., `FormatName:Direct=TCP:192.168.1.100\private$\orders_queue`).

## 4. Attack Vectors and Misconfigurations
### 4.1 Unauthenticated Queue Reading (Data Exfiltration)
If a queue's security ACL is misconfigured to allow "Everyone" or "ANONYMOUS LOGON" the "Receive Message" permission, an attacker can read messages intended for the backend application. Because MSMQ is often used to decouple authentication systems or financial transactions, these messages frequently contain plain-text credentials, PII, or authorization tokens.

### 4.2 Unauthenticated Queue Writing (Queue Poisoning)
If "Everyone" has "Send Message" permissions, an attacker can push arbitrary messages into the queue. If the consuming application inherently trusts the data in the queue, this leads to secondary injection attacks (e.g., placing a malicious SQL payload in the queue that gets executed when the backend database processes it).

### 4.3 Insecure Deserialization via MSMQ
MSMQ is purely a transport protocol; it does not dictate the payload format. Many .NET applications serialize complex objects, place them in MSMQ, and the receiver deserializes them using vulnerable formatters like `BinaryFormatter` or `XmlSerializer`. An attacker with write access to the queue can inject malicious serialized payloads generated by tools like `ysoserial.net`, resulting in arbitrary code execution when the message is processed.

### 4.4 QueueJumper (CVE-2023-21554)
Discovered in 2023, QueueJumper is a critical (CVSS 9.8) vulnerability in the MSMQ service (`mqsvc.exe`). An attacker can send a specially crafted MSMQ packet to TCP 1801 containing an oversized property field. This triggers an out-of-bounds write (heap buffer overflow), allowing unauthenticated attackers to execute arbitrary code as `NT AUTHORITY\SYSTEM`.

## 5. Enumeration Methodology
### 5.1 Nmap Port Scanning
Identify the presence of MSMQ by scanning port 1801 and RPC endpoints.
```bash
# Scan specific MSMQ ports
nmap -p 135,1801,2101-2105,3527 -sV <target-ip>

# Use the specific NSE script for MSMQ enumeration
nmap -p 1801 --script msmq-info <target-ip>
```

### 5.2 RPC Endpoint Mapping
Because MSMQ integrates tightly with RPC, `rpcdump.py` from Impacket can identify the specific MSMQ interfaces bound to dynamic ports.
```bash
rpcdump.py <target-ip> | grep -i msmq
```
Look for `qmcomm` (Queue Manager Communication) and `mqac` (Message Queuing Access Control).

### 5.3 Local Enumeration (Post-Foothold)
If you have a low-privileged shell on a Windows machine, you can enumerate local MSMQ queues using PowerShell:
```powershell
Get-WmiObject -Class Win32_PerfRawData_MSMQ_MSMQQueue
# Or via native cmdlets if the MSMQ module is loaded:
Get-MsmqQueue
```

## 6. Exploitation Techniques
### 6.1 Interacting with Queues via PowerShell
If network ACLs allow it, you can interact with a remote MSMQ queue directly from a compromised Windows host using native .NET classes.
**Reading from a Queue:**
```powershell
[Reflection.Assembly]::LoadWithPartialName("System.Messaging")
$queuePath = "FormatName:Direct=TCP:192.168.1.100\private$\target_queue"
$queue = New-Object System.Messaging.MessageQueue($queuePath)

# Receive removes the message. Peek just reads it.
$message = $queue.Peek( [TimeSpan]::FromSeconds(5) )
$reader = New-Object System.IO.StreamReader($message.BodyStream)
$reader.ReadToEnd()
```

**Writing a Malicious Payload to a Queue:**
```powershell
$msg = New-Object System.Messaging.Message
$msg.Body = "Malicious Payload or Serialized Object"
$queue.Send($msg)
```

### 6.2 Exploiting CVE-2023-21554
To exploit QueueJumper, attackers use Python scripts that construct the raw MSMQ packet. The exploit involves manipulating the SRMP (SOAP Reliable Messaging Protocol) headers embedded within the MSMQ packet to overflow a buffer during property processing. 
*Note: Public exploitation often results in immediate SYSTEM-level reverse shells, but requires careful alignment of heap chunks depending on the specific Windows Server build.*

## 7. Post-Exploitation & Lateral Movement
- **Complete Domain Compromise:** Because MSMQ often runs on critical infrastructure servers (like Exchange or core application servers) as SYSTEM, compromising the service yields credentials that can be used to pass-the-hash or dump LSASS, often leading directly to Domain Admin.
- **Service Disruption:** By constantly reading and dropping messages from a queue, an attacker can create a silent Denial of Service (DoS), breaking business logic without crashing the application.
- **NTLM Relaying:** Attackers can force the MSMQ service to authenticate to an attacker-controlled machine by sending specific RPC requests. The resulting NTLMv2 hash can be captured via Responder or relayed to another server (e.g., LDAP or SMB).

## 8. Defensive Evasion
Attackers often utilize the MSMQ protocol itself as a covert channel for C2 communications (e.g., using tools like `Malandro`). Because MSMQ traffic is expected between application servers, writing C2 instructions into a hidden private queue and having a local agent read from it bypasses traditional network egress monitoring and firewalls.

## 9. Incident Response & Detection
### 9.1 Network Traffic Analysis
- **Wireshark Filter:** `msmq`
- Look for anomalous traffic on TCP 1801, particularly large packets or excessive connection attempts from non-application servers.
- For CVE-2023-21554, IDS/IPS rules (like Snort/Suricata) can detect the oversized SRMP property structures in the raw TCP stream.

### 9.2 Endpoint Monitoring
Monitor Windows Event Logs for unexpected MSMQ service state changes or anomalous PowerShell activity loading the `System.Messaging` assembly.
- Event ID 7036: Service State change (Look for "Message Queuing" starting unexpectedly).
- Monitor for crashing `mqsvc.exe` processes (Event ID 1000 in Application logs), which strongly indicates failed exploitation attempts of QueueJumper.

## 10. Remediation & Hardening Guide
- **Patch Management:** Immediately apply Microsoft security updates, specifically addressing CVE-2023-21554 and related MSMQ memory corruption flaws.
- **Uninstall MSMQ:** The most effective mitigation is to completely remove the MSMQ feature from Windows Server via Server Manager if it is not strictly required by business applications.
- **Network Segmentation:** Block inbound access to TCP 1801, TCP 135, and high RPC ports from user subnets and the internet. MSMQ traffic should only flow between dedicated, trusted application servers.
- **Strict Queue Access Controls:** Enforce Principle of Least Privilege on all MSMQ queues. Remove "Everyone" and "ANONYMOUS LOGON" permissions. Only explicitly authorized service accounts should have "Send Message" or "Receive Message" rights.
- **Implement Message Authentication:** Require MSMQ messages to be digitally signed and authenticated to prevent queue poisoning and MitM tampering.
