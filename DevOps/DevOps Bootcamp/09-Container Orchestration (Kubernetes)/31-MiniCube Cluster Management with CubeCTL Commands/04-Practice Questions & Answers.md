## Practice Questions & Answers

**Q1. Explain the purpose of the `CubeCTL` command in managing a MiniCube cluster.**

CubeCTL is a command-line tool used to interact with the MiniCube cluster. It allows users to perform various operations such as creating, updating, and deleting Kubernetes components, checking the status of different resources, and debugging applications running within the cluster. Essentially, CubeCTL serves as the primary interface for managing the cluster and its resources.

**Q2. How do you create a deployment in MiniCube using CubeCTL? Provide an example command.**

To create a deployment in MiniCube using CubeCTL, you use the `create deployment` command followed by the necessary parameters. An example command to create a deployment named `EngineX-deployment` using the `nginx` image is:

```bash
CubeCTL create deployment EngineX-deployment --image=nginx
```

This command creates a deployment named `EngineX-deployment` and deploys the `nginx` image as a pod under this deployment.

**Q3. What is the role of a ReplicaSet in Kubernetes, and how is it related to a deployment?**

A ReplicaSet ensures that a specified number of pod replicas are running at any given time. It is responsible for maintaining the desired state of the pods. When a deployment is created, Kubernetes automatically generates a ReplicaSet to manage the pods. The ReplicaSet ensures that the number of replicas matches the desired state defined in the deployment. If a pod fails, the ReplicaSet will start a new one to maintain the desired number of replicas.

**Q4. How can you view the logs of a running container in a pod using CubeCTL? Provide an example command.**

To view the logs of a running container in a pod using CubeCTL, you use the `logs` command followed by the pod name. An example command to view the logs of a pod named `mongo-pod` is:

```bash
CubeCTL logs mongo-pod
```

This command retrieves and displays the logs generated by the application running inside the `mongo-pod`.

**Q5. How do you delete a deployment and its associated pods and ReplicaSets using CubeCTL? Provide an example command.**

To delete a deployment and its associated pods and ReplicaSets using CubeCTL, you use the `delete deployment` command followed by the deployment name. An example command to delete a deployment named `EngineX-deployment` is:

```bash
CubeCTL delete deployment EngineX-deployment
```

This command deletes the deployment, which in turn removes the associated pods and ReplicaSets.

**Q6. Explain how to use a Kubernetes configuration file to create and update a deployment using CubeCTL. Provide an example configuration file and the corresponding command.**

To use a Kubernetes configuration file to create and update a deployment, you use the `apply` command with the `-f` flag to specify the configuration file. An example configuration file (`engine-x-deployment.yaml`) might look like this:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: engine-x-deployment
spec:
  replicas: 2
  selector:
    matchLabels:
      app: engine-x
  template:
    metadata:
      labels:
        app: engine-x
    spec:
      containers:
      - name: engine-x
        image: nginx
        ports:
        - containerPort: 80
```

To apply this configuration file, you would use the following command:

```bash
CubeCTL apply -f engine-x-deployment.yaml
```

This command creates the deployment if it does not exist, or updates it if it already exists, according to the specifications in the configuration file.

**Q7. What is the purpose of the `describe` command in CubeCTL, and how can it be used to troubleshoot issues with a pod?**

The `describe` command in CubeCTL provides detailed information about a specific resource, including events and status conditions. It is particularly useful for troubleshooting issues with a pod. To use it, you specify the type of resource and the name of the resource. For example, to describe a pod named `mongo-pod`, you would use:

```bash
CubeCTL describe pod mongo-pod
```

This command outputs detailed information about the pod, including its status, events, and any errors that may have occurred during its creation or operation. This information can help diagnose and resolve issues with the pod.
