## Introduction to JWT Attacks

In this section, we will delve into the intricacies of JSON Web Tokens (JWT) and explore a specific type of attack: JWT authentication bypass via flawed signature verification. This attack exploits vulnerabilities in the way servers handle JWTs, particularly when they fail to properly verify the signatures of these tokens.

### Background Theory

JSON Web Tokens (JWTs) are a compact, URL-safe means of representing claims to be transferred between two parties. They are typically used to transmit information between parties as an alternative to cookies. A JWT consists of three parts separated by dots (`.`):

1. **Header**: Contains metadata about the token, such as the type of token and the signing algorithm used.
2. **Payload**: Contains the claims, which are statements about an entity (typically the user) and additional data.
3. **Signature**: Ensures the integrity of the token and is generated by applying a cryptographic hash function to the encoded header and payload, along with a secret key.

The general structure of a JWT looks like this:

```
<base64UrlEncode(header)>.<base64UrlEncode(payload)>.<signature>
```

### JWT Structure and Encoding

Let's break down the components of a JWT:

#### Header

The header typically contains two fields:
- `typ`: The type of the token, usually `"JWT"`.
- `alg`: The algorithm used to sign the token, such as `"HS256"` (HMAC SHA-256).

Example header:

```json
{
  "typ": "JWT",
  "alg": "HS256"
}
```

This header is base64url encoded:

```plaintext
eyJ0eXAiOiJKV1QiLCJhbGciOiJIUzI1NiJ9
```

#### Payload

The payload contains the claims. Claims can be registered (standardized) or custom. Common registered claims include:
- `iss`: Issuer of the token.
- `sub`: Subject of the token (usually the user ID).
- `aud`: Audience for whom the token is intended.
- `exp`: Expiration time of the token.
- `nbf`: Not before time (token cannot be used before this time).
- `iat`: Issued at time (when the token was issued).

Example payload:

```json
{
  "sub": "1234567890",
  "name": "John Doe",
  "admin": true,
  "iat": 1516239022
}
```

This payload is base64url encoded:

```plaintext
eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIiwiYWRtaW4iOnRydWUsImlhdCI6MTUxNjIzOTAyMn0
```

#### Signature

The signature is created by taking the encoded header and payload, concatenating them with a dot (`.`), and then applying the specified algorithm (e.g., HMAC SHA-256) with a secret key.

```plaintext
HMACSHA256(
  base64UrlEncode(header) + "." +
  base64UrlEncode(payload),
  secret)
```

### Flawed Signature Verification

In the context of our lab, the server is insecurely configured to accept unsigned JWTs. This means that the server does not properly verify the signature of the JWTs it receives. As a result, an attacker can modify the payload of a JWT and still have it accepted by the server.

### Real-World Examples

One notable real-world example of a JWT-related vulnerability is the case of the `auth0` library, which had a vulnerability (CVE-2018-16810) where it failed to validate the `kid` (key ID) field in the JWT header. This allowed attackers to craft malicious JWTs that could bypass authentication checks.

Another example is the `firebase-admin` SDK, which had a vulnerability (CVE-2020-0101) where it failed to properly validate the `aud` (audience) claim in the JWT. This allowed attackers to impersonate users and gain unauthorized access.

### How to Prevent / Defend

#### Detection

To detect JWT-related vulnerabilities, you can use the following methods:
- **Static Analysis**: Analyze the code to ensure proper validation of JWTs.
- **Dynamic Analysis**: Use tools like Burp Suite to intercept and modify JWTs during runtime.

#### Prevention

To prevent JWT-related vulnerabilities, follow these best practices:
- **Verify the Signature**: Always verify the signature of the JWT to ensure its integrity.
- **Use Strong Algorithms**: Use strong cryptographic algorithms (e.g., HS256, RS256) for signing JWTs.
- **Validate Claims**: Validate all claims in the JWT to ensure they meet the expected criteria.
- **Secure Secret Keys**: Store secret keys securely and rotate them regularly.

#### Secure Coding Fixes

Here is an example of a vulnerable and a secure implementation of JWT validation in Python:

**Vulnerable Code**

```python
import jwt

def authenticate(token):
    try:
        decoded = jwt.decode(token, options={"verify_signature": False})
        return decoded
    except jwt.exceptions.DecodeError:
        return None
```

**Secure Code**

```python
import jwt

def authenticate(token):
    try:
        decoded = jwt.decode(token, SECRET_KEY, algorithms=["HS256"])
        return decoded
    except jwt.exceptions.DecodeError:
        return None
```
