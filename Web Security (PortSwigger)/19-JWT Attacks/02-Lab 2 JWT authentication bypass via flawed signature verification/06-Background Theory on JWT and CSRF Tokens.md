## Background Theory on JWT and CSRF Tokens

### What is JWT?
JSON Web Token (JWT) is an open standard (RFC 7519) for securely transmitting information between parties as a JSON object. This information can be verified and trusted because it is digitally signed. JWTs can be signed using a secret (with the HMAC algorithm) or a public/private key pair using RSA or ECDSA.

### Why Use JWT?
JWTs are used primarily for authentication and information exchange. They are compact and can be easily transmitted through HTTP requests and responses. They are also useful for implementing Single Sign-On (SSO) across multiple domains.

### How Does JWT Work?
A JWT consists of three parts separated by dots (`.`):

1. **Header**: Contains the type of token and the signing algorithm being used.
2. **Payload**: Contains the claims. Claims are statements about an entity (typically, the user) and additional data.
3. **Signature**: Used to verify the integrity of the message. It is created by hashing the encoded header and payload along with a secret.

### What is a CSRF Token?
Cross-Site Request Forgery (CSRF) tokens are used to protect against CSRF attacks. A CSRF token is a unique identifier generated by the server and sent to the client. The client must include this token in subsequent requests to prove that the request originated from the same user session.

### Why Use CSRF Tokens?
CSRF tokens ensure that a request comes from the intended user and not from a malicious third party. Without CSRF tokens, an attacker could trick a user into performing actions they did not intend to perform.

### How Does a CSRF Token Work?
When a user logs in, the server generates a unique CSRF token and sends it to the client. The client includes this token in subsequent requests. The server verifies the token to ensure the request is legitimate.

### Example of a CSRF Token in Action
Consider a login form:

```html
<form action="/login" method="POST">
  <input type="hidden" name="csrf_token" value="abc123">
  <input type="text" name="username">
  <input type="password" name="password">
  <button type="submit">Login</button>
</form>
```

The server generates the `csrf_token` and includes it in the form. When the form is submitted, the server checks the `csrf_token` to ensure the request is valid.

### Real-World Example: CVE-2021-21972
CVE-2021-21972 is a vulnerability in the Jenkins Continuous Integration server that allows attackers to bypass CSRF protection. This vulnerability highlights the importance of properly implementing and validating CSRF tokens.

### How to Prevent / Defend Against CSRF Attacks
- **Generate Unique Tokens**: Ensure each CSRF token is unique and tied to a specific user session.
- **Validate Tokens**: Always validate the CSRF token on the server side.
- **Use Secure Cookies**: Store CSRF tokens in secure cookies to prevent them from being accessed by JavaScript.
- **Regenerate Tokens**: Regenerate CSRF tokens after successful authentication to prevent reuse.

### Code Example: Generating and Validating CSRF Tokens
```python
import secrets
from flask import Flask, request, session

app = Flask(__name__)
app.secret_key = 'your_secret_key'

@app.route('/login', methods=['GET'])
def login_form():
    csrf_token = secrets.token_hex(16)
    session['csrf_token'] = csrf_token
    return f'<form action="/login" method="POST"><input type="hidden" name="csrf_token" value="{csrf_token}"><input type="text" name="username"><input type="password" name="password"><button type="submit">Login</button></form>'

@app.route('/login', methods=['POST'])
def login():
    if request.form['csrf_token'] != session.get('csrf_token'):
        return 'Invalid CSRF token', 400
    # Perform login logic here
    return 'Logged in successfully'
```

### Explanation of the Code
- **Generating CSRF Token**: The `secrets.token_hex(16)` function generates a random 16-byte hexadecimal string.
- **Storing CSRF Token**: The CSRF token is stored in the session.
- **Validating CSRF Token**: The server compares the CSRF token in the request with the one stored in the session.

### Pitfalls and Common Mistakes
- **Not Validating Tokens**: Failing to validate the CSRF token on the server side.
- **Using Predictable Tokens**: Using predictable or weak tokens that can be guessed.
- **Exposing Tokens**: Exposing CSRF tokens in URLs or other places where they can be intercepted.

### Detection and Prevention
- **Automated Tools**: Use tools like Burp Suite or OWASP ZAP to detect CSRF vulnerabilities.
- **Code Reviews**: Regularly review code to ensure proper implementation of CSRF protection.
- **Security Headers**: Use security headers like `Content-Security-Policy` to mitigate CSRF attacks.
