## Practice Questions & Answers

**Q1. Explain what clickjacking is and why it is considered a significant security threat.**

Clickjacking, also known as UI redress attack or user interface redressing, is a malicious technique in which an attacker tricks a user into clicking on a button or link that appears to perform a different action. This is typically achieved by overlaying transparent or opaque layers on top of the legitimate user interface. The threat is significant because it can lead to unauthorized actions such as deleting accounts, making financial transactions, or granting permissions without the user's knowledge. This can result in data loss, financial harm, and compromise of personal information.

**Q2. How does a CSRF token protect against clickjacking attacks?**

A Cross-Site Request Forgery (CSRF) token is a unique value generated by the server and sent to the client. This token is required for any state-changing operation, such as updating a profile or deleting an account. When a user performs an action, the token is included in the request. If an attacker tries to perform a clickjacking attack, they cannot predict or forge the CSRF token, thus preventing the unauthorized action from being executed. This adds an additional layer of security by ensuring that only legitimate requests with valid tokens are processed.

**Q3. Describe how to exploit a clickjacking vulnerability to delete a user's account, considering the presence of CSRF token protection.**

To exploit a clickjacking vulnerability while the site uses CSRF token protection, follow these steps:

1. **Obtain the CSRF Token**: First, log in to the application and capture the CSRF token from the response headers or form fields. This token is necessary to bypass the CSRF protection mechanism.

2. **Create the Exploit Page**: Set up an attacker-controlled server and create an HTML page that includes an `iframe` pointing to the vulnerable application's delete account page. Ensure the CSRF token is embedded in the form within the `iframe`.

3. **Style the Exploit Page**: Use CSS to style the `iframe` to be invisible or to blend seamlessly with the decoy content. For example:
    ```html
    <style>
        iframe {
            position: absolute;
            top: 0;
            left: 0;
            width: 100%;
            height: 100%;
            opacity: 0.001;
            z-index: 1000;
        }
    </style>
    ```

4. **Overlay Deceptive Content**: Add a deceptive button or link on top of the `iframe`. When the user clicks on this button, they inadvertently click on the delete account button within the `iframe`.
    ```html
    <div style="position: absolute; top: 50%; left: 50%; transform: translate(-50%, -50%);">
        <button onclick="document.getElementById('deleteForm').submit();">Click Here</button>
    </div>
    <iframe src="http://example.com/delete-account" id="deleteIframe">
        <form id="deleteForm" method="POST" action="http://example.com/delete-account">
            <input type="hidden" name="csrf_token" value="captured_token_value">
        </form>
    </iframe>
    ```

5. **Deliver the Exploit**: Send the exploit page to the victim via phishing emails, social media, or other means. When the victim clicks on the deceptive button, the delete account action is performed with the valid CSRF token.

**Q4. Why is it important to test the clickjacking exploit specifically on Chrome?**

Testing the clickjacking exploit specifically on Chrome is important because different browsers may handle certain aspects of web content differently, including how they render and interact with `iframe` elements and CSS properties. Chrome is one of the most widely used browsers, and ensuring that the exploit works on Chrome helps to cover a large portion of potential victims. Additionally, Chrome has specific security features and settings that might affect the success of the exploit, such as the `X-Frame-Options` header, which can prevent framing if not properly configured.

**Q5. What recent real-world examples demonstrate the impact of clickjacking attacks?**

One notable real-world example of a clickjacking attack is the Facebook Likejacking incident in 2010. Attackers exploited a vulnerability in Facebook’s Like button, tricking users into liking malicious content without their knowledge. This led to widespread distribution of malware and compromised user accounts. Another example is the LinkedIn clickjacking attack in 2013, where attackers used a similar technique to trick users into joining fake groups, leading to privacy concerns and potential data breaches. These incidents highlight the importance of robust security measures to prevent such attacks.
