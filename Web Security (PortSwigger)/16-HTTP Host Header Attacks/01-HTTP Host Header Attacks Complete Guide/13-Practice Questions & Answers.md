## Practice Questions & Answers

**Q1. What is the purpose of the HTTP host header, and why is it necessary in modern web architectures?**

The HTTP host header is a mandatory request header that specifies the domain name the client wants to access. It is necessary in modern web architectures due to the widespread use of virtual hosting and shared IP addresses. Virtual hosting allows multiple websites to be hosted on a single server, and the host header helps the server determine which website to serve based on the requested domain name. Additionally, in scenarios involving load balancers or reverse proxies, the host header is crucial for directing traffic to the appropriate backend server.

**Q2. Explain how an attacker can exploit a host header vulnerability to perform password reset poisoning.**

In a password reset poisoning attack, an attacker manipulates the host header to inject their own domain into the password reset link generated by the application. Here’s how it works:

1. A user initiates a password reset request.
2. The application constructs the password reset link using the host header value from the request.
3. An attacker intercepts this request and modifies the host header to point to a domain controlled by the attacker.
4. The application generates the password reset link using the attacker’s domain.
5. The user receives the poisoned link and clicks on it, believing it to be from a legitimate source.
6. The attacker’s server captures the reset token and can use it to reset the user’s password, compromising the account.

**Q3. How can you identify if an application is vulnerable to host header injection?**

To identify if an application is vulnerable to host header injection, you can follow these steps:

1. **Supply Arbitrary Values**: Modify the host header to an arbitrary value and observe if the application accepts it and displays the client-side code.
2. **Bypass Validation Logic**: Test various payloads to bypass flawed validation logic, such as:
   - Injecting payloads via the port number.
   - Using subdomains that match the legitimate domain.
   - Sending duplicate host headers.
   - Indenting HTTP headers with spaces.
   - Supplying absolute URLs in the request line.
3. **Inject Host Override Headers**: Use headers like `X-Forwarded-Host` to override the host header and see if the application processes the new value.

**Q4. Describe how an attacker can use the host header to bypass authentication in an application.**

An attacker can bypass authentication by manipulating the host header if the application incorrectly relies on it to determine the user’s perspective (internal vs. external):

1. The application may require authentication for external users but not for internal users.
2. If the application uses the host header to decide whether a user is coming from an internal or external perspective, an attacker can change the host header to mimic an internal request (e.g., setting it to `localhost`).
3. This can allow the attacker to access the application without providing any credentials.

**Q5. What are some best practices to prevent host header attacks?**

To prevent host header attacks, follow these best practices:

1. **Avoid Using Host Header in Server-Side Code**: If possible, avoid extracting the host header value from the request. Instead, use a trusted source like a configuration file or backend code.
2. **Validate Host Header Against Whitelist**: Ensure that the host header is validated against a whitelist of allowed domains. Reject or redirect requests with unrecognized hosts.
3. **Disable Host Override Headers**: Ensure that headers like `X-Forwarded-Host` are not supported by your framework or application to prevent attackers from overriding the host header.
4. **Manually Specify Current Domain**: When using absolute URLs, specify the current domain manually in the configuration file or backend code instead of relying on the host header.
5. **Segregate Internal and Public Content**: Avoid hosting internal-only websites and applications on the same server as public-facing content to limit the impact of host header attacks.

**Q6. How can web cache poisoning be exploited through host header manipulation?**

Web cache poisoning occurs when an attacker manipulates the host header to include malicious content that is then cached and served to other users. Here’s how it works:

1. An attacker modifies the host header to include a malicious payload.
2. The caching system stores this content based on the manipulated host header.
3. When legitimate users request the same resource, the caching system serves the malicious content from the cache.
4. This can result in the delivery of harmful content to unsuspecting users, leading to various security issues such as cross-site scripting (XSS) or malware distribution.

**Q7. Provide an example of a recent real-world vulnerability related to HTTP host header attacks.**

One notable example is the CVE-2020-2569 vulnerability in the Apache Struts framework. This vulnerability allowed attackers to exploit a flaw in the framework’s handling of the `Content-Type` header, which could be manipulated to execute arbitrary code. Although this specific vulnerability involved the `Content-Type` header rather than the `Host` header, it highlights the importance of properly validating and sanitizing all user-supplied input, including headers.

**Q8. How can an attacker use the host header to perform routing-based SSRF (Server-Side Request Forgery)?**

Routing-based SSRF can be performed by manipulating the host header to direct requests to internal servers within an organization:

1. The attacker sends a request to the application with a modified host header pointing to an internal IP address or hostname.
2. If the application’s intermediary system (like a reverse proxy) forwards requests based on the host header without proper validation, the request is directed to the specified internal server.
3. This allows the attacker to perform an IP scan and access internal services, potentially leading to unauthorized data exfiltration or further exploitation of internal vulnerabilities.
