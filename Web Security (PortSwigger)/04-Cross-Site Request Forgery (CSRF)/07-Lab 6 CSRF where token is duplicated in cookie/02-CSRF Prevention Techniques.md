## CSRF Prevention Techniques

To prevent CSRF attacks, web developers and security professionals employ various techniques. One of the most common methods is the use of CSRF tokens.

### CSRF Tokens

A CSRF token is a unique, unpredictable value generated by the server and sent to the client. The client must include this token in subsequent requests to the server. If the token is missing or incorrect, the server rejects the request.

#### Double-Submit CSRF Prevention Technique

The double-submit CSRF prevention technique involves sending the same token both in a custom header and in a cookie. This ensures that even if an attacker can read cookies, they cannot forge the request because they do not have the token in the custom header.

However, this technique can be insecure if implemented incorrectly, as we will see in the next section.

### Lab No. 6: CSRF Where Token is Duplicated in Cookie

In this lab, we will explore a scenario where the double-submit CSRF prevention technique is used, but the token is duplicated in the cookie, making it vulnerable to CSRF attacks.

#### Background Theory

The double-submit CSRF prevention technique works as follows:

1. The server generates a unique token and sends it to the client.
2. The client includes this token in both a custom header and a cookie in subsequent requests.
3. The server verifies that the token in the custom header matches the token in the cookie.

If the tokens match, the request is considered valid. However, if the token is duplicated in the cookie, an attacker can read the cookie and use the token to craft a malicious request.

### Step-by-Step Mechanics

Let's walk through the steps to exploit this vulnerability:

1. **Identify the Vulnerable Parameter**: The vulnerable parameter in this lab is the email change functionality.
2. **Craft the Malicious Request**: The attacker crafts a malicious request that includes the token in both the custom header and the cookie.
3. **Trick the Victim**: The attacker tricks the victim into interacting with the malicious request, often through social engineering techniques.

#### Complete Example

Here is a complete example of how the attack works:

```http
GET /change-email?email=new@example.com HTTP/1.1
Host: vulnerable-app.com
Cookie: csrf_token=abc123
X-CSRF-Token: abc123
```

In this example, the attacker crafts a GET request to change the email address. The `csrf_token` is included in both the `Cookie` header and the `X-CSRF-Token` header.

### Detection and Prevention

To detect and prevent CSRF attacks, web developers should implement the following measures:

1. **Use Secure CSRF Tokens**: Ensure that CSRF tokens are unique, unpredictable, and securely stored.
2. **Validate Tokens**: Verify that the token in the custom header matches the token in the cookie.
3. **Use HTTPOnly Cookies**: Set the `HttpOnly` flag on cookies to prevent them from being accessed by JavaScript.
4. **Implement Content Security Policy (CSP)**: Use CSP to restrict the sources of content that can be loaded by the browser.

### Secure Coding Fixes

Here is an example of how to implement a secure CSRF token mechanism:

#### Vulnerable Code

```python
# Vulnerable code
def generate_csrf_token():
    return "abc123"

def handle_request(request):
    token = request.headers.get("X-CSRF-Token")
    cookie_token = request.cookies.get("csrf_token")
    if token == cookie_token:
        # Process request
        pass
```

#### Secure Code

```python
# Secure code
import secrets

def generate_csrf_token():
    return secrets.token_hex(16)

def handle_request(request):
    token = request.headers.get("X-CSRF-Token")
    cookie_token = request.cookies.get("csrf_token")
    if token == cookie_token:
        # Process request
        pass
```

In the secure code, the `secrets` module is used to generate a unique and unpredictable CSRF token.

### Configuration Hardening

To further harden the configuration, ensure that the following settings are in place:

1. **Set HttpOnly Flag**: Set the `HttpOnly` flag on cookies to prevent them from being accessed by JavaScript.
2. **Use Secure Flag**: Set the `Secure` flag on cookies to ensure they are only transmitted over HTTPS.
3. **Content Security Policy (CSP)**: Implement a strict CSP to restrict the sources of content that can be loaded by the browser.

### Common Mistakes and Pitfalls

Common mistakes and pitfalls in implementing CSRF protection include:

1. **Using Predictable Tokens**: Using predictable tokens makes it easier for attackers to guess the token.
2. **Not Validating Tokens**: Failing to validate the token in both the custom header and the cookie.
3. **Not Setting HttpOnly Flag**: Not setting the `HttpOnly` flag on cookies allows them to be accessed by JavaScript.
4. **Not Using Secure Flag**: Not using the `Secure` flag on cookies allows them to be transmitted over HTTP.

### How to Prevent / Defend

To prevent and defend against CSRF attacks, follow these steps:

1. **Generate Unique and Unpredictable Tokens**: Use a secure random number generator to generate unique and unpredictable tokens.
2. **Validate Tokens**: Ensure that the token in the custom header matches the token in the cookie.
3. **Set HttpOnly and Secure Flags**: Set the `HttpOnly` and `Secure` flags on cookies to prevent them from being accessed by JavaScript and ensure they are transmitted over HTTPS.
4. **Implement Content Security Policy (CSP)**: Use CSP to restrict the sources of content that can be loaded by the browser.
