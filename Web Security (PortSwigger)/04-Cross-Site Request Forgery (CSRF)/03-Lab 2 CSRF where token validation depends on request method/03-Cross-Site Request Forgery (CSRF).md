## Cross-Site Request Forgery (CSRF)

### Introduction to CSRF

Cross-Site Request Forgery (CSRF) is a type of attack that tricks a user's browser into executing unwanted actions on a web application in which the user is authenticated. This attack exploits the trust that a web application places in the user's browser. The attacker crafts a malicious request that appears to come from the user, thus bypassing the authentication mechanisms.

### Why CSRF Matters

CSRF attacks can lead to serious consequences such as unauthorized transactions, data manipulation, and account hijacking. For instance, if an attacker can change the email address associated with a user's account, they can then reset the password and take full control of the account. This scenario highlights the importance of securing web applications against CSRF attacks.

### Conditions for CSRF Vulnerability

For a web application to be vulnerable to CSRF, several conditions must be met:

1. **Session Handling via Cookies**: The application must use cookies to maintain user sessions. This is typically the case with most modern web applications.
2. **No Unpredictable Request Parameters**: The attacker must be able to predict all the parameters needed to make a successful request. If there are unpredictable parameters, the attacker cannot craft a reliable malicious request.

### Example Scenario: Changing Email Address

Consider a scenario where an attacker wants to change the email address associated with a user's account. If the attacker can successfully change the email address, they can then use the "forgot password" feature to reset the password and gain full access to the account. This demonstrates the severity of CSRF attacks.

### Detailed Analysis of CSRF Vulnerability

#### Session Handling via Cookies

In the given scenario, the web application uses a session cookie named `session`. This cookie contains the session identifier that the server uses to track the user's session. Here is an example of how the session cookie might appear in an HTTP request:

```http
GET /profile HTTP/1.1
Host: example.com
Cookie: session=abc123
```

The presence of this cookie indicates that the user is authenticated. If an attacker can trick the user's browser into sending a request with this cookie, the server will process the request as if it came from the authenticated user.

#### No Unpredictable Request Parameters

In the given scenario, the attacker needs to know all the parameters required to change the email address. The parameters include the new email address and a CSRF token. The new email address is known to the attacker, but the CSRF token appears to be random. However, the implementation of the CSRF token generation is crucial in determining whether the application is vulnerable to CSRF.

### CSRF Token Implementation

A CSRF token is a unique, unpredictable value generated by the server and sent to the client. The client includes this token in subsequent requests to verify that the request originated from the legitimate user. Here is an example of how a CSRF token might be included in an HTML form:

```html
<form action="/change_email" method="POST">
    <input type="hidden" name="csrf_token" value="random_token_value">
    <input type="email" name="new_email" placeholder="Enter new email">
    <button type="submit">Change Email</button>
</form>
```

When the form is submitted, the server verifies the CSRF token to ensure that the request is legitimate. If the token is missing or invalid, the server rejects the request.

### Potential Vulnerabilities in CSRF Token Implementation

Despite the presence of a CSRF token, the implementation details can still leave the application vulnerable to CSRF attacks. Specifically, if the server accepts the CSRF token only for certain request methods (e.g., POST), an attacker can exploit this by crafting a request using a different method (e.g., GET).

#### Example of Vulnerable Implementation

Suppose the server only validates the CSRF token for POST requests. An attacker can craft a GET request that does not include the CSRF token, yet still changes the email address. Here is an example of such a request:

```http
GET /change_email?new_email=attacker@example.com HTTP/1.1
Host: example.com
Cookie: session=abc123
```

Since the server does not validate the CSRF token for GET requests, the request is processed, and the email address is changed.

### Real-World Examples of CSRF Attacks

#### Recent CVEs and Breaches

One notable example of a CSRF attack is the CVE-2019-11510, which affected the WordPress REST API. The vulnerability allowed attackers to perform unauthorized actions, such as changing user roles, by exploiting the lack of proper CSRF protection.

Another example is the breach of a popular cryptocurrency exchange, where attackers used CSRF to steal funds from users' accounts. The attackers tricked users into clicking on malicious links that executed unauthorized transactions.

### How to Prevent / Defend Against CSRF

#### Detection

To detect CSRF vulnerabilities, you can use automated tools such as Burp Suite or OWASP ZAP. These tools can help identify endpoints that are susceptible to CSRF attacks by analyzing the request and response patterns.

#### Prevention

To prevent CSRF attacks, you should implement robust CSRF protection mechanisms. Here are some best practices:

1. **Use CSRF Tokens**: Ensure that every form or API endpoint that performs sensitive actions includes a CSRF token. The token should be unique per session and regenerated frequently.
2. **Validate CSRF Tokens for All Methods**: Ensure that the server validates the CSRF token for all HTTP methods (GET, POST, PUT, DELETE, etc.). This prevents attackers from bypassing the CSRF protection by using different request methods.
3. **Use SameSite Cookies**: Set the `SameSite` attribute on session cookies to `Strict` or `Lax`. This helps prevent cross-site request forgery attacks by ensuring that the cookie is only sent with requests originating from the same site.
4. **Secure Coding Practices**: Follow secure coding practices to avoid introducing vulnerabilities. For example, ensure that all forms and API endpoints include CSRF tokens and that the server validates these tokens properly.

#### Secure Code Fix Example

Here is an example of how to implement CSRF protection in a web application:

**Vulnerable Code:**

```python
@app.route('/change_email', methods=['POST'])
def change_email():
    new_email = request.form['new_email']
    # Change the email address in the database
    return "Email changed successfully"
```

**Fixed Code:**

```python
@app.route('/change_email', methods=['POST'])
def change_email():
    csrf_token = request.form['csrf_token']
    if csrf_token != session['csrf_token']:
        abort(403)  # Forbidden
    new_email = request.form['new_email']
    # Change the email address in the database
    return "Email changed successfully"
```

In the fixed code, the server checks the CSRF token before processing the request. If the token is missing or invalid, the server returns a 403 Forbidden error.

### Configuration Hardening

To further harden the application against CSRF attacks, you can configure the web server to enforce strict security policies. For example, you can set the `X-Frame-Options` header to `DENY` to prevent clickjacking attacks, which often accompany CSRF attacks.

```nginx
server {
    listen 80;
    server_name example.com;

    location / {
        add_header X-Frame-Options DENY;
        add_header X-XSS-Protection "1; mode=block";
        add_header X-Content-Type-Options nosniff;
    }
}
```
