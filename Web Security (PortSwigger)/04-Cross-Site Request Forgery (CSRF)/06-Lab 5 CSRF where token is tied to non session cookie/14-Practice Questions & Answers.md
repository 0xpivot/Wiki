## Practice Questions & Answers

**Q1. Explain the concept of CSRF and how it can be mitigated using tokens.**

CSRF (Cross-Site Request Forgery) is an attack where an attacker tricks a user's browser into executing unwanted actions on a web application in which the user is authenticated. To mitigate CSRF, tokens are often used. These tokens are unique values generated by the server and sent to the client. When the client sends a request back to the server, it includes this token. The server verifies the token to ensure that the request originated from a legitimate source and not an attacker. This helps prevent unauthorized actions.

**Q2. How does tying a CSRF token to a non-session cookie introduce vulnerabilities? Provide an example from the lab.**

Tying a CSRF token to a non-session cookie can introduce vulnerabilities if the token is not properly integrated with the session handling mechanism. In the lab, the CSRF token is tied to a non-session cookie called `CSRF key`. If this `CSRF key` is not tightly coupled with the session management, an attacker can exploit this by injecting their own `CSRF key` cookie into the victim's session. For instance, if the application allows setting cookies via HTTP headers and an attacker can inject a `CSRF key` cookie, they can then craft a CSRF attack using their own CSRF token, effectively bypassing the CSRF protection.

**Q3. Describe how HTTP header injection can be exploited to perform a CSRF attack in the context of the lab.**

In the lab, the application allows setting a cookie (`last search term`) based on user input during a search operation. By manipulating the input to include a newline character (`%0d%0a`) followed by a `Set-Cookie` directive, an attacker can inject a `CSRF key` cookie into the victim's session. Once the `CSRF key` is injected, the attacker can craft a CSRF attack using their own CSRF token. This combination of HTTP header injection and CSRF exploitation allows the attacker to change the victim's email address without needing to know the actual CSRF token of the victim.

**Q4. How would you fix the vulnerability described in the lab to prevent CSRF attacks?**

To fix the vulnerability described in the lab, the CSRF token and the `CSRF key` cookie should be tightly integrated with the session handling mechanism. Specifically:

1. **Ensure CSRF Tokens are Session-Bound:** The CSRF token should be generated per session and stored in a secure manner, such as in a session variable. This ensures that the token is unique to each session and cannot be reused across sessions.

2. **Validate CSRF Tokens Properly:** The server should validate both the CSRF token and the `CSRF key` cookie against the session data. If they do not match, the request should be rejected.

3. **Prevent HTTP Header Injection:** Ensure that user inputs are properly sanitized to prevent HTTP header injection. This can be achieved by validating and escaping user inputs to prevent malicious characters from being interpreted as HTTP headers.

By implementing these measures, the application can effectively prevent CSRF attacks and ensure that the CSRF tokens are properly integrated with the session handling mechanism.

**Q5. How can you verify whether a CSRF token is tied to a specific session in the lab environment?**

To verify whether a CSRF token is tied to a specific session in the lab environment, you can perform the following steps:

1. **Obtain CSRF Token and Cookie:** Log in to the application as the victim and obtain the CSRF token and the `CSRF key` cookie.

2. **Test with Another User's Token:** Log in as another user and obtain their CSRF token and `CSRF key` cookie. Use these tokens and cookies in the victim's session to see if the request is accepted.

3. **Inject Custom CSRF Key:** Use HTTP header injection to inject a custom `CSRF key` cookie into the victim's session and attempt to perform a CSRF attack using the attacker's CSRF token.

If the request is accepted when using another user's tokens or a custom `CSRF key`, it indicates that the CSRF token is not tightly integrated with the session handling mechanism, leading to potential CSRF vulnerabilities.

**Q6. How would you exploit the CSRF vulnerability in the lab if you had access to the victim's session?**

To exploit the CSRF vulnerability in the lab, assuming you have access to the victim's session, you can follow these steps:

1. **Obtain CSRF Token and Cookie:** Obtain the CSRF token and the `CSRF key` cookie from the victim's session.

2. **Craft CSRF Attack:** Create an HTML page that includes a form with the CSRF token and a hidden field for the email address you want to change to. Additionally, include an image tag that attempts to load a URL that sets the `CSRF key` cookie to the attacker's value.

3. **Trigger CSRF Attack:** Send the HTML page to the victim via a phishing link. When the victim visits the page, the image tag will attempt to load the URL, setting the `CSRF key` cookie. Upon encountering an error, the form will be automatically submitted, changing the victim's email address.

Here is an example payload:

```html
<html>
<body>
<h1>Hello World</h1>
<form id="csrfForm" action="http://victim-app.com/my-account/change-email" method="POST">
    <input type="hidden" name="email" value="attacker@example.com">
    <input type="hidden" name="csrf" value="attacker-csrf-token">
</form>
<img src="http://victim-app.com/search?term=hat%0d%0aSet-Cookie:CSRF-key=attacker-csrf-key-cookie-value" onerror="document.getElementById('csrfForm').submit()">
</body>
</html>
```

This payload sets the `CSRF key` cookie to the attacker's value and then triggers the form submission, effectively changing the victim's email address.

**Q7. What recent real-world examples demonstrate the importance of proper CSRF token integration with session handling?**

Recent real-world examples highlight the importance of proper CSRF token integration with session handling:

- **CVE-2021-21972:** A vulnerability in the WordPress REST API allowed attackers to perform CSRF attacks by tricking users into making unauthorized requests. This vulnerability was due to improper validation of CSRF tokens, allowing attackers to bypass CSRF protections.

- **CVE-2020-14182:** A vulnerability in the Atlassian Jira software allowed attackers to perform CSRF attacks by exploiting a flaw in the CSRF token validation mechanism. This flaw allowed attackers to reuse CSRF tokens across sessions, leading to unauthorized actions.

These examples underscore the critical nature of ensuring that CSRF tokens are tightly integrated with session handling mechanisms to prevent unauthorized actions and protect user data.
